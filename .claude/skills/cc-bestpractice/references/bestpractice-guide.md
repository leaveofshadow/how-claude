## Parse 阶段详解

Parse 是编排者自身执行的意图解析（不派 agent），输出四元组 `{ domain, scope, format, mode }`。解析失败或歧义时必须 AskUserQuestion，绝不猜测后直接生成。

### 解析目标四元组

| 字段 | 含义 | 取值示例 |
|------|------|---------|
| `domain` | 领域名（生成文档的主体） | Python / React / FastAPI / Kubernetes |
| `scope` | 范围限定（可选，收窄聚焦） | 后端 API / 状态管理 / 3.12+ |
| `format` | 输出形态 | 完整文档 / 检查清单 / 两者 |
| `mode` | 新建 or 更新 | create / update |

### 关键词匹配规则（按优先级）

按以下顺序匹配，命中即定：

1. **模式识别**（mode）
   - 含 `更新` / `update` / `refresh` / `升级` / `刷新` → `mode=update`
   - 其余默认 `mode=create`
   - 若 `mode=create` 但检测到已有同名文档 → 改 mode 并提示用户选择

2. **格式识别**（format）
   - 含 `检查清单` / `checklist` / `速查` / `速查卡` → `format=检查清单`
   - 含 `完整` / `详细` / `全面` → `format=完整文档`
   - 同时出现两者 → `format=两者`
   - 默认 → `format=完整文档`

3. **文档类型识别**（隐式决定模板，详见 Route 章节）
   - `编码规范` / `style guide` / `coding standard` → coding-standard
   - `架构指南` / `architecture guide` → architecture-guide
   - `项目X规范` / `我们项目` → project-standard
   - `最佳实践` / `best practice` / `领域指南` → domain-guide

4. **领域提取**（domain）
   - 取触发短语中的技术名词：`{领域}+最佳实践/规范/指南` 中的 `{领域}`
   - 若只有一个技术名词直接取；多个名词（如 "React 和 TypeScript"）→ domain 取主，scope 记次

### 歧义处理（必须追问）

遇到以下三类歧义**禁止**直接生成，必须 AskUserQuestion：

**歧义1：类型不清**。例 "Python规范" — 是编码规范（怎么写）还是领域指南（怎么用）？
- 追问："你要的是『Python 编码规范』（命名/格式/lint 等书写规则）还是『Python 最佳实践』（完整开发指南，含架构/测试/部署）？"

**歧义2：范围过宽**。例 "前端最佳实践" — 前端范围太大。
- 追问："前端范围很大，聚焦哪个栈？React / Vue / 原生 JS / 移动端 H5？"

**歧义3：新建 vs 更新冲突**。例 用户说"生成 Python 最佳实践"但已有 `python-bestpractice.md`。
- 追问："检测到已有 `python-bestpractice.md`（v1.2.0，更新于 2026-03），你是要【增量更新】（只补新内容）还是【全量重生成】（覆盖重写）？"

### Parse 输出示例

```
domain: "Python"
scope: "3.12+ 后端 API"
format: "完整文档"
mode: "create"
```

Parse 预算 ~500 token，禁在此步做任何 WebSearch 或代码扫描。

## Route 阶段详解

Route 也是编排者自身执行，查路由表得到三路信息权重，再映射成搜索轮次。权重静态表 + 轮次映射，避免凭感觉分配预算。

### 三路信息权重表

权重四级：**重 / 中 / 轻 / 0**。下表为 4 种文档类型下的默认权重。

| 文档类型 | web（外部搜索） | code（项目代码） | user（用户描述） |
|---------|:---:|:---:|:---:|
| 领域指南（domain-guide） | 重 | 轻 | 中 |
| 编码规范（coding-standard） | 中 | 中 | 轻 |
| 架构指南（architecture-guide） | 中 | 轻 | 重 |
| 项目规范（project-standard） | 轻 | 重 | 中 |

**读法**：领域指南主要靠 web 搜索外部最佳实践；项目规范主要靠扫描项目代码；架构指南主要靠用户描述（架构决策高度项目特化）。

### 权重 → 搜索轮次映射

| 权重 | Web 搜索轮次 | 代码扫描 | 用户提问 |
|------|:----------:|:------:|:------:|
| 重 | 4 轮 | 全量扫描清单 | 2-3 个问题 |
| 中 | 2 轮 | 全量扫描清单 | 1-2 个问题 |
| 轻 | 0 轮（跳过） | 关键文件扫描 | 0-1 个问题 |
| 0 | 跳过 | 跳过 | 跳过 |

**轮次硬上限**：Web 搜索全局最多 4 轮（20 次搜索），达上限强制停止（见上下文预算保护机制）。

### Route 输出示例（领域指南场景）

```
type: domain-guide
weights: { web: 重, code: 轻, user: 中 }
maxSearchRounds: 4
routes:
  RouteA(Web): 4 轮搜索（启用）
  RouteB(代码): 关键文件扫描（启用）
  RouteC(用户): 1-2 个问题（启用）
```

### 路由例外（覆盖静态表）

- **离线/内网环境**：WebSearch 失败 → webWeight 降为 0，权重自动转移给 code+user，并告知用户"web 不可用，文档深度可能下降"。
- **空项目**：代码扫描无文件可扫 → codeWeight 降为 0，权重转移给 web。
- **用户显式指定**：用户说"只看官方文档"→ 强制 web=重，user/code=轻。

## Research 阶段详解

Research 按 Route 结果并行/串行执行三路采集，每路返回**结构化摘要**（不是原文），编排者把摘要落盘到 `.claude/bp-research/{domain}.md` 供 Generate 阶段读取。

### ★v1.11 信息源扩展 + 分阶段采集

> v1.11 修正（对抗审查）：旧版只有 web/code/user 三路，搜不到橙皮书级深度（原理/演变/人物/争议）。新版扩到 7 类采集源，但每个新源**必须配降级路径**（防标准环境工具缺失），且 Research 本身**也分阶段**（防上下文爆炸，呼应渐进式生成）。

#### 7 类信息源（M1-M7，每类配探测+降级）

| 源 | 探测命令/方式 | 失败降级 |
|----|--------------|---------|
| M1 WebSearch | 标准能力 | 失败→webWeight 降为 0（已有机制，见 L128） |
| M2 GitHub挖掘 | `command -v gh && gh auth status` | **失败→WebFetch 官方 release/changelog 页面**（功能子集，只丢 PR/issue） |
| M3 代码扫描（Route B） | Glob/Read/Grep | 已有 |
| M4 多模型咨询 | `/ask` 是否注册 | **失败→WebSearch + 自我对抗**（不同 prompt 角色模拟多视角，置信度降级到 ★） |
| M5 项目记忆/wiki | wiki_query/project_memory_read | 不存在→跳过（不报错） |
| M6 会话历史 | session_search | 无结果→跳过 |
| M7 用户深化（Route C） | AskUserQuestion | 已有 |

**探测时机**：Parse 阶段后、Research 阶段前，编排者先探测 M2/M4/M5/M6 是否可用，落盘到 `.claude/bp-research/{domain}-sources.json`，决定本次实际启用哪些源。

#### Research 分阶段（防上下文爆炸）

```
阶段 R1（基础信息源组）:
  → 轮1-4 WebSearch（M1）+ M3代码扫描 + M7用户深化
  → 每路摘要 ≤2000 token（保持现有硬约束 L634）
  → 落盘到 .claude/bp-research/{domain}-basic.md
  → 这是承重物，后续都读它

阶段 R2（深度信息源组，独立 agent，只读 R1 摘要）:
  → 轮5-6 WebSearch（原理+人物 / 争议+方向，见下"搜索公式扩展"）
  → M2 GitHub挖掘（降级兼容）+ M4多模型（按需，见"预算分级"）
  → 只读 {domain}-basic.md，不读原始采集数据
  → gh输出强制蒸馏（只取 release标题+日期+breaking changes，不取全量CHANGELOG）
  → 多模型回复强制 ≤500 token 结构化
  → 落盘到 .claude/bp-research/{domain}-deep.md
```

**为什么分阶段**：旧版渐进式三阶段只在 Generate 阶段分（A大纲→B审查→C填充），Research 是一次性全量采集。扩到 7 源后会撑爆单会话（M2 的 gh CHANGELOG 可上千行、M4 两个外部模型完整意见）。分阶段后，深度源组只读基础组摘要，不读原始数据，上下文压力隔离。

#### 搜索公式扩展（4轮→6轮，轮5-6 深度档才触发）

```
轮1（基础+版本校正）: "{domain} latest version {year}" + "{domain} stable release"
                     + "{domain} best practices {year}"
                     ★新增: "{domain} creator / lead maintainer" → 找核心人物名（供轮5用）
轮2（反模式+踩坑）: "{domain} anti-patterns / common mistakes"
                   ★新增: "{domain} site:reddit.com OR site:news.ycombinator.com" → 社区真实吐槽
轮3（工具+生态）: "{domain} recommended tools / linter / formatter {year}"
                 ★新增: "{domain} ecosystem comparison {year}"
轮4（版本+迁移）: "{domain} {version} new features / migration"
                 ★新增: "{domain} breaking changes / deprecation" → 演变脉络素材
轮5（★深度档: 原理+人物）: "{domain} how it works internally / architecture / under the hood"
                          "{核心人物名} {domain} opinion / talk / blog"
                          "{domain} design decisions / why {核心设计选择}"
轮6（★深度档: 争议+方向）: "{domain} controversy / debate / criticism / vs {替代方案}"
                          "{domain} roadmap / future / trend {year+1}"
                          "{domain} what's changing / what's deprecated"
★v1.11.1 量化采集（每个领域必搜，支撑D1量化强化）:
                          "{domain} pypistats / npm downloads / monthly downloads" → 采用率
                          "{domain} benchmark / TechEmpower / performance comparison" → 性能基准
                          "{domain} github stars / community size / stackoverflow tag" → 社区规模
                          "{domain} release frequency / latest release date" → 版本节奏
```

**轮5-6 触发条件**：用户选深度档 / 文档类型是领域指南或架构指南 / 阶段B审查判定深度不足。轻量档不跑轮5-6。

#### 信息源 × 深度层映射（每层不必跑全套 7 源）

| 深度层 | 主力采集源 | 补充源 |
|--------|-----------|--------|
| 量化证据 | M1 官方+深度站 | M2 仓库 star/issue 数 |
| 人物引言 | M1 轮5 人物搜索 | M2 核心维护者 profile |
| 底层原理 | M2 仓库 DESIGN/docs | M1 轮5 under the hood |
| 演变脉络 | M2 CHANGELOG/release | M1 轮5 history |
| 心智模型 | M1 综合 + M4 多模型 | （Claude 自身综合能力） |
| 争议/权衡 | M1 轮6 争议 + L3 社区 | M2 高票 issue + M4 多模型 |
| 方向感 | M1 轮6 roadmap | M4 多模型 |

### Route A：Web 搜索（当 webWeight > 0）

**搜索关键词构造公式**：

```
轮1（基础+版本校正）: "{domain} latest version {year}" + "{domain} stable release" → 校正版本基线（门禁7 依据）
                   + "{domain} best practices {year}"
轮2（深入）: "{domain} anti-patterns / common mistakes"
轮3（工具）: "{domain} recommended tools / linter / formatter {year}"
轮4（版本）: "{domain} {version} new features / migration"
```

> ★v1.1 强制：轮1 必须含版本校正搜索。轮1 先确定 PyPI/npm/crates 实际稳定版，后续所有版本声明以此为准。对抗验证发现凭 Claude 训练记忆写版本号会滞后 6-12 个月（攻击C #1：FastAPI 实际 0.137 却写 0.110）。

若 `scope` 非空，每轮关键词追加 `+ {scope}`（如 `+ 后端 API`）。

#### 版本基线校正子节（★v1.1 新增，供门禁7）

- **校正动作**：轮1 搜 `{framework} latest version {year}` + `{framework} stable release`，从 PyPI/npm/crates/官方 release notes 核实当前实际稳定版
- **版本声明规则**：文档头部版本基线 = web 核实的实际稳定版 ±2 minor。超出必须说明理由（如"本项目锁旧版因为 X 兼容"）
- **反模式**：凭 Claude 训练记忆写版本号——训练数据可能滞后 6-12 个月，且对"已移除/已弃用"类事实常错（如把 Starlette 1.0 才移除的 `on_event` 写成"0.39 移除"，攻击C #2）
- **校正示例**：
  - ❌ 反模式：标"更新于 2026-06-16"却用 FastAPI 0.110（凭记忆），实际 PyPI 已 0.137.1 → 落后 27 个 minor，门禁7 失败
  - ✅ 正确：轮1 搜出 latest=0.137.1 → 头部基线标 `FastAPI 0.135+`（±2 minor 内）→ 所有版本声明以此为准

**每轮"搜索-深读-摘要"循环逻辑**：

```
每轮:
  1. 执行 3-5 次 WebSearch（按公式变体关键词）
  2. 从搜索结果挑 ≤2 个高权威 URL → WebFetch 深读
  3. 把深读内容蒸馏成结构化摘要段落
  4. 摘要必须含: [主张] + [来源 URL] + [置信度初判]
轮间衔接: 下一轮关键词基于上一轮发现的盲区调整
```

**搜索结果质量判断标准**（决定是否深读）：

| 维度 | 高质量（深读） | 低质量（跳过） |
|------|--------------|--------------|
| 域名权威 | 官方域名 / 知名技术站 | 内容农场 / 无作者转载 |
| 时效 | 1 年内 / 对应目标版本 | 3 年前 / 版本不符 |
| 具体 | 有代码示例 / 版本号 | 纯观点 / "据说" |

例：搜 "Python 3.12 best practices" → 深读 `python.org/docs/3.12`、`realpython.com`；跳过 `some-blog.xyz/我觉得`。

### Route B：代码扫描（当 codeWeight > 0）

**扫描清单（按读取优先级）**：

1. **文件树**（Glob，零成本）：`**/*` 取结构，识别项目类型（monorepo / 单包 / 多服务）
2. **依赖文件**（Read，关键）：`pyproject.toml` / `package.json` / `go.mod` / `Cargo.toml` → 提取依赖版本
3. **配置文件**（Read，关键）：lint/format/build/test 配置 → 提取已有规则
4. **已有文档**（Read，如存在）：`README.md` / `CONTRIBUTING.md` / `docs/` → 提取已有规范
5. **AST 签名层**（Grep+Read 签名，**不读实现**）：用 Repomix 压缩思路，只看函数签名/类定义/导出

**扫描返回**：项目特征摘要 = `{ 技术栈版本, 架构模式, 已有规范, 代码风格初判 }`。

**硬约束**：最多读 30 个文件签名，不读实现细节（防上下文爆炸）。

### Route C：用户意图深化（当 userWeight > 0）

**提问模板**（基于 A/B 采集结果精准提问，不超过 3 个）：

1. **偏好类**："你的项目偏好 A 还是 B？（例：类型注解 strict / soft；同步 / 异步）"
2. **禁忌类**："有哪些是绝对不能出现的反模式？（例：禁止裸 except / 禁止 any）"
3. **特化类**："对比行业标准，你的项目有什么特殊约束？（例：必须兼容 Python 3.9 / 离线环境）"

提问时机：Route C 在 A/B 完成后执行，避免问已在代码中能看出答案的问题。

## Generate 阶段详解

Generate 由单个 general-purpose(opus) agent 执行：读摘要 + 读模板 → 按模板填充 → 写文件。编排者不参与写作，只读结果做 Gate。

### Agent Prompt 模板

```
你是领域最佳实践文档生成 agent。

任务: 为 {domain}（scope: {scope}）生成 {文档类型} 文档。

步骤:
1. Read 模板: .claude/skills/cc-bestpractice/references/templates/{type}-template.md
2. Read 研究摘要: .claude/bp-research/{domain}.md
3. 严格按模板章节填充，不增减主章节
4. 每条推荐必须满足:
   - 附 ≥1 个具体项（代码示例 / 命令 / 配置片段 / 工具名+版本）
   - 附反例或反模式（"不要 X: <bad>" → "应改为: <good>"）
   - 标注置信度 [★★★/★★/★]
   - 关键事实声明标注来源（官方/社区/版本号）
5. 反模式清单 ≥ 8 条，每条四列（反模式|为什么错|正确做法|严重度）
6. 写文件到: {目标路径}
7. 文件头部包含 YAML 元数据块（domain/type/version/sources/template）

质量红线（违反即不合格）:
- 禁止"应该注意 XX"式空话，必须落到具体做法
- 禁止无来源的版本/特性断言
- 禁止只讲正确不讲错误

输出: 文档写入 {目标路径}，返回写入确认 + 主章节数。
```

### 上下文预算控制规则（Generate 内部）

1. **章节级填充**：一次填一章，避免长上下文遗忘前面章节结构
2. **摘要引用而非复述**：研究摘要是事实源，文档是表达——不复述摘要原文，要重新组织成读者视角
3. **硬上限**：单文档 ≤ 5000 行；超出 → 截断到核心章节 + 标注"完整版见附录/分篇"
4. **置信度自动标注**：每写完一条推荐立即打置信度标签，不留到最后补

### Generate 输出

- 文档文件写入目标路径
- 返回 `{ 文件路径, 主章节数, 反模式条数, 置信度标注数 }` 给编排者做 Gate 输入

## Gate 阶段：反废话7项门禁 + γ三问

Generate 写完后，编排者 Read 文档 → 逐项检查 → 通过率 ≥ 80% 输出，否则补强 1 轮。

> v1.1 升级：从 5 项扩到 7 项。新增的 6/7 项源于 FastAPI 对抗验证——攻击者挖出"5 项全过却跑不起来 + 版本滞后 27 个 minor"。条目级门禁（1-5）保质量下限，集成级门禁（6-7）保"真能用"。

### Gate 流程（3 步）

```
1. 编排者 Read 生成文档
2. 逐项执行 7 项门禁 → 计算通过率
3. 通过率 ≥ 80% → 输出
   通过率 < 80% → 构造补强 prompt → 回 Generate（指定失败章节）
   补强后再检一次 → 最多 1 轮
   仍不通过 → 输出 + 标注"⚠️ 建议人工审阅: {章节列表}"
```

### 反废话 7 项门禁细则

#### 门禁1：具体性门禁

- **检查什么**：每条推荐是否附带 ≥1 个具体项（代码示例 / 命令 / 配置片段 / 工具名+版本）
- **通过标准**：≥ 90% 的推荐条目有具体项
- **失败处理**：标注空洞条目，要求 agent 补具体示例
- **示例对比**：
  - ❌ 失败："使用类型注解"
  - ✅ 通过："使用类型注解：`def foo(x: int) -> str:`（Python 3.5+，mypy 校验）"

#### 门禁2：反例门禁

- **检查什么**：每个规范条目是否有反例或反模式
- **通过标准**：≥ 80% 的规范条目含反例
- **失败处理**：标注缺反例条目，要求补 "不要 X: <bad> → 应改为: <good>"
- **示例对比**：
  - ❌ 失败："不要裸 except"（无反例展示）
  - ✅ 通过："不要裸 except：`except:` → 应改为 `except ValueError as e:`"

#### 门禁3：来源门禁

- **检查什么**：关键事实声明（版本特性/性能数字/兼容性）是否标注来源
- **通过标准**：100% 的版本/特性声明有来源（零容忍）
- **失败处理**：无来源 → 补来源或降低断言强度（"支持"→"通常支持"）
- **示例对比**：
  - ❌ 失败："Python 3.12 支持 match-case 语法"
  - ✅ 通过："Python 3.12 支持 match-case 语法（来源：python.org/docs/3.12）"

#### 门禁4：反模式门禁

- **检查什么**：反模式清单是否 ≥ 8 条，每条有具体描述
- **通过标准**：≥ 8 条且每条四列齐全（反模式|为什么错|正确做法|严重度）
- **失败处理**：补足到 8 条，删除"不要写烂代码"式废话条目
- **示例对比**：
  - ❌ 失败：5 条，含"代码要写好"式废话
  - ✅ 通过：8 条，如"裸 except | 吞掉所有异常含 KeyboardInterrupt | `except SpecificError` | 高"

#### 门禁5：可操作性门禁

- **检查什么**：读者读完任一章节能否立即行动（装什么/写什么/改什么）
- **通过标准**：每个推荐章节末尾有可执行步骤或命令
- **失败处理**：标注不可操作章节，要求补"安装/配置/执行"三件套
- **示例对比**：
  - ❌ 失败："配置 lint"
  - ✅ 通过："配置 lint：`pip install ruff && ruff init`，规则集见附录"

### γ嫁接·反废话三问（嵌入门禁的精炼表达）

> v1.1：三问之上新增两个**集成级**检查（门禁6 可运行性、门禁7 版本时效）。三问保单条质量，集成级门禁保"组合起来能用"。对抗验证证明：条目级全过 ≠ 整体可上手。

#### 门禁6：可运行性门禁 ★v1.1新增（集成级）

> 源于对抗验证攻击B #1/#3/#6/#8：FastAPI 文档 5 项门禁全过，但用户连 hello world 都跑不起来——import 链全是断的（`get_db`/`get_settings`/`User` model/conftest fixture 全文未定义）。

- **检查什么**：
  - 文档是否含「从零到 curl 通」的完整最小示例（入口文件 + 启动命令 + 验证命令三件套）
  - 所有散落的胶水函数（`get_db`/`get_settings`/`User` model/`Base`/conftest fixture 等）在附录D是否有完整实现
  - import 链是否自洽（不存在「正文引用了 `from app.api.deps import get_db` 但全文无 `get_db` 实现」的断链）
- **通过标准**：100% 的 import 链自洽（零断链）；附录D 完整骨架能 `uvicorn`/`cargo`/`npm` 启动并通过至少 1 个 curl/请求验证
- **失败处理**：
  - 标注断链的 import → 要求 agent 补胶水函数实现到附录D
  - 缺最小示例 → 补「§0 五分钟跑通」章节（单文件 main + 启动命令 + curl 验证）
  - conftest fixture 缺失 → 补含 DB fixture + `dependency_overrides` 清理的完整 conftest.py
- **示例对比**：
  - ❌ 失败：正文 `from app.api.deps import get_db` 但全文无 `get_db` 实现；测试 `client.post("/api/v1/users/")` 但 POST endpoint 从未定义；§8 Dockerfile 用 `<(uv export ...)` 进程替换在 sh 下语法错（攻击B #1/#3/#4）
  - ✅ 通过：正文引用 `get_db` + 附录D 有 `def get_db(): yield session` 完整实现 + main.py 可 `uvicorn app.main:app` 启动 + `curl /health` 返回 200

#### 门禁7：版本时效门禁 ★v1.1新增（集成级）

> 源于对抗验证攻击C #1/#2：文档标"更新于 2026-06-16"却用 FastAPI 0.110，实际已是 0.137.1（落后 27 个 minor）；反模式写"`@on_event` Starlette 0.39+ 移除"实际在 1.0.0 才移除——凭 Claude 训练记忆断版本号。

- **检查什么**：
  - Research 第 1 轮搜索是否含 `{framework} latest version {year}`（供版本基线校正）
  - 文档头部版本基线 vs web 核实的实际稳定版，差距是否 ≤ 2 个 minor
  - 关键事实声明（"X API 已移除/弃用"）是否有 web 证据（来源 URL），禁凭记忆断版本号
- **通过标准**：版本基线落后实际稳定版 ≤ 2 个 minor；所有"移除/弃用"类声明有来源 URL；头部 `updated` 日期与版本基线自洽
- **失败处理**：
  - 版本滞后 → 补搜最新版本 + 抬高基线（如 0.110 → 0.135+）
  - 版本声明无来源 → 补 web 证据或降置信度到 ★
  - 版本号事实错（如"0.39+ 移除"实为"1.0.0 移除"）→ 订正并补来源
- **示例对比**：
  - ❌ 失败：标"更新于 2026-06-16"却用 FastAPI 0.110，实际已 0.137.1（落后 27 个 minor）；"Starlette 0.39+ 移除 `on_event`" 无来源且事实错（实为 1.0.0）；推荐 `python-jose` 却不知其 2024 多个未修 CVE（攻击C #1/#2，攻击A #2）
  - ✅ 通过：Research 搜出 latest version 0.137.1，基线标注为 `FastAPI 0.135+`（±2 minor 内）；"1.0.0 移除"声明附 starlette.dev/release-notes 来源 URL

### γ嫁接·反废话三问（嵌入门禁的精炼表达）

每条推荐在 Generate 阶段先过三问，三问全过才标注 ★★★：

1. **具体吗？**（对应门禁1）— 有没有落到代码/命令/版本？
2. **可操作吗？**（对应门禁5）— 读者能立即照做吗？
3. **有证据吗？**（对应门禁3）— 有来源支撑吗？

三问任一不通过 → 降级为 ★★ 或 ★，并触发补强。三问是门禁1/3/5 的执行口诀，不是独立检查。

### 通过率计算

```
通过率 = (门禁1..7 各自通过) / 7
门禁3（来源）、门禁6（可运行性）、门禁7（版本时效）是硬门禁：任一不通过直接判整体不通过
```

## 置信度标注规则

每条推荐标注可信度，让读者知道哪些可放心用、哪些需谨慎。置信度在 Generate 阶段由 agent 标注，Gate 阶段检查覆盖率。

### 三级定义

| 标注 | 等级 | 判定标准 |
|------|------|---------|
| `★★★` | 高置信 | 官方文档明确推荐 **且** 社区广泛共识 **且** 有版本号佐证 |
| `★★` | 中置信 | 社区主流做法但无官方推荐 / 官方推荐但社区争议 / 来自单一可信来源 |
| `★` | 低置信 | 个人经验 / 社区小众做法 / 无来源佐证 |

### 标注格式

在每条推荐末尾加标注（独立成段或行内）：

```markdown
推荐：使用类型注解 + mypy 静态检查 [★★★]

推荐：用 dataclass 替代手写 __init__ [★★★]
```

整段规范可统一标注在章节标题后：

```markdown
### 3.3 类型系统 [整体 ★★★]
```

### 生成时指令

- Generate agent 写每条推荐时**立即**打标签，基于该条的来源强度判断
- 三问（具体/可操作/证据）全过 → ★★★；缺证据或缺可操作 → ★★；纯经验 → ★
- 不确定时取较低等级（保守标注）

### Gate 检查覆盖率

- **覆盖率标准**：≥ 90% 的推荐条目有置信度标注
- **分布合理性**：一篇文档不应 100% 是 ★★★（说明没认真区分）也不应 100% ★（说明没采到可信源）
- **失败处理**：覆盖率 < 90% → 回 Generate 补标注

## 交叉验证策略

交叉验证嵌入 Research 阶段，三路信息两两交叉，冲突时按优先级消解。结果落盘到研究摘要供 Generate 参考。

### 三组交叉验证规则

#### web × 代码

- **规则**：web 搜到的推荐实践 → 到项目代码验证是否已使用 / 是否冲突
- **冲突时**：标注"行业标准推荐 X，本项目使用 Y"，由用户在 Route C 决定
- **示例**：web 说"用 pytest"，代码里全是 unittest → 摘要记"web 推 pytest | 代码用 unittest | 待用户决策"

#### web × 用户

- **规则**：web 推荐的 → 与用户描述的偏好对比
- **冲突时**：**优先用户偏好**，标注差异（项目特化场景用户说了算）
- **示例**：web 说"异步框架 FastAPI 性能更好"，用户说"团队不熟异步" → 采纳同步 + 标注"性能取舍"

#### 代码 × 用户

- **规则**：代码扫描发现的现有模式 → 与用户描述对比
- **冲突时**：不一致 → **追问用户**（代码可能历史遗留，不一定是当前意愿）
- **示例**：代码用 tabs 缩进，用户说要 spaces → 追问"代码现状是 tabs，是历史遗留要迁移，还是继续保持？"

### 冲突消解优先级

```
安全/正确性相关冲突 → 以官方文档/安全公告为准（web 官方 > 一切）
项目特定约束冲突   → 以用户偏好为准（user > web）
风格/习惯冲突      → 以项目现状为准（code > web），但标注行业标准
```

### 交叉验证落盘格式

每条冲突/验证结果按统一格式记录到研究摘要：

```
{信息源A}: {主张}
{信息源B}: {主张}
结论: {采纳方案}
理由: {为什么采纳这个}
```

**示例**：

```
web: Python 3.12 推荐 match-case 替代多重 if-elif [python.org/docs/3.12]
代码: 现有代码大量使用 if-elif 链，无 match-case
结论: 新代码用 match-case，旧代码不强制迁移
理由: match-case 是 3.10+ 特性，项目已升 3.12；但批量重构成本高，增量采用
```

## 增量融合流程

当 `mode=update` 时，走 U1-U4 增量融合而非全量重生成。成本约为首次生成的 1/3，且保留已验证内容。

### U1：读取现有文档元数据

- **输入**：现有文档文件路径
- **操作**：Read 文件头部 YAML frontmatter
- **输出**：`{ domain, version, sources, updated, template }`
- **关键**：`updated` 日期决定 U2 的差异扫描起点

### U2：差异扫描

- **输入**：U1 的 `updated` 日期 + `domain`
- **操作**：Web 搜索 `"{domain} 新版本/新实践/新工具 since {updated日期}"`
- **输出**：每个章节标记三种状态之一：
  - **过时**：内容引用了已 EOL 的版本/被废弃的 API
  - **新增**：出现了文档没有的新实践/新工具
  - **未变更**：内容仍准确
- **示例**：文档说 "用 pyright 做类型检查"，新搜索发现 "基于 pyright 的 ruff 类型检查已发布" → 标记"新增"

### U3：精准融合

按状态分别处理：

| 章节状态 | 处理方式 | 预算 |
|---------|---------|------|
| 过时 | **重写该章节**（保留章节结构，更新内容） | 中 |
| 新增 | **追加到对应位置**（不破坏现有章节） | 低 |
| 未变更 | **原样保留**（不动已验证内容） | 0 |

**融合纪律**：未变更章节一字不动——已通过 Gate 的内容不应被增量更新破坏。

### U4：更新元数据

- `version`：按版本语义表递增
- `updated`：当前日期
- `sources`：追加新来源（不删旧来源）
- 更新日志追加一行

### 版本语义表

| 变更类型 | 版本规则 | 示例 |
|---------|---------|------|
| 首次生成 | `1.0.0` | — |
| 新增章节 / 补充内容（U3 新增） | patch +1 | `1.0.0 → 1.0.1` |
| 过时章节重写（U3 过时） | minor +1 | `1.0.1 → 1.1.0` |
| 模板升级 / 大范围重构 | major +1 | `1.1.0 → 2.0.0` |

### 增量融合边界（何时该放弃增量、改全量重生成）

- **超过 6 个月未更新** → 建议全量重生成（信息偏移太大，差异扫描成本接近全量）
- **领域 major 版本更新**（如 Python 3.11 → 3.12） → 建议全量重生成（大量章节会过时）
- **仅追加少量新实践** → 增量融合（最佳场景）

边界判断由编排者在 U2 后做：若"过时"章节占比 > 50% → 提示用户"过时内容太多，建议全量重生成"。

## 深度层配置（★v1.11 — 模板×深度层矩阵 + 防兑水护栏）

> v1.11 修正（对抗审查#5/6）：D3-D7 深度层从"必含"改为"按文档类型选含"，避免所有文档长得一样（同质化）+ 形式主义。
> 矩阵让 Parse 阶段一眼定该开哪些层；护栏防深度层兑水成废话。

### 模板 × 深度层矩阵

| 深度层 | domain-guide（领域指南） | architecture（架构指南） | coding-standard（编码规范） | project-standard（项目规范） |
|--------|:---:|:---:|:---:|:---:|
| D3 原理层（底层怎么运作） | ✅ | ✅ | ❌ | ❌ |
| D4 演变层（从哪来→现在在哪） | ✅ | ⭕ | ❌ | ❌ |
| D5 心智模型（统领框架） | ✅ | ✅ | ❌ | ❌ |
| D6 争议/权衡（放弃了什么） | ⭕ | ⭕ | ⭕ | ❌ |
| D7 方向层（往哪走） | ✅ | ⭕ | ❌ | ❌ |

**图例**：✅=必含（默认橙皮书风格时） ⭕=按领域相关性选含 ❌=默认关闭

**选含信号（⭕ 的触发条件）**：
- D4 演变：用户说"历史/演变/怎么来的" 或 领域有明确代际（如 Web框架同步→异步）→ 开
- D6 争议：用户说"选型/对比/权衡" 或 Research 发现社区分裂（多源说法矛盾）→ 开
- D7 方向：用户说"趋势/未来/方向" 或 领域快速变化（近1年有重大版本/范式变化）→ 开

### 深度层内容要求

**D3 原理层**：每个核心技术/工具，2-3句解释底层机制 + 和替代方案的架构差异 + "知道这个你会做的不同决策"。
- 示例（FastAPI 异步原理）：事件循环单线程靠 await 让出控制权并发→所以在 async 里调同步阻塞代码会卡住所有请求。
- **禁**：只复述"X 是一个 Y 框架"这种定义。

**D4 演变层**：3-5句叙事（不是时间线列表），回答：从哪来（1-2步演变）+ 现在什么阶段 + 近期最值得关注的变化（带版本/时间）。

**D5 心智模型（★防兑水护栏）**：3-5层导航框架，必须过**反直觉测试**：
```
反直觉测试:
  问: 如果把这个分层换成"目录/章节标题"，读者预期会变吗？
  → 不变 = 这是目录不是心智模型，删掉重写
  → 变了 = 有洞察，保留

❌ 目录式（删）: "Python → 标准库 → 第三方库 → 框架 → 部署"
✅ 洞察式（留）: "代码层→框架层→运维层，跳过任何一层你都不知道为什么出bug"
```
心智模型放在"概述之后、正文之前"，作为全篇导航地图。

**D6 争议/权衡**：1-2句嵌入对应推荐条目（不需要独立章节）。回答：这个推荐放弃了什么 + 有没有人不这么想 + 什么情况下不该用。
- **防硬编**：不是每个推荐都要编一个"但是"——只在真有争议/权衡时写。判断：Research 是否搜到对立观点？没有 → 不写。

**D7 方向层**：文档末尾，3-5句回答：最值得关注的方向（2-3个趋势）+ 哪些当前做法可能被淘汰 + 读者该关注什么信号。

### 深度层验证职责（挂 Phase6，不挂 Gate/阶段B）

> 对抗审查#9 修正：阶段B 只查大纲结构（槽位齐全），查不到内容质量；深度层质量验证挂 Phase6 判官。

```
Gate 阶段: 只查条目级质量（1-11门禁），不查深度层
阶段B: 只查大纲深度层槽位齐全（该有的层在大纲里有位置）
Phase6 判官（★v1.11 扩展）: 查深度层质量
```

### ★v1.11.1 教程模式执行细则

> 源于同题对照：cc-bestpractice 默认产出400行精华版，缺橙皮书§03那种"手把手带做完整项目"的陪练感。教程模式补这个缺口。

**激活判定（Parse 阶段）**：
- 用户说"教程/上手/从零/带项目/手把手/陪练/新手" → 激活
- 文档类型 domain-guide 且检测到新手信号（用户问"怎么开始""第一个项目"）→ 激活
- 纯规范/检查清单 → 不激活

**教程模式产出**（在完整文档基础上，额外加一个章节）：

```markdown
## 实战项目：从零做一个 {领域典型项目}

### 项目选型
- 选领域典型场景（FastAPI→AI新闻聚合器；Claude Code→第一个CLI工具；React→todo app）
- 复杂度适中：半天到一天能做完，但涉及领域核心能力（网络/数据/AI调用/文件IO）
- 明确终态：做完能跑、能用、能演示

### 第一步：告诉 Claude 你要什么（Describe）
{可直接粘贴的完整 prompt，含具体需求 + 技术选型 + 终态描述}
{预期：Claude 会先给方案，不是直接开写}

### 第二步：审查方案（Review the Plan）
{Claude 会给出什么样的方案}
{你该怎么反馈：补充/调整的具体话术示例}

### 第三步：看它干活（Watch it Work）
{执行过程会发生什么：初始化/创建文件/跑测试}
{权限提示怎么处理：初期每次看，熟了开Auto}

### 第四步：验证结果（Verify）
{具体验证命令 + 预期输出，可证伪}
{报错了怎么办：把错误粘给Claude}

### 第五步：迭代改进（Iterate）
{加功能的具体 prompt 示例，2-3个递进改进}
```

**教程模式质量要求**：
- 每步给**可粘贴的 prompt**（读者照抄能跑）
- 每步给**预期产出**（读者知道做对了没）
- **心态转变点**：点出"你的角色从X变成Y"（呼应橙皮书"从写代码的人变成给指令的人"）
- 末尾点出底层模式（"描述需求→审查方案→确认执行→验证结果→迭代改进"这个循环适用于任何项目）

## 与套件协同接口

cc-bestpractice 消费 cc-scanner/cc-2pp 的产出，产出被 cc-config/cc-loop/cc-goal 消费。每个接口写明数据格式、读取方式、消费时机、可选/必需。

### cc-scanner 消费接口

- **方向**：scanner → bestpractice
- **数据格式**：`.claude/skills-kb.json`（技能清单 + 分类 + 触发词）
- **读取方式**：`Read .claude/skills-kb.json`
- **消费时机**：Research 阶段 Route B（代码扫描），用于填充文档的"工具推荐 / 技能推荐"章节
- **映射规则**：技能清单按领域分类 → 文档"工具与资源"章节列出相关技能 + 触发词
- **可选/必需**：**可选**（项目规范场景必需，纯领域指南场景可跳过）

### cc-2pp 消费接口

- **方向**：2pp → bestpractice
- **数据格式**：`.2pp/{run}/50-decision.md`（设计决策记录）
- **读取方式**：`Read .2pp/{run}/50-decision.md`
- **消费时机**：Research 阶段 Route C（用户意图深化），当从设计决策生成架构指南时
- **映射规则**：2pp 的每条决策 → 架构指南的 ADR 章节（上下文/决策/理由/后果）
- **可选/必需**：**可选**（"从设计决策生成架构指南"场景必需）

### cc-config 产出接口

- **方向**：bestpractice → config
- **数据格式**：生成的规范文档 → 写入 CLAUDE.md / AGENTS.md 的规则片段
- **读取方式**：cc-config 读取 bestpractice 生成的文档
- **消费时机**：用户要求"把规范写入 CLAUDE.md"时
- **映射规则**：
  - **适合写入 CLAUDE.md**：短规则（≤3 行）、强制禁令、命名约定
  - **不适合写入**（只留在 bestpractice 文档）：长示例、完整反模式表、迁移指南
- **可选/必需**：**可选**（用户显式要求时）

### cc-loop 产出接口

- **方向**：bestpractice → loop
- **数据格式**：最佳实践文档 → 循环 prompt 的内容源
- **读取方式**：cc-loop 的循环合同引用 bestpractice 文档路径
- **消费时机**：用户要求"定期检查代码是否符合最佳实践"时
- **映射规则**：循环合同 = `TRIGGER=定时, ACTION=Read 文档 → Grep 代码 → 报告偏离`
- **可选/必需**：**可选**

### cc-goal 协同接口

- **方向**：bestpractice → goal（双向）
- **数据格式**：Gate 门禁条件 → cc-goal 的终态条件模板
- **读取方式**：把 5 项门禁的通过标准转写成 `/goal` 的可验证条件
- **消费时机**：用户要求"用 /loop 自动补强文档直到达标"时
- **映射规则**：
  - 门禁1（具体性）→ 终态条件："≥90% 推荐有具体项"
  - 门禁4（反模式）→ 终态条件："反模式清单 ≥8 条"
- **可选/必需**：**可选**（高级场景）

## 上下文预算控制

控制每阶段 token 消耗，防单次运行爆炸。静态预算（技能文件）+ 运行时预算（执行）+ 保护机制三层。

### 各阶段 token 预算表

| 阶段 | 执行者 | 预算 | 说明 |
|------|-------|------|------|
| Parse | 编排者 | ~500 token | 内部推理，无工具调用 |
| Route | 编排者 | ~300 token | 查路由表 + 轮次映射 |
| Research (web 重) | Explore(haiku) × 4 轮 | ~12k | 搜索+深读+返回摘要 |
| Research (web 中) | Explore(haiku) × 2 轮 | ~6k | 搜索+返回摘要 |
| Research (code) | Explore(haiku) × 1 | ~3k | 扫描+返回摘要 |
| Research (user) | 编排者 | ~500 token | 1-2 个 AskUserQuestion |
| Generate | general-purpose(opus) × 1 | 15-25k | 读摘要+模板+写文档 |
| Gate | 编排者 | ~2k | Read 文档 + 5 项检查 |
| Gate 补强（最多1轮） | general-purpose(opus) × 1 | ~8k | 补强失败章节 |
| 增量融合（mode=update） | 编排者 + agent | 8-15k | 只搜增量 + 精准融合 |

### 场景预算估算

| 场景 | 总 token | 总轮次 |
|------|---------|--------|
| "Python 最佳实践"（web 重 + code 轻） | ~25-35k | 3 编排 + 1 agent + 1 生成 |
| "Python 编码规范"（web 中 + code 轻） | ~15-25k | 2 编排 + 1 agent + 1 生成 |
| "项目 Python 规范"（web 轻 + code 重） | ~18-28k | 2 编排 + 1 agent + 1 生成 |
| 增量更新 | ~8-15k | 1 编排 + 1 搜索 + 1 融合 |

### 预算保护机制（硬约束）

```
Web 搜索:    最多 4 轮（20 次搜索），达上限强制停止
代码扫描:    最多读 30 个文件签名（不读实现）
Generate:    单次文档 ≤ 5000 行
Gate 补强:   最多 1 轮，防止无限循环
Research:    每路 agent 返回摘要 ≤ 2000 token（超长截断）
★v1.11 深度档: 最多 6 轮（30 次搜索），M2/M4 按需触发
```

**触发保护时**：记录降级原因到研究摘要，告知用户"因预算限制，X 部分覆盖较浅"。

### ★v1.11 预算分级（用户按需选，对抗审查修正#7）

> v1.11 修正：旧版只有"web重/中/轻"隐式分级。新版显式分三档，让用户按文档重要性选，避免轻量文档跑深度档浪费、重要文档跑轻量档深度不足。

| 档位 | Web搜索 | 信息源 | 深度层 | 适用场景 | 估算token |
|------|---------|--------|--------|---------|-----------|
| **轻量档** | 4轮20次 | M1+M3 | 不强制 | 短文档/检查清单/速查卡 | ~15-25k |
| **标准档（默认）** | 5轮25次 | M1+M3+M2(降级)+M5/M6按需 | 原理+心智模型 | 领域指南/架构指南常规 | ~25-40k |
| **深度档** | 6轮30次 | 全7源(M4按需) | 全深度层 | 重要领域/长文档/用户要求 | ~40-60k |

**档位判定（Parse阶段）**：
- 用户显式说"深度/详细/全面" → 深度档
- 用户显式说"速查/检查清单/简短" → 轻量档
- 默认 → 标准档
- 文档类型自动加权：领域指南/架构指南 → 至少标准档；检查清单 → 轻量档

**M4 多模型触发规则（按需，非默认）**：
- 仅当置信度冲突（多源说法矛盾）/ 深度不足 / 用户要求深度档 时才跑
- 每多一个模型 = 一次完整往返，显著拖慢生成
- 标准档默认不跑 M4

### 静态预算（技能文件本身的加载成本）

| 文件 | 估算 token | 加载时机 |
|------|-----------|---------|
| SKILL.md | ~2k | 触发时按需 |
| bestpractice-guide.md（本文件） | ~6k | 深度参考按需 |
| 单个模板 | ~1k | Generate 时按需 |

每次运行典型加载：SKILL.md(2k) + guide(6k) + 1 模板(1k) ≈ 9k 静态开销。

## 附录D：完整骨架产物规范

> ★v1.1 新增章节。这是门禁6（可运行性门禁）的依据产物。对抗验证证明：条目级门禁（1-5）全过，文档仍可能"看起来对，复制下来全错"——因为没有可运行的完整骨架，import 链断裂，用户连 hello world 都跑不起来（攻击B 4 个 CRITICAL 全源于此）。

### 为什么需要完整骨架

对抗验证暴露的核心结构性缺陷：**条目级门禁保不了集成级可运行性**。

- 攻击B #1：1066 行文档，目录树给了，但没有一个完整可运行的 `main.py`，`uvicorn app.main:app` 永远跑不起来
- 攻击B #6：全文反复引用 `pyproject.toml`（`uv pip compile`、CI、Docker `COPY`），却从头到尾没给 `[project] dependencies` 完整内容——用户拿什么 compile？
- 攻击B #8：`get_settings`/`get_async_db`/`get_user_by_id`/`User` model/`Base` 等 6-8 个胶水函数全文未定义，import 链是断的

结论：文档必须含一个能从「`git clone` / 建目录」走到「`curl /health` 通」的最小骨架仓库，作为附录D 产出。骨架不是业务示例——业务在正文，骨架只保证「跑得起来」。

### 附录D 必含文件清单（按项目类型）

| 文件类别 | Python/FastAPI | Node/TS | Rust |
|---------|---------------|---------|------|
| 入口文件 | `app/main.py`（`FastAPI()` + 至少 1 个 endpoint + `if __name__`） | `src/index.ts`（server listen） | `src/main.rs`（`#[actix_web::main]` / `#[tokio::main]`） |
| 依赖清单 | `pyproject.toml`（完整 `[project]` + `[project.optional-dependencies] dev`） | `package.json`（完整 `dependencies` + `devDependencies`） | `Cargo.toml`（完整 `[dependencies]` + `[dev-dependencies]`） |
| 配置文件 | `.env.example` + `core/config.py`（`get_settings` 实现，含默认值） | `.env.example` + `src/config.ts` | `.env.example` + `config.rs` |
| 胶水函数 | `api/deps.py`（`get_db`/`get_current_user` 等 Depends 函数完整实现） | `src/deps.ts`（factory/helper 函数） | `src/deps.rs` |
| 测试 fixture | `conftest.py`（DB fixture + `dependency_overrides` 的 autouse 清理） | `test/setup.ts` | `tests/common/mod.rs` |
| 启动验证 | 1 行 `uvicorn` + 1 行 `curl` + 期望输出 | 1 行 `npm run dev` + 1 行 `curl` | 1 行 `cargo run` + 1 行 `curl` |

### 自洽性要求（硬约束）

1. **import 链自洽**：附录D 的文件互相 import 必须成立，不得有断链。正文引用的所有胶水函数（`get_db`/`get_settings`/`User` model/`Base` 等）必须在附录D 有完整实现
2. **依赖清单完整**：`pyproject.toml`/`package.json`/`Cargo.toml` 必须含跑通骨架所需的全部依赖（含部署相关如 gunicorn/uvicorn worker，攻击B #4：CMD 用 gunicorn 却没在依赖里列出）
3. **配置可加载**：`.env.example` 必须提供所有必填环境变量的样例值（攻击B #8：`Settings` 依赖 `DATABASE_URL`/`SECRET_KEY` 无默认，不配 `.env` 直接崩）
4. **测试隔离**：conftest.py 必须含 `dependency_overrides` 的清理（autouse fixture + `app.dependency_overrides.clear()`），防跨文件测试污染（攻击A #7）
5. **命令可执行**：启动命令必须在 Docker `RUN` 默认 shell（`/bin/sh`）下成立——禁用 bashism 如进程替换 `<(...)`（攻击A #5/B #4）

### 最小骨架原则

- **只含跑通 hello world 所需**，不堆业务逻辑。业务示例放正文，骨架只保证「能启动 + 能 curl 通」
- **同步/异步二选一贯穿全文**，不要正文用异步、骨架用同步（攻击B #3：测试用同步 SQLAlchemy 1.x，正文其他章节用 2.0 async，用户不知该信哪套）
- **版本号统一**：骨架的依赖清单版本号与正文声明一致（攻击B #2：0.110/0.111/v2.7 三处版本号不一致）

### 附录D 示例结构（Python/FastAPI）

```

├── core/
│   ├── config.py        # Settings(BaseSettings) + get_settings()
```

## 博客模式风格规范（★v1.2 新增）

> 源于 cc-2pp 发布审查：同一份技术内容，"规范文档质量"≠"博客文章质量"。用户说"博客/发布/blog/写给人看"时切到博客模式，做**风格变换**（不是降质，技术准确性7项门禁同样必须过）。

### 触发判定（Parse 阶段）

用户请求含以下信号 → `style=blog`，否则默认 `style=spec`（规范）：
- 显式词："博客 / blog / 发布 / 文章 / 写给人看 / 可读性 / 叙事"
- 场景词："分享 / 技术分享 / 推广 / 公众号 / 知乎 / Medium"
- 反信号（保持规范）："规范 / 团队标准 / 内部文档 / 检查清单 / CLAUDE.md"

### 博客模式 vs 规范模式（风格对照）

| 维度 | 规范模式 | 博客模式 |
|------|---------|---------|
| **口吻** | 权威指令（"必须/禁止/最佳"） | 叙事引导（"我们遇到过 X，建议 Y"） |
| **作者声音** | 无第一人称 | 有作者视角 + 引子 + "为什么写这篇" |
| **证据标注** | `[置信度:★★★]` 内联每条 | 移到文末"参考来源"，正文不打元标签 |
| **结构** | 章节三件套（装/写/改） | 故事线 + 决策点，每节有"为什么这样选" |
| **开头** | frontmatter + 概述 | **TL;DR 速览**（3-5 行抓住读者） |
| **标题** | "§3.2 代码组织" | "为什么我们放弃了全局变量"（问题式/悬念式） |
| **元数据** | 完整技能元数据 | 移除 cc-bestpractice 内部元数据，只留作者/日期/版本 |
| **篇幅** | 无上限（含完整附录D） | 精简到 150-250 行，附录D 移到"完整代码见仓库链接" |
| **结尾** | 检查清单 | "我们学到的 + 下一步 + 讨论" |

### 博客模式渲染规则（Generate 阶段强制）

1. **开头必加 TL;DR**：3-5 行，回答"这篇讲什么 / 读完能得到什么 / 适合谁"
2. **每节用"问题→教训→做法"叙事**：先讲为什么（踩的坑/常见误解），再讲怎么做
3. **删除内联置信度标签**：所有 `[置信度:★★★]` 移除，置信度信息融入正文自然语言（"这是官方推荐"比"[置信度:★★★]"可读）
4. **标题去编号化**：`## 3.2 代码组织` → `## 别再 import \* 了：代码组织的三条铁律`
5. **移除内部术语**：禁出现"对抗验证/门禁6/cc-bestpractice/template@x.x"等技能内部概念
6. **附录D 精简**：博客正文只放**最小可读片段**，完整骨架改为"完整代码仓库：[链接]"
7. **加作者署名块**：文末"作者 / 审阅 / 最后更新 / 反馈渠道"
8. **免责声明**（涉及安全/生产代码时）：加"示例用于演示，生产部署请审计"

### 博客模式质量自检（Gate 阶段补充）

博客模式下，7 项门禁照常过，**额外加 3 项博客质量自检**：

| 自检项 | 通过标准 |
|--------|---------|
| **叙事性** | 开头有引子/TL;DR；≥3 处用"为什么"解释（不是只给结论） |
| **去机器化** | 0 处 `[置信度:` 内联标签；0 处"对抗验证/门禁/cc-bestpractice"内部术语 |
| **可读篇幅** | 正文（不含代码仓库链接）≤ 250 行；附录D 移外部 |

### 反例对比（同一个知识点两种模式）

**规范模式**：
```
### 3.5 异步模式
判断标准 [置信度:★★★] [来源: fastapi.tiangolo.com/async/]:
- I/O 密集 + 异步库 → async def
反例：async endpoint 调 requests.get → 阻塞事件循环
```

**博客模式**：
```
## 别在 async 函数里写同步代码——这是我见过最多的 FastAPI 性能坑

很多教程上来就教你 async def，但没人告诉你：如果你的 async 函数里
调了一个同步阻塞操作（比如 requests.get），整个事件循环会卡死，
所有其他请求都在排队等它。

怎么判断该不该用 async？记住一条：
- 你的函数里有 await（异步 I/O）→ 用 async def
- 没有 await，纯 CPU 或同步库 → 用普通 def，FastAPI 会自动丢进线程池

（代码示例：❌ async 里调 requests / ✅ 换 httpx 异步）

> 这是 FastAPI 官方反复强调的点（来源：fastapi.tiangolo.com/async/）。
```

博客版特点：有引子（"很多教程…"）、问题导向标题、置信度变自然语言、叙事流动。

### 与规范模式的切换

- 同一份研究素材（Research 结果）可**同时产出**规范版（给团队）+ 博客版（给社区）
- 博客版不重新研究，只**重新表达**——技术事实不变，叙事方式变
- 用户说"两版都要" → Generate 阶段产出两个文件 `{domain}-spec.md` + `{domain}-blog.md`

## 门禁 8/9 细则 + 博客诚实性规范（★v1.3 新增）

> 源于博客版对抗验证：v1.2 博客模式让 AI 学会了"博客口吻"，但暴露两个新盲区——AI 会**虚构人类经验**、代码片段会**残缺不可跑**。v1.3 补门禁 8（诚实性）+ 门禁 9（片段可跑）。

### 门禁 8：诚实性门禁（博客模式必过）

**检查什么**：博客模式下，文档是否虚构了人类经验/人设。
- ❌ 禁止：无真人背书时使用"我踩坑 N 年""我和团队在生产中""我们曾经"等虚构第一人称叙事
- ❌ 禁止：标题用"踩坑两年/实战总结"暗示人类经验，署名却是 AI
- ✅ 允许（路线A 诚实AI）："本清单由 cc-bestpractice 基于官方文档与社区共识整理"
- ✅ 允许（路线B 真人背书）：真人署名 + 该真人逐条审过 + 能为每条负责

**通过标准**：0 处虚构人类经验叙事；署名与实际生成方式一致（AI 生成就标 AI，真人审就标真人）

**失败处理**：删除虚构人设，改为诚实表述；或找真人审阅署名

**示例对比**：
- ❌ 失败：标题"FastAPI 踩坑两年"，正文"我和团队踩出来"，署名"cc-bestpractice 技能生成"——信任自杀
- ✅ 通过：标题"FastAPI 常见踩坑清单"，正文"基于官方文档与社区实践整理"，署名"AI 辅助 + 待审/已审 by XXX"

### 门禁 9：片段可跑门禁

**检查什么**：每个代码块是否自洽可跑，或显式标注"片段"。
- ❌ 禁止：凭空出现的标识符（未 import 的函数/未定义的类，如 `get_remote_address`/`AsyncSessionLocal`/`UserRead`）
- ✅ 允许A：代码块自洽（含必要 import + 工厂定义 + 类型注解 import）
- ✅ 允许B：显式标注"片段，完整见仓库/附录D" + 关键标识符在附录D有完整实现

**通过标准**：每个代码块的标识符要么在块内 import，要么在附录D定义，要么标注来源

**失败处理**：补 import；或补工厂定义；或加"见附录D.X"指针

**示例对比**：
- ❌ 失败：`limiter = Limiter(key_func=get_remote_address)` 但没 import get_remote_address
- ✅ 通过：`from slowapi.util import get_remote_address` + `limiter = Limiter(...)` 或标注"完整挂载见附录D"

### 门禁 7 强化：事实时效（v1.1 版本号 → v1.3 所有可核查事实）

v1.1 只查版本号，v1.3 扩展到**所有可核查事实**：

| 事实类型 | 核查方式 | v1.3 要求 |
|---------|---------|----------|
| 版本号 | WebSearch `{framework} latest {year}` | 落后实际≤2 minor |
| CVE 编号 | WebSearch 核实存在 + 描述对应 | 禁凭记忆编 CVE |
| OWASP/标准排序 | WebSearch 官方 cheat sheet | 禁写反顺序 |
| API 弃用/移除时间线 | WebSearch release notes | 禁凭记忆断版本 |
| 性能数字 | 标复现条件或删除 | 禁无来源夸大 |

**新增反模式**（v1.3）：
- 凭记忆写 OWASP 排序（argon2>bcrypt>scrypt 实际是 Argon2id>scrypt>bcrypt>PBKDF2）
- 凭记忆写 CVE 编号（可能记错或编造）
- 无复现条件的性能数字（"万级 RPS"无机器/参数）

### 博客诚实性规范（博客模式渲染补充）

博客模式下 Generate 阶段额外遵守：

1. **禁虚构第一人称经验**：用"开发者常遇到""社区共识是"替代"我遇到过"
2. **标题去人设化**：禁"踩坑 N 年/实战/我的"，用"常见清单/避坑/2026 基线"

4. **免责前置**：不只在结尾免责，开头就声明"基线 + 安全决策请结合团队标准"
5. **代码块自洽或标注**：见门禁 9，每个代码块要么能跑要么标"片段"

|--------|---------|

## 门禁 10 幽灵资源 + 语义层门禁细则（★v1.4 新增）

> 源于第4轮对抗验证：v1.3 门禁"升级了形式，没升级到语义"。前9项抓"有没有标来源/import"，v1.4 抓"来源是否支撑声明 + 引用资源是否真实存在"。这是从形式层到语义层的跃迁。

### 门禁 7 再强化（v1.4）：从"编号存在"到"声明真实性"

v1.1 查版本号、v1.3 查 CVE 编号存在——都是**形式层**。v1.4 升级为**语义层**：核实声明本身是否成立。

| 声明类型 | 形式层（旧） | 语义层（v1.4） |
|---------|------------|--------------|
| "X 已迁移到 Y" | 查 X 有没有迁移讨论 | **确认真迁移了**（官方文档当前页面用的就是Y，不是社区提案/讨论） |
| "X 背着未修的 CVE" | 查 CVE 编号存在 | **查 CVE 当前状态**（已修/未修/哪个版本修） |
| "X 是官方推荐" | 查官方有提过 X | **确认当前官方文档明确推荐**（不是历史推荐/边缘提及） |
| "因为 A 所以 B" | 查 A 存在 | **A 真的导致 B 吗**（若 A 已失效，因果崩塌） |

**v1.4 语义层三问**（Gate 阶段对每个承重声明追问）：
1. 来源支撑的是"已X"还是"建议X"？（提案≠落地，讨论≠结论）
2. 这个因果（因为A所以B），A 现在还成立吗？（A已失效→因果崩）
3. 我引用的数字/版本/状态，是当前的还是历史的？

**反模式（v1.4 新增）**：
- 把 Discussion/Issue 提案当成"已落地事实"（如把"建议弃用jose"的讨论当成"官方已弃用"）
- 把"曾经未修"当成"现在未修"（CVE 已在后续版本修复）
- 凭记忆断"当前官方推荐"（官方文档可能已改）

### 门禁 8 强化（v1.4）：免责上限 + 禁凑数整数

v1.3 门禁8 只管"虚构人设"，v1.4 补两条：

**免责上限**：单篇免责声明 ≤2 处（开头1次+结尾1次）。
- ❌ 反模式：TL;DR免责 + 免责前置 + 关于本文 + 结尾免责 + 署名免责 = 5处堆砌，等于不自信
- ✅ 正例：开头1次"基线+审阅状态"，结尾1次"安全审计提醒"，足矣
- 理由：多次免责反噬可信度，读者觉得"作者这么心虚，内容能信吗"

**禁凑数整数**：标题的"N条/N个点"必须=实质条目数。
- ❌ 反模式：标题"12个点"但第10/12条明显薄于前几条（凑数）
- ✅ 正例：要么每条同等深度，要么诚实砍到"10个点"，要么合并弱条目
- 检查：每条的字数/代码量/深度是否均衡；不均衡→砍或合并，别保整数

### 门禁 10：幽灵资源门禁（★v1.4 全新）

**检查什么**：文档引用的所有外部资源是否真实存在。

| 资源类型 | 检查方式 | 失败示例 |
|---------|---------|---------|
| 仓库 | "见仓库app/目录" → 必须真有仓库或真URL | 独立md文件却暗示有配套仓库（克隆不到） |
| 附录 | "见附录D" → 必须真有附录D | 正文引用附录但附录不存在 |
| 链接 | 每个URL可访问 | 死链/拼错（如 fastai-best-practices 漏 p） |
| 版本号 | `pip install X==Y` 装得到 | 推荐不存在的版本/未发布版本当稳定版 |
| 文件路径 | "app/main.py" → 仓库里真有 | 画了目录树但文件不存在 |

**通过标准**：每个被引用的资源，读者按指引真的能拿到（仓库可克隆/附录可读/链接可访问/版本可装/路径可找）

**失败处理**：
- 资源不存在 → 二选一：(a) 真的提供资源（补附录/给真URL）；(b) 删除引用，改为"本文为示意，读者自行组装"
- 禁止"既不提供又暗示存在"——这是最隐蔽的不诚实

**示例对比**：
- ❌ 失败：独立博客md，4处暗示"配套仓库app/目录"，但没给URL也没附录——读者克隆不到
- ✅ 通过：要么文末给真GitHub URL，要么删所有"见仓库"表述改"本文代码为示意片段"

| 叙事性 | 开头有引子/TL;DR；≥3处用"为什么" |
| 去机器化 | 0处`[置信度:`内联；0处内部术语 |
| 可读篇幅 | 正文 ≤ 250行 |
| 诚实性 | 0虚构人设；署名一致 |
| 片段可跑 | 0凭空标识符；自洽或标注 |
| **声明真实性** ★v1.4 | 每个"已X"声明，来源支撑"已X"非"建议X"；每个因果，前提当前成立 |
| **无幽灵资源** ★v1.4 | 引用的仓库/附录/链接/版本/路径全部真实可取；免责≤2处；标题整数=实质条目 |

## 最佳实践的本质（★v1.6 元章节 — 目的层）

> 源于一次元反思："最佳实践文档应该是什么样子的？"——这个问题逼 cc-bestpractice 从"门禁设计者"退到"本质思考者"。结论：11 项门禁是手段，"让读者获得判断力"才是目的。本章节锚定目的，防止技能陷入"为门禁而门禁"的形式主义。

### 一句话定义

> 最佳实践文档 = **让读者用 10 分钟，获得作者用 10 个月踩坑换来的判断力**。

关键词每个都对应一次对抗验证教训：
- **判断力**（不是知识）→ 门禁11 Why：读者要能判断边界，不只记规则
- **踩坑换来的**→ 门禁3/7：每条有真实来源，不是编的
- **可复用**→ 门禁6/9：能跑、能抄、能验证
- **固化**→ 门禁10：资源真实，不指向幽灵
- **10分钟**→ 门禁8：克制，免责≤2处，不啰嗦

### 它不是什么（5 个反例，每个对应一代进化的坑）

| ❌ 不是 | 实质 | 对应教训 |
|--------|------|---------|
| 规范集合 | 清单，读者照做不知为何 | v1.5 门禁11（缺 Why） |
| 官方文档翻译 | 复读机，无增量价值 | v1.0 踩坑实证缺失 |
| 作者经验炫耀 | 不可信不可复现 | v1.3 门禁8（虚构人设） |
| 永远正确的废话 | 等于没说 | v1.0 门禁1（具体性） |
| 静态知识 | 会过时 | v1.1 门禁7（时效） |

### 它是"被对抗过的判断力沉淀"

每条规范应经得起读者 **5 个追问**——这是"最佳实践" vs "一个做法"的分界线：

| 追问 | 读者视角 | 对应门禁 | 答不出 = |
|------|---------|---------|---------|
| **为什么？** | 凭什么是最佳 | 门禁11 Why完备 | 清单 |
| **真的吗？** | 这事是真的吗 | 门禁7 事实真实 | 编造 |
| **能跑吗？** | 抄了能跑吗 | 门禁6/9 可运行/片段 | 误导 |
| **过时吗？** | 现在还对吗 | 门禁7 时效 | 毒资产 |
| **凭什么信你？** | 作者可信吗 | 门禁8/10 诚实/幽灵 | 信任崩塌 |

**5 个追问全答得出 = 配叫"最佳实践"**；任何一个答不出 = 降级为"一个做法"。

### 6 个视角的"应该"（判官小组思维）

| 视角 | 应该 | 反例 |
|------|------|------|
| 产品 | 解决读者的真实问题（带问题来，拿答案走） | 按作者知识结构组织 |
| 架构 | 让读者敢信（置信度+反例） | 全篇"必须/最佳"无差别 |
| 测试 | 让读者能验证（骨架+可证伪检查） | 只有片段没骨架 |
| 进化 | 承认会过时（时间锚+更新机制） | 假装永远正确 |
| 诚实 | 不装（不虚构/不夸大/免责克制） | "踩坑两年"却署名AI |
| 本质 | 每条答"为什么"（Why三问） | 只有What+How |

### 最高判断：灵魂是"被对抗过"

> 一份没被对抗验证挑过刺的最佳实践，不配叫"最佳"——它只是"一个做法"。

真正的最佳实践文档，应该有**被攻击的痕迹**：每条规范旁边能隐约看到"这条曾受质疑但活下来了"或"这条因被攻击而改进了"。文档不是一次性产物，是**对抗-固化-再对抗**的沉淀。

cc-bestpractice 自己就是例子：5→11 门禁，每一项都是一次"被自己攻击后活下来的教训"。

### 元启示：门禁是手段，判断力是目的

11 项门禁存在的**唯一理由**，是逼每条规范答得出 5 个追问。如果某条规范过了所有门禁，但读者读完仍没有判断力（不能判断边界/不能教别人/遇到新情况不会迁移），那这份"最佳实践"还是失败了。

**自检（每次生成后）**：随机抽 1 条规范，问自己"读者看完这条，遇到一个文档没覆盖的相关情况，能自己推导出正确做法吗？"——能，才是判断力；不能，只是记忆。

### v1.0→v1.6 完整进化（目的层视角）

```
v1.0 (5门禁)   条目层 What对不对
v1.1 (7)       集成层 能不能用
v1.2           风格层 给谁看
v1.3 (9)       诚实层 真不真
v1.4 (10)      语义层 声明成立吗
v1.5 (11)      本质层 为什么
v1.6           ★目的层 为了让读者获得判断力
```

**v1.6 是元层**：不是加第12个门禁，而是给前11个门禁一个"为什么存在"的锚——防止门禁膨胀失控，确保每个门禁都服务于"读者判断力"这个终极目的。

## 5 步写作工作流（★v1.7 生产工艺）

> 源于"五星级最佳实践写作工作流"。cc-bestpractice 一直有 11 项**验收门禁**（事后检查），但缺**生产工艺**（事中怎么做）。v1.7 补上——Generate 阶段从"agent 自由发挥"收敛成 5 步流水线。生产服务于验收，验收校准生产。

### 核心关系：生产工艺 vs 验收门禁

```
Generate（生产）           Gate（验收）
  5步写作工作流    ──服务──►   11项门禁
  事中怎么做                  事后查什么
       ▲                         │
       └─────── 校准 ────────────┘
  （门禁发现的新盲区 → 固化为工作流的新步骤）
```

### 5 步详解

#### ① 定灵魂（黄金圈 Why）
- **做什么**：明确本文解决的**终极痛点**（Why），写进开篇引言的**最后一句**
- **对应门禁**：11 Why完备
- **为什么这步第一**：灵魂不定，后面 4 步都是无根之木。黄金圈（Why→How→What）让人说人话
- **反模式**：开篇写"本文介绍X的最佳实践"（没痛点）→ 读者不知道为什么要读

#### ② 切任务（任务导向）
- **做什么**：完备性驱动切任务——粒度由内容决定（**不设数量上限**，知识点多则主任务+子任务分层），用任务作二级标题（不是知识结构）
- **对应门禁**：5 可操作性
- **为什么用任务不用知识结构**：读者是来"办事"的，不是来"学知识体系"的。任务导向 = 办人事
- **反模式**：按作者的知识结构分章（§1概述§2原理§3实现）→ 读者找不到自己的问题

#### ③ 减负荷（认知负荷）★新维度
- **做什么**：每个任务画流程图，按流程图节点写步骤，**每步只做一件事**
- **对应门禁**：（之前无，v1.7 新增的认知负荷维度）
- **为什么这步重要**：cc-bestpractice 之前只管"信息全不全"，没管"读者读得累不累"。流程图+单步降低认知负荷
- **反模式**：一个步骤塞 3 件事 → 读者做到一半迷路

#### ④ 拆信息（DITA 结构）
- **做什么**：操作留正文，原理（>5行）剪到"背景知识"附录；正文只保留"操作"+"关键决策依据（Why this way）"
- **对应门禁**：6/9（骨架/正文分离）
- **5行阈值边界**（关键）：
  - **操作型文档**（教程/规范）→ 原理移附录（读者要办事不要理论）
  - **原理型文档**（方法论/Why 是核心价值）→ 保留 Why 正文，移的是"延伸背景"
  - **判断依据**：读者是来"办事"还是"理解"？
- **反模式**：原理型文档也把 Why 移走 → 伤判断力（门禁11 失守）

#### ⑤ 过筛子（同理心）
- **做什么**：通读删判断色彩词（显然/只需/很简单/只需三步）→ 换"建议/推荐/在X条件下"
- **对应门禁**：8 诚实性
- **为什么这步最后**：前4步写完才能通读筛词。这步是门禁8（诚实）的**操作化**——比"禁虚构人设"更细腻
- **反模式**："显然你应该X" → 读者觉得被居高临下；换"建议X（在Y条件下）" → 读者觉得被尊重

### 终极信条（贯穿 5 步）

> 最佳实践文档的最高境界，是让读者觉得"**我变聪明了**"，而不是"作者真厉害"。

- 黄金圈 → 让你说人话
- 任务导向 → 让你办人事
- 认知负荷 → 让你讲条理
- DITA → 让你好维护
- 同理心筛子 → 让你尊重读者

**5 步共同服务于这个信条**：每一步都在把焦点从"作者"转到"读者"。

### 工作流自检（Generate 后 Gate 前）

| 自检 | 通过标准 |
|------|---------|
| 灵魂句 | 开篇引言最后一句是"终极痛点"，不是"本文介绍" |
| 任务标题 | 二级标题是"任务"（读者能做什么），不是"概念"（章节名） |
| 单步 | 每步只做一件事，无嵌套多任务 |
| 信息分层 | 操作在正文，原理在附录（原理型文档 Why 保留正文） |
| 无判断词 | grep "显然\|只需\|很简单" 为 0 |

### v1.0→v1.7 完整进化（含生产工艺层）

```
v1.0 (5门禁)   条目层 What
v1.1 (7)       集成层 能不能用
v1.2           风格层 给谁看
v1.3 (9)       诚实层 真不真
v1.4 (10)      语义层 声明成立吗
v1.5 (11)      本质层 为什么
v1.6           目的层 判断力（元层）
v1.7           ★生产工艺层 怎么写（5步工作流）
```

**v1.7 补齐最后一块**：cc-bestpractice 现在三层完整——
- **本质层**（v1.6）：最佳实践是什么（让读者获得判断力）
- **生产工艺层**（v1.7）：怎么写（5步工作流）
- **验收层**（v1.0-v1.5）：怎么查（11门禁）

本质定方向，工艺管过程，验收保质量。三层闭环。

## Research 完备性自检（★v1.8 生产工艺强化）

> 源于"漏 ultrawork"案例。生成 OMC 文档时，任务二"选编排模式"只从 README 扫到 autopilot/ralph/team 就收手，漏了 ultrawork——经人类指出后 `ls skills/` 才发现。**集合格的东西最易漏**，v1.8 在 Research 阶段补完备性自检。

### 根因分析

为什么集合格容易遗漏：
1. **README 是叙述性的**，会提到几个典型例子，但不会穷举全部同级元素
2. **AI 倾向"够用就收"**，扫到 2-3 个能用的就停，不主动追问"集合完整吗"
3. **平级 vs 层级混淆**：autopilot/ralph/team 看着平级（都是模式），实际 ultrawork 和它们是层级包含关系（autopilot⊃ralph⊃ultrawork），AI 容易漏掉低层级元素

### 完备性自检规则（Research 阶段扫到任何集合时）

扫到这些"集合"时，必须做完备性自检：

| 集合类型 | 完备性验证方式 | 示例 |
|---------|--------------|------|
| 编排/执行模式 | `ls skills/` 或目录枚举 | OMC：ls 出 autopilot/ralph/ultrawork/team/ultraqa |
| agent 类型 | `ls agents/` | OMC：19 个 agent，别只写常用的几个 |
| 工具/库清单 | 官方 registry 或 `pip list`/`ls` | 别只列 README 提到的 |
| 反模式清单 | 自问"还有哪类坑没列" | 别凑到 8 条就停 |
| 配置项 | `cat config` 或源码枚举 | 别只写文档示例的项 |

### 自检三问（扫到集合必答）

1. **这个集合完整吗？** 有没有被遗漏的同级元素？
2. **我只看了 README 吗？** 有没有用源码/目录枚举交叉验证？
3. **这些元素是平级还是有层级/包含关系？** 别把不同层级的当平级写

### 答不出怎么办

- 只看了 README → 用 `ls`/`cat`/`find` 枚举源码目录，补全
- 不确定平级还是层级 → 查 SKILL.md/源码的关系声明（如 "ralph includes ultrawork"）
- 集合太大无法穷举 → 显式声明"本文只覆盖 X 类（完整清单见 Y）"，不能假装穷尽

### 反模式（v1.8 新增）

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| README 扫到 2-3 个就收手 | 漏同级元素 | `ls` 枚举源码目录交叉验证 |
| 把层级包含当平级写 | 读者误以为是并列选择 | 标注包含关系（A⊃B⊃C） |
| 凑够数量就停（反模式满8条） | 为凑数塞弱条目 | 宁可砍到少而精，不凑数 |

### 案例：漏 ultrawork → 补全

**错误过程**（v1.7 生成 OMC 文档）：
- README 提到 autopilot/ralph/team → 任务二写"3 模式"
- 没问"还有别的执行模式吗" → 漏 ultrawork
- 用户指出 → `ls skills/` 发现 ultrawork + ultraqa
- 深挖 SKILL.md → 发现 autopilot⊃ralph⊃ultrawork 层级关系（比单纯补全更有价值）

**正确过程**（v1.8）：
- Research 阶段 `ls OMC/skills/` → 看到 autopilot/ralph/ultrawork/team/ultraqa/deep-interview 全部
- 自检"编排模式完整吗" → 完整列表是这 4-5 个
- 自检"平级还是层级" → 读 SKILL.md 发现包含关系 → 标注层级

### 与门禁的关系

完备性自检是**生产工艺**（事中，Research 阶段），不是验收门禁（事后）。它防的是"集合遗漏"——这类问题事后 Gate 难查（读者不指出来，文档自己也发现不了），所以必须在事中 Research 阶段就堵住。

这也是 v1.6"防门禁膨胀"原则的体现：与其加第12个事后门禁（难自动检查"是否漏元素"），不如在事中工艺加一个"枚举完备性"动作（`ls` 可执行、可验证）。

### v1.0→v1.8 完整进化（含工艺强化）

```
v1.0-v1.5  验收层    11门禁（事后查）
v1.6       本质层    判断力（元层锚定）
v1.7       生产工艺层 5步写作工作流（事中怎么写）
v1.8       生产工艺强化 Research完备性自检（事中防遗漏）  ← 本次
```

**v1.8 不加门禁，强化工艺**——这符合 v1.6 的"防膨胀"原则。完备性问题本质是"事中没做对"，用工艺补比用门禁补更合适。

## 纯 web 一手源核实规则（★v1.9 生产工艺强化）

> 源于 Harness 文档对抗验证。v1.8 完备性自检假设有本地源码可 `ls`，但纯 web 项目（陌生开源/无本地代码）没有——完备性自检退化为"搜到几个写几个"，三类陈述失真。v1.9 补一手源核实规则。

### 根因分析

为什么纯 web 生成容易失真：
1. **搜索摘要 ≠ 一手源**：WebSearch 返回的是二手摘要/博客，官方 chart/release notes/UI 枚举页才是真相
2. **"听起来合理"被当成"官方这么说"**：AI 写作时把推导当事实，还自标 ★★★
3. **二手文章跟着错**：第三方博客常复制错误（如 "K8s Deployment" 在博客泛滥，但官方 chart 是 StatefulSet）

### 三类高风险陈述必须核实一手源

| 陈述类型 | 风险 | 一手源核实方式 | 不核实的后果 |
|---------|------|--------------|------------|
| **推导式陈述**（"为什么是 X 不是 Y"） | 推导当事实 | 找官方原文引用支撑；找不到 → **降置信度到 ★** | "听起来合理"≠"官方这么说"，因果错 |
| **枚举断言**（"N 种完备""只有这几种"） | 完备二字成本最高 | 核实官方枚举页（UI 值/chart templates/registry/`kubectl explain`） | 漏元素被打脸（如漏 Feature Flags） |
| **来源标注**（"官方/社区"） | 组织混淆 | 区分组织归属（`org` vs `org-community`） | 社区仓库标官方，违反置信度体系 |

### v1.9 核实三问（纯 web 场景必答）

1. **这条"为什么是 X"的推导，有官方原文支撑吗？** 没有就降 ★，别标 ★★★
2. **这个"N 种完备"的断言，核实过官方枚举页吗？** 没核实就别写"完备"
3. **这个标"官方"的来源，真是官方组织吗？** 社区组织（`*-community`）≠ 官方

### 什么算"一手源"（按权威性排序）

```
官方 chart/release notes（最权威，可验证资源类型/版本）
  > 官方 Developer Hub 文档（developer.xxx.io）
  > 官方 GitHub repo（org/repo 非 community）
  > 官方 UI 枚举/registry
  > 第三方博客/对比文章（最弱，仅参考）
```

### Harness 案例对照（v1.8 失守 → v1.9 该怎么做）

| Harness 文档的错误（v1.8） | v1.9 该怎么做 |
|--------------------------|--------------|
| "Delegate 3.x 是当前主版本"（实为 Closed Beta） | 核实官方 release notes → 真实 GA 是 NextGen 24.x/25.x |
| "Delegate 是 Deployment"（实为 StatefulSet） | 核实官方 chart templates/ → 是 StatefulSet |
| "部署策略 3 种完备"（漏 Serverless） | 核实官方枚举页 → 别写"完备"，标"K8s 下三种" |
| GitOps-Samples 标"官方"（实为社区） | 区分 `harness`(官方) vs `harness-community`(社区) |
| "Manager 进不了内网所以要 Delegate"（推导当事实） | 找官方 Delegate Overview 原文 → 多原因（安全+出站+权限） |

### 与 v1.8 的关系

```
v1.8 完备性自检: 有本地源码时用 `ls` 枚举（OMC 场景）
v1.9 一手源核实: 无本地源码时用 web 核实一手源（Harness 场景）
```

**v1.9 是 v1.8 在纯 web 场景的兜底**——v1.8 靠 `ls`，v1.9 靠"核实一手 web 源"。两者合起来覆盖"有源码"和"无源码"两种 Research 场景。

### 反模式（v1.9 新增）

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| 推导句标 ★★★ | "听起来合理"≠官方说法 | 找不到官方原文 → 降 ★ |
| 写"N 种完备"没核实枚举页 | 易被打脸 | 核实官方枚举，或改"N 种（K8s 下）"留边界 |
| 社区仓库标官方 | 违反置信度体系 | 区分 `org` vs `org-community` |
| 只信搜索摘要 | 二手源易错 | 追到一手源（chart/release notes） |

### v1.0→v1.9 完整进化（Research 工艺双覆盖）

```
v1.0-v1.5  验收层      11门禁
v1.6       本质层      判断力锚定
v1.7       生产工艺层  5步写作工作流
v1.8       Research强化 完备性自检（有源码场景：ls 枚举）
v1.9       Research强化 一手源核实（无源码场景：web 核实）  ← 本次
```

**v1.8 + v1.9 = Research 完备性的双覆盖**：有本地源码用 `ls`，纯 web 用一手源核实。不再有"完备性自检失效"的盲区。

## 渐进式生成（★v1.10 生产工艺 — Generate 三阶段）

> 源于用户需求："先生成目录，再逐步完善内容"+"大纲生成完通过头脑风暴自动审查调整后再逐段完善"。一次生成全文的痛点（本次对话实测）：①上下文爆炸（FastAPI 生成 272k > 202k 窗口报错）②结构错了才发现要重来 ③无早期反馈点。渐进式解：大纲廉价，全文昂贵，在大纲层先对抗。

### 三阶段流程

```
阶段A：生成大纲（5步的①定灵魂 + ②切任务）
   ↓ 输出：标题 + 任务列表 + 每任务一句话主旨 + 覆盖范围
阶段B：头脑风暴自动审查大纲（多维度挑刺 + 调整）★新增大纲对抗层
   ↓ 5维度都过
阶段C：逐段填充（5步的③减负荷 + ④拆信息 + ⑤过筛子，逐任务）
   ↓
全文 Gate（11门禁）
```

### 阶段A：生成大纲（机制详解）

大纲不是"套模板章节"，是"从痛点+Research 反推任务"。核心机制四步：

**步骤1：盘点知识域**（完备性，v1.8）
从 Research 结果列出项目**所有独立知识块**（能力/组件/命令/常见坑）。先求全，不设上限——这是"粒度尽可能细和完备"的源头。例：Harness 盘出 SCM/CI/CD/托管环境/制品仓库/Delegate/Pipeline/CV/GitOps 等十几块，不是硬压成 3-5 个。

**步骤2：按读者旅程分组**
把知识块按"读者从零到会用"的自然阶段归类——上手类/进阶类/运维类/排错类 → 每类一个**主任务**。

**步骤3：每主任务细分子任务**（知识点多时）
主任务下知识点多，就拆**子任务**保证粒度细，不硬合并丢内容。例：Harness "部署"主任务 → 子任务：选部署策略/选部署模式（命令式vs GitOps）/配 Verify。

**步骤4：MECE + 依赖排序**
去重（合并重叠块）、确保互斥（MECE）、按学习/操作顺序排（依赖前置）。

**粒度原则（★修正：完备优先，不设固定数量上限）**
- 粒度由**内容完备性**决定，不是"3-5 个"硬约束（v1.7 旧假设已修正）
- 知识点多 → 主任务 + 子任务**两层结构**（主任务做概览，子任务保细粒度）
- 知识点少 → 扁平几个主任务即可，不强求分层
- 判断标准：每个（子）任务 = 读者能独立完成的一件事 + 有独立 Why
- **为什么完备优先于简洁**：读者要的是完整判断力（v1.6 目的），不是被压缩的摘要；为"3-5"硬合并会丢内容、损判断力

**产出**：
- 文档标题（含 TL;DR 式终极痛点，v1.5 门禁11）
- 覆盖范围声明（覆盖什么/不覆盖什么）+ 目标读者
- 主任务列表（每个含：主旨一句话 + 可选子任务 + Research 信息映射）

**产出格式**（可分层）：
```
主任务1：XXX —— 主旨
  子任务1.1：XXX
  子任务1.2：XXX
  Research映射：← A/B组件
主任务2：XXX —— 主旨
  ...
```

**为什么大纲重要**：大纲是"结构验证"——廉价（几十行），错了改它比改全文（几百行）省 10× token。阶段B审查的主任务层，子任务跟随主任务一起审。

### 阶段B：头脑风暴自动审查（★核心创新）

**做什么**：大纲生成后，**自动**做一轮多维度审查（不等用户，审查完展示结果+调整后大纲）。

**为什么自动不等人**：
- 大纲层对抗廉价（改大纲 vs 改全文省 10× token）
- 符合 Loop Engineering "设计循环让循环审查"理念——把审查交给机制，不依赖人盯着
- 这是把 cc-2pp 对抗验证思路**前置到大纲层**（廉价层），而非全文（昂贵层）

**5 审查维度**（每个对应已有规则，不是新发明）：

| 维度 | 审查问什么 | 呼应 | 没过的调整 |
|------|-----------|------|-----------|
| 完整性 | 该有的任务都有吗？漏了什么同级元素？ | v1.8 完备性自检 | 补漏任务 |
| MECE | 任务互斥吗？有重叠？ | — | 合并/拆分 |
| 逻辑顺序 | 顺序合理吗？依赖关系对吗？ | — | 调序 |
| 读者价值 | 每个任务读者真需要吗？有凑数？ | v1.5 Why / v1.8 禁凑数 | 删凑数任务 |
| 边界 | 范围清晰吗？什么不覆盖？ | v1.6 目的锚定 | 明确边界/补范围声明 |

**通过标准**：5 维度都过，才进阶段 C。不过则调整大纲（附理由），再审。

**审查方式**（可选）：
- 轻量：编排者自审（按 5 维度逐条过）
- 重量：用 brainstorming 技能 或 多视角 agent 各审一维（大文档/复杂结构时）

### 阶段C：逐段填充（写作层）

**做什么**：按通过的大纲，逐任务生成完整内容。

**每段执行 5 步的**：
- ③减负荷：流程图 + 单步（每步一件事）
- ④拆信息：操作留正文，原理移附录（5行阈值边界）
- ⑤过筛子：去判断词（显然/只需→建议/在X条件下）

**一致性保证**（逐段生成的特有风险）：
- 术语统一（同概念用同词，别这段叫"技能"下段叫"能力"）
- 风格连贯（每段开头重读前段末尾，保叙事连贯）
- 前后呼应（前段提的概念后段要用上，别断链）

**降上下文压力**：逐段生成 = 每次只处理一段，避免一次塞全文（解 FastAPI 272k 爆炸）。长文档尤其受益。

### 为什么渐进式（Why 三问）

1. **Why 存在**：一次生成全文无早期反馈点，结构错要重来；渐进式提供大纲层廉价反馈
2. **Why 必要**（实测）：FastAPI 生成 272k > 202k 窗口报错；结构错了 300 行白写
3. **Why 选择三阶段**（vs 两阶段）：阶段B头脑风暴审查是关键——没有它，"大纲→填充"只是分两段一次生成，没有质量门禁；阶段B 让大纲层有对抗，问题在廉价层暴露

### 与现有架构的映射

渐进式**不取代** 5 步工作流，而是把它**沿时间轴展开 + 加大纲对抗点**：

```
阶段A = 5步的 ①定灵魂 + ②切任务
阶段B = 大纲层对抗（新，呼应 cc-2pp 思路 + v1.8 完备性）
阶段C = 5步的 ③减负荷 + ④拆信息 + ⑤过筛子（逐任务执行）
全文 Gate = 11 门禁（不变）
```

### 边界

- **短文档（预估 <150 行）**：可跳过阶段 B，A 完直接 C（大纲即全文骨架，审查成本不划算）
- **大纲粒度**：任务/章节级，不到小节
- **阶段B 审查深度**：小文档自审即可，大文档/复杂结构用多视角 agent
- **填充一致性**：逐段易风格漂移，C 阶段每段开头重读前段末尾

### 反模式（v1.10 新增）

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| 跳过阶段B直接填充 | 大纲错误传到全文，返工贵 | 阶段B审查（除非短文档） |
| 大纲粒度到小节 | 审查成本飙升，失去概览 | 任务/章节级 |
| 阶段B只审查不改 | 审查了不调整等于没审 | 审查+调整+附理由 |
| 逐段填充不保一致性 | 术语/风格漂移 | 每段重读前段末尾 |
| 长文档一次生成 | 上下文爆炸 | 渐进式分段 |

### v1.0→v1.10 完整进化（含渐进式）

```
v1.0-v1.5  验收层      11门禁
v1.6       本质层      判断力锚定
v1.7       生产工艺层  5步写作工作流
v1.8       Research强化 完备性自检（有源码：ls）
v1.9       Research强化 一手源核实（无源码：web）
v1.10      Generate强化 渐进式三阶段（大纲→头脑风暴审查→逐段填充）  ← 本次
```

**v1.10 是 Generate 工艺的强化**——把"一次生成"改成"先大纲对抗再填充"，把 cc-2pp 对抗思路前置到廉价的纲层。与 v1.8/v1.9（Research 强化）合起来，Research + Generate 两个阶段都有强化，质量从输入端和生成端双保障。

## v1.10 实测修正记录（dogfooding：技能跑自己的流程）

> 用 v1.10 三阶段重新生成 hermes-agent 文档，实测发现 3 个过度工程点，据实修正。这是 cc-2pp 批判 C2（执行率存疑）的回应——真跑一遍，发现哪些是负担。

### 实测发现 1：阶段C 逐段填充对中短文档是负担

**规则原写**：阶段C"逐段+每段重读前段末尾保连贯"
**实测**：178 行文档一次性组织比逐段更连贯（上下文都装得下，逐段反而打断）。只有超长文档（500+行，爆炸风险）才需真分段。
**修正**：阶段C 按文档长度分档（短一次性/中按主任务/长逐段重读）。见 SKILL.md 阶段C 表格。

### 实测发现 2：阶段B 5维度不必全跑

**规则原写**：阶段B 跑全 5 维度
**实测**：hermes-agent 大纲是 6 个独立任务，MECE 维度基本空走（独立任务无互斥可查）。真正抓问题的是**完整性**（抓出漏"更新"）和**逻辑顺序**（调整任务一/三）。
**修正**：阶段B 按需查——简单大纲只查完整性+逻辑顺序；复杂大纲才跑全 5 维度。

### 实测发现 3：阶段A-B 过程展示应可选

**规则原写**：阶段A-B 显式展示盘点/审查过程
**实测**：本次教学性实测，显式展示有价值（可见可审查）。但日常快速生成时，全展示会啰嗦。
**修正**：过程展示标"可选"——深度审查/教学展示；快速生成内化执行。

### 正向验证：粒度完备优先是对的

v1.10 实测版（178行，6主任务+4子任务分层）vs v1.0（308行，扁平6任务）——信息没丢反而更精炼。证明"分层"比"扁平铺开"高效，**v1.10 的粒度修正（不设上限+分层）经实测验证有效**。

### 元教训

这次实测印证了 cc-2pp 批判的核心：**机制不实测就是空话**。阶段C"逐段重读"写了 10 代没人质疑，一实测发现是负担。**规则要 dogfooding 验证，不能只靠"听起来对"。**
