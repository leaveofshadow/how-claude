## 基础假设与文件存储

> 本章节是 2pp 的根因层。SKILL.md 的「基础假设」是速查版，这里展开落地细节。

### 一、范式转移：实施者从人变成 Claude+skills

2pp 所有设计建立在这个前提上——过去人实施，现在 Claude+skills 实施，实施内容统一为"内容生成"（代码/skills/文章都是 token 输出）。

四推论（详见 SKILL.md 假设1）：
- 能力边界 = 模型 + 可装载 skills（缺能力先找/造 skill，联动 cc-scanner）
- 实施成本 = token/上下文/验证（不是人天；token 仅作实施成本度量，决策改用三项好估维度——功能需求/技术选项/运维成本，token ROI 不好估见假设5 / `org-claude.md`[`#token-roi`]）
- 交接界面读者是 Claude（锚文件按"Claude 好读、好自验证"设计，联动 cc-config）
- skills 可装载（方案产出含"需要哪些 skills"）

### 二、Claude 实施者的系统性缺陷 → 2pp 对策

| 缺陷 | 对策 |
|------|------|
| 偷懒/最快交付 | effort=max + 评分含完整度 + 攻击者专攻捷径 |
| 经常忘记 | 落盘文件 + 锚文件，下个 Phase 从文件读 |
| 认知漂移 | 终态条件 + 循环合同 + 重读锚文件 |
| 幻觉 | Phase 0b web 搜索 + 攻击者验证事实 + 要求证据 |
| 群体思维 | 对抗模型差异化（不同上下文/视角，统一 opus） |
| 临时代码思维 | Plan 重于实现 + 架构约束固化进 CLAUDE.md |

**核心：2pp 的每个机制都映射到一类缺陷的防御。不是"做得更花哨"，是工程必要性。**

### 三、技术选型双校正

```
技术选型 = web 搜索 × 多视角 × 三项好估维度，三位一体
  · web 搜索 = 外部校正（防幻觉/防过时）
  · 多视角   = 内部校正（防单一视角盲区/路径依赖）
  · 三项好估维度 = 功能需求/技术选项/运维成本校正（防"token ROI 真空里决策"；token ROI 不好估，降级见假设5 / `org-claude.md`[`#token-roi`]）
少任何一个都漏。

web 搜索预算（护栏）: 最多 15 轮
  · 一轮 = 一次 WebSearch，可配 WebFetch 深读
  · 每个候选核心技术 3-4 轮（官方文档/横向对比/已知陷阱/版本兼容）
  · 2-3 轮领域常识与竞品（质问用户假设）
  · 留 1-2 轮机动补缺
  · 达到 15 轮强制停止 → 用已有信息 + 向用户问未决项
（呼应循环合同三件套的"预算上限"——搜索也要有天花板，防失控消耗）

信息源（按价值挑，不只官方文档）:
  · GitHub    = 真实落地证据：仓库活跃度(commit/star/issue)、生产实践、已知 bug、替代项目
  · 社区      = 真实口碑+多方观点：HN/Reddit/SO 的踩坑、对比、迁移经验
  · 大牛文章  = 深度判断：核心维护者/资深工程师的分析、设计权衡
  · 官方文档  = 权威基准：API/版本/特性（但防"官方吹嘘"）
  · 综述论文  = 研究型任务的系统性全景：survey/review（国内知网为主，国际 arXiv/Scholar）
  优先级: ★ 工程落地优先（GitHub+社区+大牛是主线）★
          论文是条件性的——仅偏研究型任务才找:
            算法选型 / 学术方向 / 方法论选型 / 选题 / 前沿未定型领域
          工程任务（建系统、选框架）不需要论文，搜了是浪费轮次。
  原则: GitHub+社区验证官方说法；大牛文章过滤 SEO 垃圾。
        偷懒的 Claude 只用记忆里的官方描述（可能过时/吹嘘）——
        严谨的 Claude 去 GitHub 看 issue、社区看吐槽、大牛看深度判断。
```

### 四、一等公民：agent 写文件，编排者读文件做全文综合

**根因（重要）：** 实测发现 agent 确实做了实质工作，但"最终消息只返回摘要"（如"已在前文输出"）。靠返回值取结果 = 系统性拿到空壳。

```
工作流:
  agent 独立思考 → 完整产出写到 .hcc/decisions/{run}/ 文件（不靠返回值）
  编排者         → 读多个 agent 的文件 → 做全文综合判断（不是拼接）
```

**⚠️ 权限前提（写文件 agent 必须有 Write）：** 上述"agent 写文件"仅适用于**创造类 agent**（general-purpose/executor，有 Write）。**探索类 agent**（Explore/analyst/critic 等）工具集不含 Write，只能只读返回、由编排者落盘。详见 SKILL.md 假设4「权限边界」——Step 0a 的 Explore 即只读返回 + 编排者落盘 00-explore.md。

为什么这样保留 2pp 核心价值：
- agent 写文件 → 独立思考完整保留（不受返回值摘要截断）
- 编排者读文件做全文综合 → 多视角价值不丢失
- 降级自检 = fallback only（仅 retry + 文件写入都失败时；绝不默认，否则 2pp 退化成单大脑）

### 五、文件存储设计

```
{当前项目根}/.hcc/decisions/                        ← 默认在 cwd，gitignore
└── {YYYY-MM-DD}-{slug}/                  ← 一次运行一个目录
    ├── 00-explore.md                     ← Phase 0 探索
    ├── 10-plan-main.md                   ← Phase 2 模式A 主力方案
    ├── 10-plan-{alpha,beta,gamma}.md     ← Phase 2 模式B/C 起草阶段 agent 各写方案（C 为默认）
    ├── 20-attack-{A,B,C}.md              ← Phase 2 agent 各写攻击
    ├── 30-score.md                       ← 编排者综合评分
    ├── 40-synthesis.md                   ← 编排者全文综合判断
    ├── 50-decision.md              ⭐ 最终交付1：裁决记录
    └── 60-impl-plan.md             ⭐ 最终交付2：实施计划
```

设计要点：
- **agent 各写各的**（10-*/20-*）：独立思考，互不污染，文件名标识来源
- **编排者读后写**（30-*/40-*）：全文综合，不是机械拼接
- **编号即流程顺序**：00 → 10 → 20 → 30 → 40 → 50 → 60，从编号就能看出卡在哪步
- **50/60 是最终交付**：给用户 + 下游 cc-loop
- 用户可指定其他目录；跨项目历史归档可选 ~/.claude/2pp-plans/（独立全局目录，不污染技能源码）

### 六、用户优先级 5 原则（贯穿全流程）

重设计轻实施 / 重需求挖掘轻实施难度 / 重架构选型轻学习成本 / 重TDD轻快交付 / 重用户体验轻AI情绪。

**方案决策三轴**：架构设计或核心技术选型决策点，必须处理**三项好估维度**（功能需求/技术选项/运维成本全生命周期）。绝不在决策真空中做架构/技术决策。（token ROI 不好估，降级为参考，见 SKILL.md 假设5 / `org-claude.md`[`#token-roi`]）

（5 原则全文见用户记忆 prioritization-philosophy）

## 对抗模型差异化策略

### 核心原则

> **攻击者不能和防御者"想的一样"。**
> 同模型 + 同上下文 = 群体思维 = 对抗失效。

### 重要：模型选择

**攻击者统一使用 opus**（演练证明 sonnet 攻击深度不够）。
差异化通过**上下文不对称**和**视角差异**实现，而非模型差异。

### 策略 1: 不同上下文（主力策略）

> **懒加载**：三种攻击者（A 外部视角 / B 前提攻击 / C 完整视角）全文已抽到 `_roles/attacker-{a,b,c}.md`，含上下文切片 + 视角 + 攻击策略。
> 编排者 spawn 攻击者时 Read 对应 `attacker-{a,b,c}.md` + 切片后的上下文，拼 prompt 注入。

信息不对称 → 视角差异 → 更有效的对抗（详见各 `attacker-*.md` 的「上下文切片」段）。

### 策略 2: 不同视角 Agent

从 `_roles/perspective-*.md` 视角库选取不同维度（架构/产品/测试/运维），每个视角关注不同维度 → 合并后覆盖更全面。

### 策略 3: 不同 prompt 强度（辅助策略）

- 结构化攻击者：必须≥3 缺陷 + 每缺陷有技术证据 + 修复建议 + 输出严格按 schema
- 自由攻击者：不限格式不限角度（直觉有时比结构化分析更准）
- 组合：2 结构化 + 1 自由

### 组合使用建议

| 场景 | 推荐组合 |
|------|---------|
| 快速验证（5 min） | 1 opus 攻击者 + 不同上下文 |
| 标准验证（15 min） | 2 攻击者: opus(外部视角 A) + opus(前提攻击 B) |
| 深度验证（30 min） | 3 攻击者: opus(外部 A) + opus(前提 B) + opus(完整+视角库 C) |
| ★默认（2pp mode） | 模式 C: 判官小组(3方案+评分) + 深度验证(3攻击者) + Phase 4 实施计划 |
| 显式逃生口 | 模式 B 纯评分(score-only): 判官小组(3方案+评分)，无对抗 |

> 2026-07-01 修复：模式 C 从"最高级别可选"提为"2pp 默认"（兑现 Adversarial Verify 承诺）；模式 B 降为显式逃生口。
（组合建议表亦见 `_roles/attacker.md` 索引，单一来源原则）

## Workflow 增强实现（模式C 双轨，opt-in）

模式C 默认手动 spawn Agent；用户说 `ultracode`/`fan out subagents`/`用 workflow`/`动态工作流` 时升级为 Workflow 脚本（parallel fan-out）。**脚本只 fan-out，评分/综合留编排者主循环**（解决攻击依赖评分选 Top2——脚本自治跑完，中途不能插编排者评分，故拆两次调用）。

**草稿1 — 起草 fan-out**（编排者调 `Workflow({script, args:{run, constraint}})`，run=决策目录名，constraint=用户约束+探索摘要）：

```js
export const meta = {
  name: 'cc-2pp-mode-c-draft',
  description: 'cc-2pp 模式C 起草 fan-out（3 派系并行写方案）',
  phases: [{ title: '起草' }],
}
const stances = [
  { key: 'alpha', stance: '保守派' },
  { key: 'beta',  stance: '平衡派' },
  { key: 'gamma', stance: '创新派' },
]
await parallel(stances.map(s => () => agent(
  `你是判官小组${s.stance}。基于探索与约束起草方案，完整写到 .hcc/decisions/${args.run}/10-plan-${s.key}.md（含轻量章节清单+实施者度量约束）。约束: ${args.constraint}`,
  { label: `drafter:${s.key}`, phase: '起草' }   // 不带 schema，写文件；按需 agentType:'general-purpose' 保 Write
)))
return { run: args.run }
// ↓ 编排者主循环：读 10-plan-{α,β,γ} → 评分 → 写独立 30-score.md → 选 Top2
```

**草稿2 — 攻击 fan-out**（编排者调 `Workflow({script, args:{run, top2}})`，top2=评分选出的 Top2 方案全文）：

```js
export const meta = {
  name: 'cc-2pp-mode-c-attack',
  description: 'cc-2pp 模式C 攻击 fan-out（3 攻击者并行，基于 Top2）',
  phases: [{ title: '攻击' }],
}
const attackers = [
  { key: 'A', slice: '只看 Top2 方案，不看探索（外部视角）' },
  { key: 'B', slice: '只看探索，不看方案（前提攻击）' },
  { key: 'C', slice: '看全部（完整上下文；必含三项好估+可编排性向量）' },
]
await parallel(attackers.map(a => () => agent(
  `你是攻击者${a.key}，任务是推翻。上下文切片: ${a.slice}。对 Top2 找≥3 缺陷(严重度/证据/影响/修复)，写到 .hcc/decisions/${args.run}/20-attack-${a.key}.md。Top2: ${args.top2}`,
  { label: `attacker:${a.key}`, phase: '攻击' }
)))
return { run: args.run }
// ↓ 编排者主循环：读 20-attack-{A,B,C} → 综合 → 写 40-synthesis.md（引用 30-score）
```

**守则**：脚本只 fan-out（parallel）；agent 不带 schema 写文件；评分/综合留主循环编排者；drafter/attacker 用有 Write 权限的 agent 类型；30-score 独立、40-synthesis 引用 30-score。

> 编排者用法：`Workflow({ script: <上述草稿>, args: {...} })` → 等完成通知 → 读文件做评分/综合。脚本亦可 `scriptPath` 持久化复用、`resumeFromRunId` 续跑。

## 结构化输出（轻量框架）

**不用刚性 JSON schema**——会把 agent 束缚成填表员，扼杀独立思考（违背一等公民原则）。
改用**轻量章节清单**：agent 写 markdown 文件，必须包含指定章节，章节内自由发挥。

### 为什么从 JSON schema 改为轻量框架

```
旧（刚性 JSON）:
  问题 1: agent 变成填表员，独立思考被格式约束扼杀
  问题 2: 假设靠返回值——但 agent 返回值只给摘要（空壳问题）
新（轻量章节清单）:
  · agent 写文件，章节清单保证可读 + 必需字段不丢
  · 章节内自由发挥，保留独立思考
  · 编排者读文件时检查必需章节（替代 schema 校验）
```

### 三个核心文件的必需章节

章节清单定义在 SKILL.md「结构化输出」中，agent 写文件时遵守：

1. **攻击结果**（20-attack-{X}.md）— 攻击点列表（每点: 标题/严重度/证据/影响/修复，≥3 点）+ 总结（存活判断）；攻击者 C 必含可编排性向量（方案能否拆成≤2 并发的 worktree 单元？验证闸可证伪？）
2. **评分结果**（30-score.md）— 各方案多维评分（可行性/可维护性/可扩展性/实现成本/用户价值 + ★可执行性/可编排性 + ★可并行性标注哪些模块可进 worktree）+ 优缺点 + 嫁接建议 + 推荐
3. **实施计划**（60-impl-plan.md）— 技术选型(含三项好估维度) + 模块拆分 + 里程碑(含验收) + 所需skills + 执行编排(含worktree并发分配) + 执行协议(验证闸+提交回滚) + 风险 + 下一步
4. **细化需求清单**（70-requirements.md）— 按里程碑拆需求项(做什么/输入/输出/可证伪验证/依赖/预估)，保姆级

### agent prompt 末尾追加（懒加载）

> 末尾追加块（写文件+章节清单+认知锚定+度量）已合并进 `_roles/injection-template.md` 的「末尾追加块」段（单一来源，消除三处重复）。
> 编排者 spawn 创造类 agent 时 Read `injection-template.md`，拼末尾追加块 + 必注入约束。

**编排者校验**：读文件时检查必需章节是否齐全；缺失或空壳 → 触发 retry。

## 失败重试策略

### Retry 决策树

```
Agent 调用返回
    │
    ├─ 正常响应（>200字，包含结构化字段）
    │  → 使用结果，不 retry
    │
    ├─ 空响应 / 短响应（<200字）
    │  → Retry 1: 相同参数
    │  → Retry 2: 默认 agent 类型
    │  → 仍失败 → 标记 FAILED
    │
    ├─ API 错误（429 限流）
    │  → Retry 1: 立即重试
    │  → Retry 2: 默认 agent + 降低 prompt 长度
    │  → 仍失败 → 标记 FAILED
    │
    └─ 响应不匹配 schema
       → 尝试从响应中提取有效信息
       → 提取失败 → Retry 1: 强调 schema 要求
       → 仍失败 → Retry 2: 默认 agent
       → 仍失败 → 标记 FAILED
```

### Retry 后的降级策略

```
当 Agent 失败时，按规模分级降级:

单点失败（1 个 Agent FAILED）:
  → 不阻塞其他 Agent（并行调用中的其他 Agent 继续执行）
  → 在最终输出中标注 "⚠️ {视角} 失败"

批量失败（同角色 ≥2 个，或跨 agent 类型都失败）:
  → 判定为系统性失败（换 agent 类型也救不回）
  → ★ 编排者降级自检 = fallback only，绝不默认 ★
  → 仅当 retry 2 次后 agent 仍不写文件，编排者才亲自承担该角色:
       planner 批量失败 → 编排者亲自起草方案
       attacker 批量失败 → 编排者亲自做对抗分析
  → 代价明确: 损失多视角独立思考（核心价值下降），必须向用户标注

为什么降级是 fallback only（重要修正）:
  旧误判: "编排者降级自检从一开始就该这么分工"
          → 这会把 2pp 退化成单大脑，丧失多样性（核心价值）
  纠正:   agent 写文件已解决"返回空壳"问题——独立思考完整保留。
          只有 agent 真的写不出文件时（retry 耗尽），才用编排者自检兜底。

教训来源: 2026-06-15 演练，OMC planner/critic + default agent
          系统性空响应（8/8 全失败）。
          诊断修正: 非"长 prompt 衰减"，而是 agent 最终消息只返回摘要
                   （"已在前文输出"），实际工作做了但没传回。
          正解: 让 agent 写文件（完整产出），编排者读文件。
```

### 防止无限循环

```
全局限制:
  → 每个 Agent 调用: 最多 2 次 retry
  → 每个 Phase: 最多 3 轮修复循环（Step 2b→2c→修复→2b）
  → 全局: 如果总 retry 次数超过 10 次，暂停并告知用户
```

## Phase 4: 实施计划生成

### 为什么需要 Phase 4

```
2pp v1 的问题: 裁决完就结束了，用户得到"选方案 α"但不知道下一步做什么。
2pp v2 修复: 裁决后自动生成可执行的实施计划。

裁决 ≠ 计划:
  裁决: "选修改版 α，RA2 风格原创 RTS MVP"
  计划: "M0(1-2会话): 搭建项目脚手架 + 最小渲染循环，验收条件: ..."
```

### Phase 4 执行步骤

```

    → 嫁接的其他方案优点
    → 用户约束（技术栈/预算=token·会话/验证手段）
  输出（双文件，编排契约 + 需求清单分离）:
    → 60-impl-plan.md（编排契约: 怎么干——智能体/技能/模式/worktree/验证闸/提交回滚）
    → 70-requirements.md（需求清单: 做什么+怎么验证——保姆级，新实例只看一条就能动手）
  ★编排者校验: 70 每里程碑至少有需求项 + 每项有可证伪验证；缺则 retry

Step 4b: 展示实施计划
  → 模块依赖图（ASCII art）
  → 里程碑时间线（按会话度量，非周数）
  → 风险热力图（高/中/低）
```

## 动态视角创建指南

### 何时创建

- 用户说"加一个 XX 视角" / "从 XX 角度审查"
- 你判断当前视角库无法覆盖某个重要维度
- 用户提到了一个专业领域（如安全、合规、国际化）

### 创建流程

```
Step 1: AskUserQuestion（必做）
  问题 1: "这个视角的关注点是什么？"
    → 理解用户想要的审查维度
  问题 2: "使用什么模型？"
    → 选项: opus（深度分析）/ sonnet（标准）/ haiku（快速）
    → 默认: sonnet
  问题 3: "给这个视角起个名字？"
    → 用于文件命名和后续引用

Step 2: 生成视角定义
  → 按"视角库"中的格式生成
  → 包含: 关注维度 + 攻击策略 + 评分标准

Step 3: 写入文件
  → 路径: agents-custom/{name}.md
  → 用户可手动编辑

Step 4: 通知
  → "已创建 [name] 视角，保存在 agents-custom/{name}.md"
  → "你可以在该文件中编辑关注维度和评分标准"
  → 本次会话立即可用
```

### 动态视角文件格式

```markdown
# {视角名称}

> 模型: {opus/sonnet/haiku}
> 创建时间: {date}
> 创建方式: 动态创建

## 关注维度
- [维度 1]: [说明]
- [维度 2]: [说明]

## 攻击策略
从 [角度] 审查方案，关注:
- [具体关注点 1]
- [具体关注点 2]

## 评分标准
- [标准 1]: 权重 X%
- [标准 2]: 权重 Y%
```

- 查看: 扫描 agents-custom/ 目录
- 编辑: 用户直接编辑 .md 文件
- 删除: 用户手动删除 .md 文件
- 更新: 用户编辑后自动生效（下次 skill 加载时读取）

## 反模式表

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| 跳过 Explore 直接 Plan | 没有完备理解（缺 0a 内部/0b 外部）就出方案 | Phase 0 必做，不可跳过 |
| 攻击者和防御者用同模型+同上下文 | 群体思维，对抗失效 | 至少一个维度差异化（上下文或视角） |
| 一次用全部 6 个视角 | 信息过载，维度过多 | 根据场景选 2-3 个 |
| 不问用户直接选模式 | 可能选错模式 | resolveMode 按触发词/配置定 pattern（默认 C）；仅无配置且中性触发才 AskUserQuestion |
| 攻击结果不分类 | 混淆有效/无效攻击 | 每个攻击明确标记有效/无效 |
| 不持久化 Plan | 下次会话无法回顾 | 必须保存到文件 |
| Plan 只写最终方案 | 丢失决策过程 | 包含被否决方案和否决原因 |
| 无限轮对抗验证 | 浪费 token | 最多 2 轮修复 + 3 轮全局上限 |
| 裁决后无实施计划 | 用户不知道下一步做什么 | Phase 4 必做 |
| 用 OMC critic agent 做复杂评审 | 返回空响应 | 默认 agent + 内联评分标准 |
| 攻击者用 sonnet | 攻击深度不够 | 统一用 opus，差异化靠上下文 |
| Agent 失败后静默跳过 | 结果不完整 | retry → 标记 FAILED → 告知用户 |
| 把编排者降级自检当默认路径 | 丧失多样性，2pp 退化成单大脑 | 降级 = fallback only，agent 写文件保独立思考 |
| agent 靠返回值传结果 | 拿到空壳摘要 | agent 写文件，编排者读文件 |
| 架构/技术选型不处理三项好估维度 | 技术优雅/换皮偷工但不值得做 | 决策点必处理三项好估维度（功能需求/技术选项/运维成本；token ROI 降级见假设5 / `org-claude.md`[`#token-roi`]） |
| 用刚性 JSON schema 约束 agent | agent 变填表员，独立思考被扼杀 | 轻量章节清单，章节内自由发挥 |
| 设计阶段对模糊需求直接动手 | 缺陷传递到实施 | 挖需求铁律：越简单越挖 |
| 技术选型 web 搜索无上限 | 失控消耗 token，迟迟不收敛 | 最多 15 轮，达到强制停止转 Phase 0c |
| 只搜官方文档/泛搜 SEO 文章 | 被"官方吹嘘"和 SEO 垃圾误导 | GitHub+社区+大牛文章分层验证 |
| 工程任务也搜论文 | 浪费搜索轮次，论文不解决落地问题 | 工程落地优先，论文仅研究型任务才找 |

## 与 Loop Engineering 的关系

2pp 产出的是**设计决策 + 实施计划**，可以直接作为循环的锚：

```
2pp Phase 3 产出决策 → 写入 decision.md
    ↓
2pp Phase 4 产出计划 → 写入 plan.md
    ↓
cc-goal 设计终态条件 → 基于 plan.md 的验收条件
    ↓
cc-loop 执行循环     → 以终态条件为 STOP
```

在 Loop Engineering 五阶段中，2pp 处于 Stage 2-3 之间：
- Stage 2: 用 2pp 的 Explore 理解问题
- Stage 3: 用 2pp 的 Plan + Verify 制定可靠方案
- Stage 3+: 用 2pp 的 Phase 4 生成实施计划
- 然后进入 /goal（终态条件）→ /loop（执行）
