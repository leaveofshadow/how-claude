# Memory 审查指南 — 三层记忆诊断与优化深度参考

## 三层记忆体系

```
┌──────────────────────────────────────────────────┐
│ L1: 优先记忆 (Priority Context)                   │
│   工具: notepad_write_priority / notepad_read     │
│   大小: < 500 字                                   │
│   生命周期: 永久（覆盖写）                          │
│   加载: 每次会话启动自动加载                        │
│   适合: 最关键的锚点信息、当前核心任务               │
├──────────────────────────────────────────────────┤
│ L2: 工作记忆 (Working Memory)                      │
│   工具: notepad_write_working / notepad_read       │
│   大小: 无硬限制                                    │
│   生命周期: 7 天自动清理                            │
│   加载: 读取时                                      │
│   适合: 近期决策、进展、临时发现、调试记录           │
├──────────────────────────────────────────────────┤
│ L3: 持久记忆 (Persistent Memory)                   │
│   工具: Write (文件) + MEMORY.md (索引)             │
│   大小: 每个文件一个事实                            │
│   生命周期: 永久（手动管理）                        │
│   加载: 读取时                                      │
│   适合: 用户偏好、项目经验、反馈教训、外部参考       │
└──────────────────────────────────────────────────┘

还有文件级 memory 系统（与 L3 互补）:
  位置: ~/.claude/projects/{project}/memory/*.md
  格式: frontmatter (name/description/metadata.type) + 正文
  索引: MEMORY.md（每条一行，格式: - [标题](文件.md) — 钩子）
  类型: user / feedback / project / reference
```

## 审查流程详解

### Phase 1: 数据采集

```
采集工具调用（按顺序执行）:

1. notepad_read(section: "all")
   → 获取三层 notepad 全部内容

2. notepad_stats()
   → 获取大小、条目数、最旧条目

3. Glob: ~/.claude/projects/*/memory/*.md
   → 获取所有 memory 文件列表

4. Read: MEMORY.md（如存在）
   → 获取 memory 索引

5. 对每个 memory 文件: Read 前 10 行
   → 提取 frontmatter (name/description/type)
   → 不读全文（避免上下文爆炸）

采集失败处理:
  - notepad 工具不可用 → 跳过 L1/L2，只审查 L3
  - memory 目录不存在 → 报告"未使用持久记忆"
  - MEMORY.md 不存在 → 标记"需要创建索引"
```

### Phase 2: 逐层诊断

#### L1 优先记忆诊断

```
检查项:
├── 长度检查
│   ├── < 300 字 → ✅ 健康
│   ├── 300-450 字 → ⚠️ 接近上限，考虑压缩
│   └── > 450 字 → 🔴 超限，必须压缩
│
├── 内容质量
│   ├── 是否有过时信息？（上一个任务已完成的描述）
│   ├── 是否有冗余？（与 CLAUDE.md 重复的规则）
│   ├── 是否缺少关键锚点？（用户核心偏好未记录）
│   └── 是否有不精确的描述？（模糊的"注意"/"小心"）
│
└── 建议操作
    ├── 删除已完成的任务描述
    ├── 移重复规则到 CLAUDE.md
    ├── 补充关键偏好（从 L3 发现的频繁 feedback）
    └── 重写为更精炼的表述
```

#### L2 工作记忆诊断

```
检查项:
├── 时效检查
│   ├── 最近 3 天有更新 → ✅ 活跃
│   ├── 3-7 天无更新 → ⚠️ 陈旧，可能需要清理
│   └── > 7 天无更新 → 🔴 过期（等待自动清理或手动清理）
│
├── 重复检查
│   ├── 比对条目关键词，发现语义重复
│   └── 相同决策出现多次 → 应合并为一条
│
├── 遗漏检查
│   ├── 是否有重要决策只存在于对话中？
│   ├── 是否有用户反馈未记录？
│   └── 是否有当前任务进度未追踪？
│
└── 分类检查
    ├── 临时信息（调试值、文件路径）→ 适合 L2
    ├── 长期经验（用户偏好、架构决策）→ 应移到 L3
    └── 规则性内容 → 应移到 CLAUDE.md 或 Skill
```

#### L3 持久记忆诊断

```
检查项:
├── 索引检查
│   ├── MEMORY.md 是否存在？
│   ├── 每个文件是否在索引中有对应条目？
│   └── 索引描述是否与文件内容一致？
│
├── 完整性检查
│   ├── 是否有孤立文件（存在但无索引引用）？
│   ├── 是否有幽灵索引（索引引用但文件不存在）？
│   └── 文件名是否遵循 kebab-case 命名？
│
├── 质量检查
│   ├── frontmatter 是否完整（name/description/type）？
│   ├── 是否有互相矛盾的记忆？
│   ├── 是否有可从代码推导的冗余记忆？
│   ├── type 分类是否正确？
│   └── 是否有应升级为 CLAUDE.md 规则的反复 feedback？
│
├── 关联检查
│   ├── 是否使用了 [[name]] 交叉引用？
│   ├── 相关记忆是否已互相链接？
│   └── 断裂的链接（引用不存在的 name）？
│
└── 时效检查
    ├── 最近 30 天有更新 → ✅ 活跃
    ├── 30-90 天无更新 → ⚠️ 可能过时
    └── > 90 天无更新 → 🔴 需要验证是否仍然适用
```

### Phase 3: 跨层一致性检查

```
检查三层之间的信息流动:

1. L1 ↔ CLAUDE.md
   - 是否有重复？→ L1 应该是 CLAUDE.md 的动态补充，不重复静态规则
   - L1 是否引用了 CLAUDE.md 中的概念？→ 应该明确引用

2. L2 → L3 提升
   - L2 中是否有反复出现的模式？→ 提升为 L3
   - L2 中是否有超过 5 天仍有价值的信息？→ 提升为 L3

3. L3 → CLAUDE.md 升级
   - L3 中是否有反复出现的 feedback 类型记忆？→ 升级为 CLAUDE.md 规则
   - L3 中是否有 project 类型记忆描述的是不变架构？→ 移到 CLAUDE.md

4. L3 去重
   - 多个 memory 文件描述同一事实？→ 合并为一个
   - user 类型记忆分散描述用户特征？→ 合并到一个用户画像文件
```

### Phase 4: 调用 /memory 深度分析

当满足以下条件时，建议用户调用 `/memory` 做更深入的分析：

```
触发条件（满足任一即可建议）:
├── 持久记忆文件 > 20 个
├── 工作记忆条目 > 15 条
├── 用户明确说"深度审查"/"全面分析"
├── 发现 3+ 处互相矛盾的记忆
└── 发现记忆系统有结构性问题（如无索引、大量孤立文件）

提示方式:
  在审查报告末尾追加:
  "建议: 输入 /memory 让 Claude 对记忆系统做全面诊断和重组。"
```

## 常见问题诊断表

| 症状 | 可能原因 | 诊断步骤 | 修复方案 |
|------|---------|---------|---------|
| Claude 反复犯同样的错 | feedback 未持久化 | 检查 L3 是否有对应 feedback | 写入 feedback 类型记忆 |
| Claude 忘记用户偏好 | user 记忆缺失 | 检查 L3 user 类型记忆 | 补充用户画像记忆 |
| 新会话不知道之前的进度 | 进展只存 L2 | 检查关键进度是否在 L3 | 提升到 L3 或 priority |
| 上下文消耗太快 | L1 过大 | 检查 L1 长度 | 压缩 L1，移内容到 L3 |
| 记忆互相矛盾 | 多次写入未去重 | 交叉比对同主题记忆 | 合并或标记过时 |
| MEMORY.md 缺失 | 从未创建索引 | 检查文件是否存在 | 生成索引文件 |
| 记忆文件命名混乱 | 无命名规范 | 检查文件名格式 | 重命名为 kebab-case |

## 优化操作模板

### 压缩 L1 优先记忆

```
将 L1 压缩到 < 300 字的原则:
1. 只保留: 当前核心任务 + 最关键的 2-3 条用户偏好 + 活跃约束
2. 移除: 已完成的任务、与 CLAUDE.md 重复的规则、冗长的上下文
3. 用关键词替代句子: "用户偏好函数式编程，避免 OOP" → "偏好: FP > OOP"
4. 移出的内容: 写入 L3 对应类型的 memory 文件
```

### 提升 L2 → L3

```
判断标准:
- 信息在未来 7 天后仍有价值？→ 提升
- 同类信息出现 2+ 次？→ 合并提升
- 用户明确表示"记住这个"？→ 提升

操作:
1. 从 L2 提取有价值的信息
2. 创建/更新 L3 memory 文件（正确的 type + frontmatter）
3. 更新 MEMORY.md 索引
4. L2 中对应条目等待自动过期（不手动删除，避免丢失时间戳）
```

### 升级 L3 → CLAUDE.md

```
判断标准:
- feedback 类型记忆出现 3+ 次同主题？→ 升级为规则
- project 类型记忆描述的是永久架构约定？→ 升级
- 用户说"每次都要这样"？→ 升级

操作:
1. 将记忆内容提炼为一条 CLAUDE.md 规则（具体、可执行）
2. 添加到 CLAUDE.md 的适当分组
3. 在 L3 记忆中标注"已升级为 CLAUDE.md 规则，可删除"
4. 更新 MEMORY.md 索引
```

### 重组 L3 索引

```
当 L3 文件 > 10 个时，建议按以下结构组织 MEMORY.md:

# MEMORY.md

## 用户画像
- [用户偏好](user-preferences.md) — 编程风格、沟通偏好
- [用户角色](user-role.md) — 专业背景、技术栈

## 项目经验
- [架构决策](arch-decisions.md) — 关键架构选择及原因
- [经验教训](lessons-learned.md) — 踩过的坑和解决方案

## 反馈教训
- [工作方式](workflow-feedback.md) — 用户对 Claude 行为的纠正
- [沟通偏好](communication-style.md) — 输出格式、语言偏好

## 外部参考
- [API 文档](api-reference.md) — 关键 API 文档链接
```

## 反模式表

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| L1 写 800 字 | 每次启动都消耗大量上下文 | L1 < 500 字，详细内容放 L3 |
| L2 存永久信息 | 7 天后自动丢失 | 永久信息放 L3 |
| L3 不建索引 | 无法发现已有记忆，导致重复 | 维护 MEMORY.md |
| 重复写入相同 feedback | 浪费空间，可能矛盾 | 先搜索再写入 |
| 把代码结构存 L3 | 代码本身已记录，冗余 | 只存"为什么"不存"是什么" |
| memory 不用 frontmatter | 无法自动分类和检索 | 必须使用 name/description/type |
| 把临时调试值存 L3 | 永久占用空间 | 临时值存 L2 |
| 不清理过时记忆 | 越积越多，搜索变慢 | 定期审查，删除过时条目 |

## Loop Engineering 视角：循环中的记忆写回

在 Loop Engineering 中，记忆不仅是"跨会话记住东西"——它是**循环的持久化层**。

### 循环的三个记忆层级

```
循环中的记忆分层:

  锚文件（CLAUDE.md/VISION.md）
    → 每轮迭代都读，不变化
    → 类比：程序的常量

  磁盘状态（git/TODO.md/progress.md）
    → 每轮迭代读+写，记录进度
    → 类比：程序的数据库

  对话上下文（conversation context）
    → 当前迭代的临时记忆
    → 类比：程序的内存
    → ⚠️ 迭代间可能丢失（compact/reset）
```

### 记忆写回时机

```
在循环中，何时写回记忆：

  ┌─ 阶段完成时
  │  → 写入 memory: "阶段 X 完成，产出 Y"
  │
  ├─ 做出重要决策时
  │  → 写入 memory: "选择方案 Z，因为 W"
  │
  ├─ 发现非显而易见的模式时
  │  → 写入 memory: "发现模式 P，适用于 Q"
  │
  ├─ 遇到阻碍时
  │  → 写入 BLOCKED 文件 + memory
  │
  └─ 循环即将结束时
     → 写入总结: 做了什么、改变了什么、需要注意什么
```

### 循环记忆 vs 普通记忆的区别

| 维度 | 普通记忆 | 循环记忆 |
|------|---------|---------|
| 写入频率 | 偶尔 | 每轮都可能 |
| 读取消耗 | 按需 | 每轮必读（锚文件） |
| 过期风险 | 低 | 高（compact 可能丢失） |
| 冗余风险 | 中 | 高（重复写入相同进度） |
| 建议 | 正常管理 | 额外注意去重+精炼 |

### 循环结束后的记忆审计

```
循环完成后，检查记忆状态：
  1. 是否有循环过程中的临时记忆需要清理？
  2. 是否有应升级为 CLAUDE.md 规则的 feedback？
  3. 是否有冗余的进度记录可以合并？
  4. MEMORY.md 索引是否更新？
```
