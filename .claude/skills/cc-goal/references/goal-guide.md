# Goal 终态条件设计 — 深度参考

## 这是什么

本文档是 cc-goal 技能的深度参考。核心主题：如何把模糊的"我想做 X"变成 **Claude 能自主驱动到完成的终态条件**。

灵感来源：supergoal 插件的方法论 — 专注帮用户写好 /goal 条件，通过侦察→定义→自评→预检的流程。

---

## 一、终态条件 vs 任务描述

这是理解 /goal 的最关键概念。

| 维度 | 任务描述 | 终态条件 |
|------|---------|---------|
| 视角 | 过程导向：做什么 | 结果导向：完成后什么样 |
| Claude 的角色 | 执行者（你说一步做一步） | 自主 agent（自己决定怎么到终点） |
| 终止判断 | 人判断"好了" | 条件自动验证 |
| 适合模式 | 直接工作 / TaskCreate | /goal / /loop |
| 风险 | 做了但不满意，来回修改 | 一次对齐，agent 自己循环到对 |

### 对比实例

```
任务描述（L1）：
  "重构认证系统，把 session 迁移到 JWT"

终态条件（L4）：
  "认证已完全迁移到 JWT：
   - 所有 /api/* 端点使用 JWT 验证
   - npm test 全部通过
   - curl -H 'Authorization: Bearer <token>' /api/me 返回 200
   - grep -r 'session' src/auth/ 无结果（无残留）
   - API 响应 p99 < 200ms（用 autocannon 验证）
   - 不改动 /api/public/* 的公开访问"
```

## 二、五层质量模型

| 层级 | 质量 | 特征 | Claude 的表现 |
|------|------|------|-------------|
| **L0** ❌ | 模糊 | 只有动词，无方向 | 猜测意图，结果不可预测 |
| **L1** 🟡 | 具体 | 有方向，无终点 | 知道做什么，不知何时停 |
| **L2** 🟢 | 可验证 | 有完成标准 | 知道停止条件 |
| **L3** ✅ | 有约束 | 有边界限制 | 知道不能改什么 |
| **L4** 🏆 | 自验证 | 有自动验证方法 | **可自行确认完成** |

## 每层实例对比

### 实例 1: 重构
```
L0 ❌  "重构认证系统"
L1 🟡  "将认证从 session 迁移到 JWT"
L2 🟢  "迁移到 JWT，所有现有测试通过"
L3 ✅  "迁移到 JWT，测试通过，API 响应时间不退化，保持向后兼容"
L4 🏆  "迁移到 JWT，运行 npm test 全绿，curl /api/health 返回 200，
        响应时间 p99 < 200ms（用 autocannon 验证）"
```

### 实例 2: 新功能
```
L0 ❌  "加个搜索"
L1 🟡  "实现全文搜索功能，用 Elasticsearch"
L2 🟢  "实现搜索，搜索结果包含标题+摘要+高亮，响应 < 500ms"
L3 ✅  "实现搜索，响应 < 500ms，支持中文分词，不改动现有 API 签名"
L4 🏆  "实现搜索，运行 test:search 通过，手动查询'测试'返回相关结果，
        响应时间用 curl -w '%{time_total}' 验证 < 0.5s"
```

### 实例 3: Bug 修复
```
L0 ❌  "修复登录 bug"
L1 🟡  "修复用户在 Chrome 上无法登录的问题"
L2 🟢  "修复登录问题，在 Chrome/Firefox/Safari 上都能正常登录"
L3 ✅  "修复登录，三大浏览器通过，不改动 session 存储逻辑"
L4 🏆  "修复登录，写回归测试（test/login.spec.ts）验证三浏览器行为，
        运行 npx playwright test 通过"
```

### 实例 4: 性能优化
```
L0 ❌  "优化一下"
L1 🟡  "优化首页加载速度"
L2 🟢  "首页 LCP 从 3s 降到 1.5s 以下"
L3 ✅  "LCP < 1.5s，不增加打包体积超过 10%，不删除任何现有功能"
L4 🏆  "LCP < 1.5s（用 Lighthouse CI 验证），bundle size 增长 < 10%
        （用 bundlesize 工具验证），npm run build 成功"
```

### 实例 5: 文档
```
L0 ❌  "写个文档"
L1 🟡  "给 API 写文档"
L2 🟢  "给所有 public API 端点写 OpenAPI 文档，覆盖率 100%"
L3 ✅  "OpenAPI 文档，使用项目现有的 YAML 格式，包含示例请求/响应"
L4 🏆  "OpenAPI 文档，运行 spectral lint 无错误，
        每个端点有 curl 示例（可直接复制使用）"
```

## 三、终态条件设计方法论

### 方法论来源

supergoal 插件的方法论核心理念：**不帮用户执行任务，帮用户写好 /goal 条件**。通过结构化的侦察→定义→自评→预检流程，输出一个打磨好的终态条件。

### Step 1: 侦察（Recon）

在写终态条件之前，静默收集信息：

```
侦察清单：
□ 读 CLAUDE.md → 了解项目约束和技术栈
□ 读相关代码 → 了解现状和依赖
□ 读 tests/ → 了解已有验证手段
□ 读 specs/ROADMAP → 了解全局目标
□ 读 .claude/ → 了解已安装工具和技能
```

### Step 2: 定义终态

用终态条件模板构建：

```markdown
## 终态条件

### 必须满足（MUST）
- [可验证的条件 1，含验证方法]
- [可验证的条件 2，含验证方法]

### 不能破坏（MUST NOT）
- [约束 1]
- [约束 2]

### 验证方法（HOW TO VERIFY）
- `命令 A` → 期望输出
- `命令 B` → 期望输出

### 范围限定（SCOPE）
- 只改 [这些文件/目录]
- 不改 [这些文件/目录]
```

### Step 3: 自评通过（Self-Critique）

输出终态条件之前，做三轮自评：

**第一轮：可证伪性（Falsifiability）**

每个条件必须可以被证明为 FALSE：

```
❌ "代码质量好"       → 无法证伪，主观判断
❌ "性能合理"         → 无法证伪，标准不明
✅ "测试覆盖率 > 80%"  → 可以证伪：运行 coverage 命令
✅ "无 SQL 注入漏洞"   → 可以证伪：sqlmap 扫描
✅ "API 响应 < 200ms"  → 可以证伪：压测工具测量
```

**第二轮：原子性（Atomicity）**

每个条件应该独立可验证：

```
❌ "功能完整且性能好"  → 两个条件绑在一起
✅ "所有端点返回正确结果"（功能）
✅ "p99 延迟 < 200ms"  （性能）

检查：一个条件失败是否阻止其他条件的验证？
```

**第三轮：最弱依赖（Weakest Dependency）**

找出最可能失败的验证步骤：

```
终态条件依赖链：
  代码修改 → 测试通过 → 性能达标 → 部署成功

最弱环节：性能达标（依赖运行环境、数据量）

修复路径：
  如果性能不达标 → 回退到上一版本 + 记录瓶颈
  不应该 → 无限重试同一方案
```

### Step 4: 预检烟雾测试（Pre-flight）

交付前的最后检查：

```
□ 终态条件描述了"完成时什么样"，而非"怎么做到"？
□ 有至少一个自动化验证手段？
□ 有明确的"不能破坏"约束？
□ 10 次迭代仍无法完成时有退出策略？
□ 无"适当"、"合理"、"好"等模糊词？
□ 每个 MUST 条件都有对应的验证命令？
□ 范围限定清晰（改什么/不改什么）？
```

## 四、Goal 生成器（引导流程）

当用户给出模糊目标时，依次引导补充：

### L0 → L1: 补充方向
```
问：你希望完成后系统/代码处于什么状态？
   用"完成后，我应该能..."的句式描述。

例：用户说"优化性能"
引导 → "完成后，用户应该能在 1 秒内看到页面内容"
```

### L1 → L2: 定义终点
```
问：怎么判断做到了？给出一个可以测量/检查的标准。

例：用户说"让页面更快"
引导 → "首页 LCP（最大内容绘制）< 1.5 秒"
```

### L2 → L3: 设定边界
```
问：有什么是不能改变的？有什么约束？

例：
- "不改变现有 API 签名"
- "不增加打包体积超过 10%"
- "只能改 src/ 目录"
```

### L3 → L4: 自动化验证
```
问：怎么自动验证？用什么命令/工具可以证明？

例：
- "npm test 全绿"
- "curl -w '%{time_total}' < 0.5"
- "lighthouse --only=performance 得分 > 90"
- "npx tsc --noEmit 无错误"
```

## 五、Goal → 工作模式映射

```
分析用户的 Goal 层级：
│
├─ L0-L1（模糊/具体但无终点）
│  └─ → 直接工作
│     Claude 猜测路径，可能偏离
│     适合：探索性任务、学习项目
│
├─ L2（可验证）
│  └─ → TaskCreate 列表 + 逐步执行
│     创建任务清单，逐个完成并勾选
│     适合：有明确完成标准的中等任务
│
├─ L3（有约束）
│  └─ → EnterPlanMode 先规划 → 再执行
│     先制定计划，用户确认后再动手
│     适合：需要小心不破坏现有功能的任务
│
└─ L4（自验证）
   └─ → /goal 或 /loop + 自动验证循环
      自动执行 + 自动验证 + 不通过自动修复
      适合：有清晰自动化验证标准的大型任务
```

## 六、与 Loop / Agent / Workflow 的联动

### Goal 质量决定工具选择

```
L4 Goal + 需要反复验证
  └─ /goal 或 Loop（ScheduleWakeup）+ 每轮自动验证

L3 Goal + 多文件并行修改
  └─ 并行 Agents + 各自完成后汇总验证

L2 Goal + 确定性步骤
  └─ TaskCreate 列表 + 逐步执行

L4 Goal + 大规模扫描
  └─ Workflow pipeline + 自动验证每阶段
```

### 组合模式：Goal + Loop

```
用户的 L4 Goal：
"确保所有 API 端点都有认证检查，运行 npm test 验证"

→ 转化为 Loop：
/loop 逐个检查 src/routes/ 下的端点，
每个端点验证是否有 auth middleware，
缺少的自动添加，
全部完成后运行 npm test 确认通过，然后停止。
```

### 组合模式：Goal + Workflow

```
用户的 L4 Goal：
"重构数据库层，所有测试通过，性能不退化"

→ 转化为 Workflow：
Phase 1: 扫描所有数据库调用点（parallel agents）
Phase 2: 逐个迁移到新层（pipeline）
Phase 3: 运行全量测试 + 性能基准（agent）
Phase 4: 对比结果，不通过则自动修复（loop-until-dry）
```

## 七、反模式表

| ❌ 反模式 | 问题 | ✅ 修正 |
|----------|------|--------|
| "优化一下" | 无方向，Claude 猜测 | "将 LCP 从 3s 降到 1.5s 以下" |
| "和之前一样" | Claude 不知道"之前"是什么 | 明确描述期望行为 |
| "做到完美" | 无终止条件 | 定义可测量的质量指标 |
| "全部重构" | 无边界，可能改几百文件 | 指定范围和优先级 |
| "尽快完成" | 催促导致质量下降 | 设质量标准而非速度要求 |
| "按你想的来" | 放弃控制权 | 给出方向+边界 |
| "别管测试" | 技术债累积 | "包含测试，但可用简化版" |
| "用最新的技术" | 追新而无目的 | 明确要解决什么问题 |
| "代码质量好" | 无法证伪 | "测试覆盖率 > 80%" |
| "性能合理" | 标准不明 | "p99 延迟 < 200ms" |
