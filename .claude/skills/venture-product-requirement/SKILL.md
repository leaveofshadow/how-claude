## 1. 六块内容契约（IEEE 830 轻量 + EARS）

> 落盘路径：`.hcc/product/venture-product-requirement/N3.5_需求规格_prd.md`（by-design `.hcc/product/`，**勿改回 `.venture/artifacts/`**）。
> 文件名规范：`{phase=N3.5}_{feature=需求规格}_{type=prd}.md`。
> 每条需求带 ID（R1/R2...）+ 对应 AC（AC1/AC2...）便 N5 git diff 对照（R{n}↔AC{n} 一一对应强化）。

### §1 背景与目标

- **定义**：问题陈述 + 可证伪成功标准（**业务级定性**，主题8）。
- **指标归属**：成功标准用业务级定性语言，如"用户能完成 X 流程"、"系统能交付 Y 输出"，**不用工程级定量指标**（定量指标归 §4）。
- **示例**：
  ```
  ## §1 背景与目标
  问题：当前 X 流程需人工 5 步，错误率 30%。
  成功标准（业务级）：用户通过系统完成 X 流程，无需手动介入中间步骤。
  ```
- **长度下限**：块正文 ≥ 50 字（闸E）。

### §2 范围与边界

- **定义**：in-scope / out-scope 列表 + **项目类型声明（必 cite 证据，主题3）**。
- **项目类型 ∈** `{PoC, MVP, 通用框架, 生产级}`——类型不由 agent 凭空自选，**必须 cite 证据**防止自证循环（agent 故意选 PoC 绕过 §4/§6）。
- **类型声明模板**（行首锚定，闸A 正则校验块内首行）：
  ```
  ## §2 范围与边界
  项目类型: PoC ← 因 N1 机会调查 §X 显示探索阶段
  # 或：项目类型: 生产级 ← charter 显式要求 p99 < 200ms
  # 或：项目类型: MVP ← N3 决策结论 §Y 标注需最小可用版本
  # 或：项目类型: 通用框架 ← charter 块2 声明复用跨场景
  in-scope:
    - R1 ...
    - R2 ...
  out-scope:
    - 用户画像/TAM/SAM（商业调研产物，本节点不产出）
    - 市场规模分析
  ```
- **长度下限**：块正文 ≥ 50 字（闸E）。

### §3 功能需求

- **定义**：EARS 句式需求条目 R1/R2...（**完整枚举五种 EARS**，主题1 闸C）。
- **五种 EARS 句法**（Easy Approach to Requirements Syntax——结构化、无歧义）：
  - **Ubiquitous**：`The system SHALL [response]`（常态必为）
  - **Event-driven**：`WHEN [trigger] the system SHALL [response]`（事件触发）
  - **State-driven**：`WHERE [state] the system SHALL [response]`（状态条件）
  - **Optional**：`WHERE [condition] the system MAY [response]`（可选行为）
  - **Unwanted**：`IF [trigger] THEN the system SHALL [response to prevent]`（异常防护）
- **示例**：
  ```
  ## §3 功能需求
  R1: WHEN 用户提交搜索请求，系统 SHALL 在 2s 内返回结果（查询长度 < 100 字符）。
  R2: WHERE 系统处于离线状态，系统 SHALL 显示缓存结果并提示"离线模式"。
  R3: IF 查询超过速率限制，THEN 系统 SHALL 返回 429 并在响应头标注 Retry-After。
  R4: 系统 SHALL 支持中英文查询（Ubiquitous）。
  R5: WHERE 用户为管理员，系统 MAY 展示审计日志面板。
  ```
- **长度下限**：每条 R{n} 正文 ≥ 50 字（闸B）。
- **可观测判据**：每条 R{n} 必含可观测判据标记（验证动词/数值/命令——为 §5 AC 预留映射，主题6）。

### §4 非功能需求

- **定义**：性能/约束/质量（**工程级定量**，主题8）。
- **指标归属规则（主题8 核心）**：
  - §1 成功标准 = 业务级（定性）
  - §4 非功能 = 工程级（定量，如 p99 < 200ms、可用性 ≥ 99.9%、内存占用 < 512MB）
  - **§4 定量指标必须 mirror 到 §5 AC**（§5 有对应验收命令）——闭环（修主题6 的上游）。
- **同源思考（主题7 核心，非 DAG 并行）**：§4 与 §3 **同时思考**（生成时一起想），不是拓扑并行执行。生成序见 §3。
- **示例**：
  ```
  ## §4 非功能需求
  NFR1（性能）: p99 响应延迟 < 200ms（负载 100 QPS）→ mirror §5 AC：bench 脚本阈值
  NFR2（可用性）: 月度可用性 ≥ 99.9% → mirror §5 AC：监控告警阈值
  NFR3（资源）: 单实例内存占用 < 512MB → mirror §5 AC：RSS 采样阈值
  ```
- **长度下限**：块正文 ≥ 50 字（闸E）。

### §5 验收标准

- **定义**：每条 R{n} → AC{n}（**一一对应**，主题1 闸B），AC 含**可执行验证命令**（测试命令/检查脚本/数值阈值），**非自然语言对照**（主题6 N5 验证映射上游）。
- **N5 验证范式**（主题6）：N5 验证 = 跑这些命令，**不是 git diff 文本对照**。每条 AC 绑可执行验证命令为 N5 实装建接口（M5/M6 建立 AC↔验证命令映射表）。
- **可观测判据（主题1 闸D 正向化）**：每条 AC 必含可观测判据（正向要求）——验证动词/数值/命令；模糊词黑名单（"友好"/"快速"/"易用"）降为辅助校验。
- **示例**：
  ```
  ## §5 验收标准
  AC1（对应 R1）: 执行 `node bench/search.bench.js --qps=100` → p99 < 2000ms，exit 0
  AC2（对应 R2）: 断网后调用 `curl /api/search` → 返回体含 "离线模式"，HTTP 200
  AC3（对应 R3）: 压测超过 QPS 上限 → 响应码 429，Header 含 Retry-After
  AC4（对应 R4）: `node test/i18n.test.js` → 中英文 case 全绿
  ```
- **长度下限**：每条 AC{n} 正文 ≥ 30 字（闸B）。

### §6 里程碑

- **定义**：阶段交付节点 + 验收闸（引 §5 AC{n}）。
- **示例**：
  ```
  ## §6 里程碑
  M1（原型）: 完成 R1/R2，过 AC1/AC2
  M2（集成）: 完成 R3/R4，过 AC3/AC4
  M3（生产）: NFR1-3 全达标，§4 mirror AC 全绿
  ```
- **长度下限**：块正文 ≥ 50 字（闸E）。

## 6. 三项好估维度（方案/裁剪决策主轴，cc-2pp 假设5）

> Boss 方法论修正：Claude 实施者 token ROI **不好估**（投入侧 = token 是 Claude 运行成本非用户直接感知；产出侧 = 规格/闸/契约价值难提前货币化；两端都模糊）→ **不作方案/裁剪决策主轴**。token 仍作实施成本度量，但不作决策依据。

1. **功能需求**：方案是否完整满足 N3.5 节点功能职责（提供正确工程需求规格，不换皮、不偷工）。

**token ROI 仅作度量**（如本 skill 估算 ~1 会话 ~8-12k token 输出），**不作决策主轴**。
