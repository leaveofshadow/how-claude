# 使用指南 — how-claude 教练套件

> 怎么装、怎么触发、典型工作流。技术细节见各技能 `SKILL.md` + `references/`。

## 1. 安装

how-claude 是 Claude Code 技能套件。**推荐用 hcc-cli 一键装**（技能形式 CLI）：

```
> hcc install                          # 当前项目装全套件（软链）
> hcc install --minimal                # 只装 claude-coach + cc-loop + cc-2pp
> hcc install --skills=cc-2pp,cc-loop  # 自选
> hcc update                           # git pull 更新
> hcc doctor                           # 5 维诊断
> hcc uninstall [--purge]              # 卸载（--purge 删全局 clone）
```

hcc-cli 自动：全局 clone `~/.claude/how-claude/`（一处克隆多项目共享）+ 逐技能软链到项目 `.claude/skills/` + 写 manifest（`<项目>/.claude/how-claude-manifest.json`）。Windows 用 junction（不需管理员），Unix 用 symlink。

> 无 Claude Code / 想手动：`git clone https://github.com/leaveofshadow/how-claude.git` + 复制/软链 `.claude/skills/<技能>` 到你的项目。

技能被 Claude Code 自动发现，触发时按需加载（不全量占 context）。

## 3. 典型工作流

### A. 复杂设计决策（cc-2pp 两阶段）

适用：技术选型 / 架构决策 / 多未知方案。

**三种 pattern（2pp 重 mode 内部）**：

| pattern | 触发词 / 命令 | 流程 | 对抗 |
|---|---|---|---|
| **C 判官小组+对抗（★默认）** | 「2pp」「判官小组」「对抗设计」「对抗验证」/ `/2pp:full` | 起草3方案→评分→Top2对抗→综合 | ✅ |
| **A 单方案对抗** | 已有明确倾向方案时 | 起草1方案→3攻击者对抗→综合 | ✅ |
| **B 纯评分（逃生口）** | 「纯评分」「score-only」/ `/2pp:score-only` | 起草3方案→评分→输出最优 | ❌ 标 `assumption2_risk:high` |

> 默认走 **模式C**（兑现技能 description 的 Adversarial Verify：探索→Plan→对抗验证→计划）。模式B 是显式逃生口——明确不要对抗时才走，`50-decision.md` 标 `assumption2_risk: high`，下游 cc-loop 自动加严验证闸。

> **Workflow 增强（opt-in 双轨）**：模式C 下说 `ultracode`/`fan out subagents` → 起草+攻击升级为 Workflow 脚本 fan-out（parallel 真并行 + 可 resume），agent 仍写文件（保假设4，不用 schema）；不说则默认手动 spawn。详见 cc-2pp「Workflow 增强模式」。

**plan mode（轻量，单方案）**——方向明确 / 单点改动 / 验证类任务用，不走判官小组+对抗。触发：「轻量 plan」「快速 plan」「单方案 plan」/ `/2pp:plan`。三档粒度（`plan_granularity`）：

- `minimal`（极简）：Explore 0a + 编排者直出，跳 web/对抗（`assumption2_risk: high`）
- `moderate`（默认）：Explore 0a+0b(≤5轮) + 单方案 + 三项好估自检
- `full`（完整 plan）：Explore 全套 + 单方案 + 1 攻击者（介于 moderate 和 2pp 之间）

plan mode 用编排者自评清单（5 项：完整度 / 三项好估 / 认知锚定 / 可编排性 / 终态可证伪）替代多 agent 对抗，落盘 `25-self-check.md`；2 轮自评仍 fail 建议升级 2pp。

**完整流程（默认模式C）**：

```
/cc-2pp <你的决策问题>
  → Phase 0   Explore（0a 内部 + 0b 外部 web ≤15轮 + 0c 挖需求）
  → Phase 2a  并行起草 3 方案（α保守/β平衡/γ创新 → 10-plan-{α,β,γ}）
  → Phase 2b  编排者评分 → 独立 30-score.md（★禁并进 40-synthesis）
  → Phase 2c  对 Top2 启动 3 攻击者（20-attack-{A,B,C}，全 opus）
  → Phase 2d  综合 40-synthesis（引用 30-score）→ 有效攻击修复循环
  → Phase 3   裁决 50-decision
  → Phase 3.6 架构设计 55-architecture（选型后、实施前必出）
  → Phase 4   实施计划 60-impl-plan + 70-requirements
```

产出落盘 `.hcc/decisions/{date}-{slug}/`（完整套件：10/20/30/40/50/55/60/70）。

**权限模式 `permission_scope`**（在 `.claude/hcc-config.json` 配，决定 Phase 4 产出 plan 后怎么衔接下游执行）：

| 值 | Phase 4c 行为 | 适用 |
|---|---|---|
| `workspace` | 全自动——产出 plan 直接衔 cc-loop 执行（不问）| 信任决策、要全自动 |
| `approval-required`（默认）| ExitPlanMode 审批——plan 呈现，你批准才执行 | 重大决策要点头 |
| `read-only` | 停在展示——plan 只读，你手动复制去执行 | 最保守、只看方案 |
| `dynamic` | 按决策重大性动态——重大(score_2pp≥5)才审批，非重大自动衔 cc-loop | "只重大才问" |

> 无 `.claude/hcc-config.json` 时：`thinking_depth` 默认 `2pp`（→模式C）、`permission_scope` 默认 `approval-required`。

### B. 长期自治任务（cc-loop + cc-goal）

适用：多步迭代、怕跑偏的任务。

```
1. cc-goal    定终态条件（可证伪：命令 + 期望输出，禁「适当/合理」）
2. /loop      自驱循环（每轮重读锚文件防漂移 + 护栏：最大迭代/无进展/预算）
3. 闭环反馈   每轮验证闸 + 经验蒸馏（compact-snapshot 续跑）
```

### C. 产品从 0 到 1（venture-pipeline）

适用：做个产品（不只是写代码）。

```
N1 机会调查 → N2 竞品 → N3 七维决策 → N3.5 需求 PRD（grill-me 打磨）
  → 架构设计 → 原型 → N5 验证 → N6 产品化 ⇄ N7 UIUX → N8 规模化
```

- 普通段：`set-signal green` + `advance` 推进
- HG 闸：`awaiting_human` 停等，boss `resolve-hg` 越闸
- monitor：用户不显式声明变更时，主动漂移检测（baseline 50/60/70 vs current diff/stagnation）

### D. 多 agent 编排（cc-orchestration + Workflow opt-in）

适用：多 agent 并行/流水线（审查/研究/迁移/对抗验证等**通用编排**，非设计决策）。

**三个模式**（谁持有计划）：
- **Subagent**（Agent tool）：独立子任务并行，无状态一次性（如并行搜多文件）
- **Workflow**（动态脚本）：固定多步骤流程，有序可重复——**说 `ultracode` / `fan out subagents` 触发**（opt-in，因烧 token）
- **Team**：角色分工协作，有状态持续交互（如前端+后端+测试）

**Workflow opt-in 触发词**（官方 tips）：

| 触发词 | 效果 | 使用场景 |
|---|---|---|
| `ultracode` | 写脚本编排多 agent（parallel/pipeline，可 resume、确定性）| 多阶段 fan-out、大规模迁移、对抗验证 |
| `fan out subagents` | 派一组 agent 各自深挖（广度覆盖，不漏）| 多维度审查、多源研究、撒网探索 |

> **反场景**（别 opt-in，普通 subagent 够）：单次查找 / 单文件改 / 1-2 个独立小任务 / 简单问答——白烧 token。
> **设计决策类**（判官小组+对抗）说 `ultracode` → **cc-2pp 模式C**（已内置起草+攻击 fan-out 脚本，别在通用编排里重造）。

## 4. 配置层（成本从低到高）

| 层 | 成本 | 作用 |
|---|---|---|
| hooks | 零（不占 context）| compact-snapshot 续跑 / 微信通知 / 自动 lint |
| `contracts/` | 按需 | 三类契约单一来源（语义锚引用）|
| `memory/` | 按需 | 跨会话记忆（用户偏好 / 反馈 / 项目 / 引用）|
| `CLAUDE.md` | 常驻 | 项目指令（每个会话加载）|
| skills | 按需加载 | 触发时加载，省 context |

## 5. 核心约束（套件设计原则）

- 技能扫描来源优先级：project > personal > skills-cli > omc > custom > builtin
- Loop 指南中 300s 间隔是反模式（缓存冷 + 没省频率），**不要用**
- Goal 终态条件必须自评通过（可证伪 + 原子 + 最弱依赖）才输出给用户
- 循环合同必含三护栏：最大迭代数 / 无进展检测 / 预算上限
- 配置六层有明确上下文成本（Hook 零 / CLAUDE.md 常驻 / Skill 按需）
- 锚文件体系（VISION/CLAUDE/AGENTS/PROMPT）是循环稳定性基础

## 7. 常见问题

**Q：技能加载占 context 吗？**
A：按需加载——只在触发时载入对应子技能，不一次性全载。路由器 `claude-coach` 常驻但精简（只诊断分发）。

**Q：和 OMC（oh-my-claudecode）什么关系？**
A：how-claude 是 Loop Engineering 方法论的**教练**（教你设计循环），OMC 是**编排层**（多 agent 协调）。两者可互补。

**Q：适合谁？**
A：想把 Claude Code 用到极致的人——尤其是长期/复杂任务（怕跑偏、怕忘记、怕幻觉）。
