## §1 背景与目标

（业务级定性成功标准，主题8——工程级定量指标归 §4 非功能需求）

**问题**：search 流程当前无统一接口，各消费方自行实现，导致结果不一致、延迟不可观测、离线无降级、超额请求直击后端。

**成功标准（业务级）**：用户经统一 search 接口完成查询，系统在可接受延迟内返回匹配结果；离线场景有降级返回而非报错；超额请求被限流而非击穿后端。

> 成功标准 = 业务级定性语言（"用户能完成 X 流程 / 系统能交付 Y 输出"），**不用工程级定量指标**；工程级定量（p99 延迟 / 可用性 / 资源占用）见 §4 并 mirror 到 §5 AC。

## §2 范围与边界

项目类型: MVP ← 因 N3 决策结论标注需最小可用版本（闸A cite 证据，防 agent 凭空自选类型绕 §4/§6）

## §3 功能需求

（EARS 句式，闸C 完整枚举五种；每条含可观测判据——验证动词/数值/命令，为 §5 AC 预留映射，主题6）

**R1（Event-driven）**: WHEN 用户提交搜索请求且查询长度 < 100 字符，系统 SHALL 在 2s 内对有效查询返回匹配结果列表，结果按相关性排序，列表非空时附命中计数。

**R2（State-driven）**: WHERE 系统处于离线状态（依赖服务不可达），系统 SHALL 返回本地缓存结果并在响应体显式标注 "离线模式"，缓存为空时返回空列表而非抛错。

**R3（Unwanted）**: IF 单源查询速率超过 100 QPS 持续窗口，THEN 系统 SHALL 返回 HTTP 429 并在响应头标注 Retry-After（秒），防止超额请求击穿后端。

**R4（Ubiquitous）**: 系统 SHALL 对中英文查询统一处理（Unicode 正常化 + 双语分词），不因语言差异导致结果偏移或编码错误。

**R5（Optional）**: WHERE 请求方携带管理员角色凭证，系统 MAY 在响应中附加审计日志面板字段（含操作者 / 时间戳 / 查询词脱敏）。

> 每条 R{n} → §5 AC{n} 一一对应（闸B）。触发条件（WHEN/WHERE/IF）∈ §2 in-scope（grill-me 回炉边校验，主题2：§3 R{n} 触发条件必落在 §2 边界内）。

## §4 非功能需求

（工程级定量，主题8——§4 定量指标必须 mirror 到 §5 AC，闭环主题6 上游）

**NFR1（性能）**: p99 响应延迟 < 500ms（负载 100 QPS）→ mirror §5 AC1（bench 阈值）。

**NFR2（可用性）**: 月度可用性 ≥ 99.9% → mirror §5 AC2（离线降级不报错，可用性下游 AC）。

**NFR3（资源）**: 单实例内存占用 < 512MB → mirror §5 AC3（限流防护下 RSS 采样）。

> §4 与 §3 同源思考（主题7，非 DAG 并行执行）——生成 §3 功能需求时同时想非功能约束，不是两个独立任务拓扑并行。

## §5 验收标准

（R{n}↔AC{n} 一一对应，闸B；每条 AC 绑可执行验证命令，主题6——N5 验证 = 跑这些命令，**不是 git diff 文本对照**）

**AC1（对应 R1 / mirror NFR1）**: 执行 `node bench/search.bench.js --qps=100 --warmup=10s` → 连续 60s 采样 p99 < 500ms，进程 exit 0。（R1 业务级 2s 内 + NFR1 工程级 500ms；过 AC1 必过 R1）

**AC2（对应 R2）**: 切断下游依赖后调用 `curl -s http://localhost:3000/api/search?q=test` → 响应体 JSON 含字段 offline_mode=true 与 "离线模式" 标注，HTTP 200。

**AC3（对应 R3 / mirror NFR3）**: 以 150 QPS 压测 `/api/search` 持续 10s → 超额请求响应码 429，Header 含 `Retry-After: \d+`（秒）；同时单实例 RSS < 512MB（mirror NFR3）。

**AC4（对应 R4）**: 执行 `node test/i18n.test.js` → 中文 / 英文 / 混合查询 case 全绿，断言无编码偏移，exit 0。

**AC5（对应 R5）**: `curl -s -H "X-Role: admin" http://localhost:3000/api/search?q=test` → 响应体含 audit_log 字段且 query 已脱敏。

> AC 绑可执行验证命令为 N5 实装建接口（主题6 N5 验证映射上游）。N5 实装时（M5/M6，现 placeholder）建立 AC↔验证命令映射表真正消费。

## 变更流程引用

本基线经产品部 venture-product-requirement 写定（工程六块，非商业五块）。后续需求变更走：

- 开发部 hcc-dev 提申请 → `N3.5_需求变更_request.md`（原需求 R{n} / 拟改 / 技术理由 / 影响范围，条目 ID R{n}）。
- 产品部评审 append → `N3.5_需求变更_changelog.md`（`[需求ID R{n}][日期] 原:... 改:... 技术理由:... 决策:接受/否决`）。
- 决策=接受 → 同步改本基线对应 R{n}/AC{n} + 通知开发部；决策=否决 → 基线不变，changelog 留否决理由。

> N5 验证对照基准（产品部被动 review）：`git diff` 检测 N4 实施 vs 本基线走偏，对照粒度 = §5 AC{n} + §3 R{n} 条目 ID（非整份 PRD 文本），见 `hcc-product/SKILL.md` §4.1。
