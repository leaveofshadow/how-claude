### [2] γ 编排力造假：fan_out 原语首发 reserved 不用 = 付了编排原语的价却拿不到任何编排收益（双重亏）

- 严重度: **CRITICAL**
- 证据:
  - γ §3.1 原语表：fan_out / fan_in "**⚠️ 首发 reserved 不用，引擎支持**"；subgraph 同。

  - γ §4.4 line 523"fan_out 组的多个 target 节点在同一 tick 内 dispatch（受 concurrency_cap=2 限制）"——这是 γ 宣称的编排力来源。

  - 30-score §3 line 99 已点破："fan_out 原语天生支持并行**但首发 reserved 不用**——编排能力造了却不用，反而因复杂度拖累"。
- 影响:
  - **双重亏**：①付了 12k token + 4 轮的编排原语实现 + 测试代价；②首发 cc-venture 实际并发仍 = 1（线性 DAG，frontier 每次只一个节点）。即"为通用性付了价，却拿不到编排收益"——编排收益（worktree 并发）要等"第二个流水线"才兑现，而那时才验证通用性（γ §12.1 自陈）。
  - 这直接打击 γ 的可编排性维度（30-score §3 给 γ 可编排性 9 分，理由"fan_out 原语支持并行但首发 reserved 不用"）——**9 分是虚的**，首发可编排性实际 = β（线性，并发 1）。γ 在可编排性上**不优于 β**，却多付 12k token。

- 修复建议:
  - **若选 γ 则必须首发就用 fan_out**——否则砍掉。但 cc-venture 是线性 DAG（50-decision §2.1），首发无处可用 fan_out → 砍掉 → γ 退化为 β。**这是 γ 无法自圆的逻辑死结**：要么首发用（无场景），要么砍掉（退化为 β）。
  - 结论：γ 的 fan_out/subgraph 在首发 cc-venture 场景下**不可能存活**，必然降级。这条攻击不是"可能"，是"必然"。

### [3] β/γ 共享致命盲区：ScheduleWakeup 在笔记本休眠时不触发，直接动摇 charter 7×24 承诺（γ 完全未提，β 标"中"风险但缓解无效）

- 证据:

  - **γ 全文 grep "休眠" 零命中**——γ §4.3 ScheduleWakeup 用法、§12.2 风险登记 R-γ1..R-γ10 **完全没有休眠盲区这一项**。γ 用 ScheduleWakeup（§4.3 line 498）却没识别这个盲区 = **设计盲区**。
  - β 的缓解"休眠恢复后 cron 补触发"**无证据支撑**：CronCreate 的 `durable:true`（β §4.4 line 436、γ §4.3）保证"任务持久化到 `.claude/scheduled_tasks.json` 跨 session"（CronCreate 工具描述），但**持久化 ≠ 休眠唤醒后自动补跑错过的 tick**——CronCreate 是"下次匹配时间触发"，不是"补跑错过的所有时间点"。笔记本盖盖子休眠 8 小时，期间本应触发的 tick **不会在唤醒后补跑**，只会等下一个 cron 匹配。

  - 这条攻击对 β/γ 都成立，且**无法通过架构层修复**（休眠是 OS 级事件，层2 纯 Node fs 无法阻止休眠也无法唤醒）——只能靠"boss 手动唤醒后 /venture-resume"（β §4.4 已有此入口），但这等于**承认 7×24 不成立，降级为"boss 在线时推进"**。

  - **诚实承认**：7×24 在单机笔记本 + ScheduleWakeup 模型下**不可能真正成立**。charter 的 7×24 应重新定义为"**会话级断点续传**"（session 结束可续），而非"**wall-clock 7×24 推进**"（休眠也推进）。
  - β 的 /venture-resume（§4.4）是正确的退路——boss 唤醒后手动续跑。但 β 必须在 charter 层面**显式标注"休眠期间停滞，唤醒后手动 resume"**，不能让 7×24 承诺悬空。
  - γ 必须补登 R-γ11（休眠盲区），且不能标"低"——这是高频确定性事件。
  - **这条攻击不否决 β/γ 的架构，但否决它们对 7×24 的兑现度**——若 charter 7×24 是硬验收点，β/γ 都需 charter 层重新谈判。

### [6] β 可编排性实为 0：3 个 stage-loop 被 HG 严格串行隔离，无无依赖并行单元可拆 worktree

- 严重度: **MAJOR**（核验项：可编排性）
- 证据:
  - β §4.1 三 stage 划分：Stage A (N1→N2→N3) → **HG1** → Stage B (N4) → **HG2** → Stage C (N5→N6⇄N7→N8)。**HG1/HG2 是 boss 决策点，必须串行等待**。
  - β §4.1"为什么三段而非一段"理由 2："HG 决策是 stage 边界天然断点，loop 在此自然终止/重启"——即 **stage 间是强串行**（前 stage 必须等 HG 决策才能启动下 stage）。
  - stage 内部：Stage A 的 N1→N2→N3 是线性 edge（condition:always，β §9.1 E1/E2），**N2 依赖 N1 产物，N3 依赖 N2 产物**——stage 内也无并行单元。
  - 唯一候选：Stage C 的 N6⇄N7 互锁。但 β §9.3 / loop_back 原语定义是"**lockstep 单调收敛**"（M2：N6 仅收窄 segment）——这是**收敞性的来回迭代**，不是独立并行（N6 和 N7 不能同时跑，必须 N6 完→N7 完→检查收敛→回 N6）。30-score §2 line 70 自己都承认"N6/N7 互锁阶段可占 2 槽位（互锁是收敞性的来回，非独立并行——**实际仍偏串行**）"。
  - β §3.3 主动放弃 fan-out："并行由节点内 skill 自行用 Subagent"——即 β 把并行**推给节点内部**（skill 自决），**层2 不提供任何跨节点并行能力**。
- 影响:
  - β 的 worktree 并发候选 = **0 个真正无依赖并行单元**。3 个 stage 串行（HG 隔开），stage 内线性，N6⇄N7 是串行互锁。
  - 30-score §2 给 β 可编排性 11 分是**虚高**——11 分基于"每 stage 一 loop 可编排"，但可编排性维度（30-score §0）的定义是"**能拆成 ≤2 并发的 worktree 工作单元**"。β 拆不出 2 个并发单元，实际可编排性应 ≤ α（α 13 分，但 α 也是串行——这里 30-score 的评分标准本身有问题，α/β 都串行却给高分）。
  - 即 β 的"可编排性"和 γ 一样，在首发 cc-venture 线性 DAG 下**都是 1（无并发）**。β 不比 γ 强（γ 造假 fan_out，β 诚实承认无 fan-out），两者首发可编排性**等价 = 1**。
- 修复建议:
  - β 应诚实承认首发可编排性 = 1（线性 DAG 无并行），不追求 worktree 并发。
  - 可编排性的真实价值要等"第二个流水线有并行节点"才兑现——这又回到 [5] 的"未来假设"问题。即期 β/γ 可编排性都 = 1，**不构成 β 优于 γ 的区分项**。

### [9] β 的双写竞态（F2）在产品+运维视角下比架构视角更严重：HG 是高频操作点，竞态窗口 = 每次 HG 进入/退出

## 总结

### β 能否存活？—— **有条件存活**

- **存活条件**：
  1. 接受嫁接 1（HG 只写 pipeline-state，消除 F2 双写竞态）——这是 β 落地的头号工程风险，必须修。
  2. 接受即期可编排性 = 1（线性 DAG 无并行），不追求 worktree 并发。
  3. 接受 7×24 降级为"会话级断点续传"（休眠停滞，唤醒后 /venture-resume）——这是 charter 层重新谈判，不是 β 架构能解决的。
  4. 工作量估算显式分离"引擎验证"（~45k）与"业务 skill 建设"（后续）。
- **不死理由**：
  - β 比 γ 诚实（不造假 fan_out 编排力，F1/F2/F3 三弱点全自曝）。
  - β 的通用性（dag.json 驱动）有即期可证伪点（cc-venture 跑通即证骨架正确），不像 γ 依赖未验证未来。
  - β 是 γ 的降级目标（γ §12.1 自陈），是同一设计轴的 MVP。
- **残余风险**：F2 双写竞态在 autocompact 下中高频，必须嫁接 1 修复；7×24 在单机休眠下实质不成立，需 charter 重新定义。

### γ 能否存活？—— **即期场景下不能存活，必然降级 β**

- **不能存活理由**：
  1. **即期 ROI 为负**（攻击 [1]）：114k token（4 倍 α）换"未来第二个流水线才验证的通用性"，其中 ≥12k 是首发零收益的 reserved 代码。
  2. **编排力造假**（攻击 [2]）：fan_out 首发 reserved 不用，付了 12k token + 4 轮的编排原语代价，首发实际并发仍 = 1。双重亏。
  3. **ROI 论证循环**（攻击 [5]）：γ 用"未来会有并行需求"这个待证假设 justify 即期投入，逻辑前提正是本次对抗要质疑的。
  4. **休眠盲区未登记**（攻击 [3]）：γ R-γ1..10 完全没有休眠项，实施时会直接撞停滞。
- **唯一存活路径**：γ §12.1 自陈退路 = 降级 β。即 γ 的"存活"= 变成 β。独立的 γ 方案在"本次只服务 cc-venture"约束下**没有存活理由**。
- γ 的价值（graph_hash、subgraph/fan_out 接口预留、graph_version 解耦）应作为**嫁接项**并入 β（30-score §6 嫁接 1+2+3），而非作为独立方案存活。

### 工作量度量核验结论

- **β**：§11 全文用 token/轮次/skill 配置/验证复杂度，**无偷渡人天/人周/排期/团队熟练度/学习曲线/人岗分工**。β §1.1"三方职责"（What/How/Who）是架构分工非人岗分工。**度量合规**。
- **γ**：§11 全文用 token/轮次，**无偷渡人天**（grep 全文确认"人天/人周/排期/团队/学习曲线/上手/熟练度"零业务命中，仅出现在约束声明"禁人天/人周"和 30-score 合规检查表）。γ §1.4"三段分工"、§10"所需 skills 清单"是架构分工。**度量合规**。
- **但**：γ §12.1 的 ROI 论证用"未来返工成本"（改核心 vs 写声明文件）——这是**隐性未来工作量假设**，不是即期可证伪度量。虽不偷渡人天，但**用未验证未来 justify 即期 token 投入**，属于 ROI 论证逻辑不合规（攻击 [5]）。
- **未揪到直接的人天偷渡**——两方案在 Phase 2 prompt 注入约束下度量纪律良好。揪到的是更深层的"**用未来假设 justify 即期投入**"（γ）和"**即期通用性零收益**"（β/γ 共有）。

### 对抗验证给编排者的核心信号

1. **γ 应被否决为独立方案**，其价值作为嫁接项（graph_hash + 接口预留）并入 β。
2. **β 作为基底**，但必须修复 F2（嫁接 1）+ 接受可编排性=1 + charter 层重定义 7×24。
3. **α 的零改动优势**（30-score §6 嫁接 1）应进一步嫁接——HG 只写 pipeline-state 可能让 shift-direction.js 也零改动，进一步降 β 的 74k 到接近 α 的 28-40k。
4. **即期 ROI 排序**：α（28k，零改动）> β（74k，需修 F2）> γ（114k，即期负 ROI）。在"本次只服务 cc-venture"约束下，**α+嫁接**才是即期最优，β 是带通用性溢价的次优，γ 是即期不合理的过度设计。
