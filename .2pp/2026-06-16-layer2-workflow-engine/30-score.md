## §1 方案 α — 保守派（最小通用 · 二原语单 DAG-loop）

**核心架构**：二原语（node + transitions），单 DAG-loop。pipeline-state 最小字段集（含 subloop 是 N6⇄N7 专用 hack）。condition 7 枚举。**shift-direction.js 零改动** → 基线层 18 测试绝对不破。HG 停等靠 pipeline-state.gate_status="awaiting" + loop 不 ScheduleWakeup。

| 维度 | 得分 | 一句理由 |
|------|------|---------|
| 可行性 | **12** | 二原语极简，纯 fs 可落地；subloop hack 是已知的局部技术债，不阻塞落地 |
| 可维护性 | **12** | 代码量最小、INV 不新增（沿用 INV-1'，第四项层2 自维护），未来维护负担最低 |
| 可扩展性 | **7** | 单 DAG-loop + 7 枚举 condition，换流水线要改代码——**与用户 Q4「通用引擎」决策正面冲突** |
| 实现成本 | **13** | shift-direction 零改动 + ~28k token / 13-15 轮，三方案最便宜 |
| 用户价值 | **10** | 能兑现痛点3/4，但首发专用 = 长期价值受限，下一个流水线要重做 |
| 可执行性 | **13** | 步骤最少、验证闸最清晰，最容易拆成 /goal 一次性单元 |
| 可编排性 | **13** | 串行为主、验证简单、worktree 无竞态负担 |
| **合计** | **80** | |

**可并行性标注**：α 是单 DAG-loop 串行推进，模块间强依赖（N1→N2→...→N8），**无天然并行候选**。worktree 槽位最多用 1 个（若 N6⇄N7 互锁拆成双 worktree 各写一份再 merge，但 α 用 subloop hack 内嵌处理，不拆）。→ 喂给 60 的 worktree 分配：α 基本不利用并发。

**优势**：
1. **基线层绝对不破**——shift-direction.js 零改动，张力 C（基线层 18 测试 vs 层2 需求）的最稳解。这是 α 的压倒性优点。
2. 最快落地（28k token，4 倍快于 γ）。
3. INV 不新增，认知负担最低。

**致命弱点**：
- **L1（可扩展性死结）**：condition 7 枚举 + subloop hack = 专用流水线的形状烙死在代码里。下一个 DAG（不止 cc-venture）必须改代码。**这是对 Q4「通用工作流引擎」的根本违背**——若选 α 等于推翻用户已做的 Q4 决策。
- L2：subloop 是为 N6⇄N7 量身定制的 hack，污染了"通用引擎"的语义边界。
- L3（HG 对齐最弱）：Grep 仅 15 次提及，未明确用 direction.status:awaiting_human（层1 C1 修订的全局开关），靠 pipeline-state.gate_status 单文件承担 HG 语义——对齐层1 C1 最弱，HG 停等正确性存隐患。

## §2 方案 β — 平衡派（中庸 · 四原语每 stage 一 loop）

**核心架构**：四原语（node / edge / human_gate / loop_back），每 stage 一个 loop（共 3 个 stage-loop）。pipeline-state 独立 schema_version + pipeline_version + direction_version 三版本解耦。**HG 双文件协同**（direction.status:awaiting_human 全局开关 + pipeline-state.awaiting_human 工作流细节，Grep 34 次提及，对齐层1 C1 最强）。转移函数查 dag-definition.json（真通用：换流水线只换数据零改代码）。shift-direction.js 扩展 `--update-status` 子命令（HG 用，不换向）。新增 INV-7/8/9。

| 维度 | 得分 | 一句理由 |
|------|--------|---------|
| 可行性 | **11** | 四原语可落地、真通用；但 F2 双写竞态（direction + pipeline-state 同时改）是真风险，扣分 |
| 可维护性 | **10** | 中等——新增 INV-7/8/9 增加复杂度，但每 stage 一 loop 边界清晰 |
| 可扩展性 | **11** | 转移函数查 dag-definition.json，换流水线只换 JSON 数据零改代码——**真通用，对齐 Q4** |
| 实现成本 | **10** | ~74k token / 28-35 轮，是 α 的 2.6 倍，中等偏贵 |
| 用户价值 | **12** | 契合 Q4 通用引擎 + pipeline_version 解耦，长期跨流水线复用价值高 |
| 可执行性 | **11** | 步骤适中，3 个 stage-loop 各自可拆，验证闸清晰 |
| 可编排性 | **11** | 每 stage 一 loop 可编排；但 F1 fan-out 阻塞限制跨节点并行 |
| **合计** | **76** | |

**可并行性标注**：β 的 3 个 stage-loop 串行（HG1 / HG2 gate 隔开），**stage 内**的节点若同属无依赖可并行（如 N6 画像与 N7 需求在互锁前可双 worktree）。→ worktree 候选：N6/N7 互锁阶段可占 2 槽位（互锁是收敞性的来回，非独立并行——实际仍偏串行）。整体可编排性中等：能拆但 fan-out 受限。

**优势**：
1. **HG 对齐层1 C1 最强**（34 次 Grep 提及，双文件协同）——HG 停等正确性三方案最有保障，直接关系痛点3/4 的兑现质量。
2. 真通用（dag-definition.json 驱动），契合 Q4，且 pipeline_version 解耦了"工作流版本"与"方向版本"。
3. 四原语平衡——比 α 通用、比 γ 克制，落在 YAGNI 甜区。

**致命弱点**：
- **F2（双写竞态）**：direction.status 与 pipeline-state.awaiting_human 双文件协同 = 两个写入点，存在竞态窗口。层1 没有 direction + pipeline-state 联合原子写，β 需自己保证顺序（先写谁、失败回滚谁）——这是 β 落地的头号风险。
- F1：fan-out 阻塞——四原语没有 fan_out，跨节点并行靠手动 spawn agent，编排表达力受限。
- F3：ScheduleWakeup 跨 session 在笔记本休眠时可能不触发——β 用 /loop + ScheduleWakeup 自调度（用户 Q3），笔记本盖盖子时段是真实盲区（7×24 约束的张力）。

## §3 方案 γ — 创新派（激进通用 · 六原语 tick 推进运行时）

**核心架构**：六原语（node / edge / human_gate / loop_back / **subgraph** / **fan_out**）。每 tick 推进一个 frontier 节点（通用运行时）。Kahn 算法拓扑序。pipeline-state 引擎中立（graph_id / graph_version / graph_hash / frontier / in_flight / completed / node_status / iter_counters / subgraph_stack / fan_out_groups）。**graph_version 独立于 direction_version**（解耦图结构 vs 方向）。分层 INV（D 系 / P 系 / X 系）。evalEdge 受限 JS 子集求值。triggerGate 原子联动。shift-direction.js 扩展换向重置 pipeline-state（if exists 守卫）+ 新增 advance-node.js（轻量推进）。

| 维度 | 得分 | 一句理由 |
|------|--------|---------|
| 可行性 | **9** | 六原语复杂、evalEdge 受限 JS 求值器 + triggerGate 原子联动，落地风险三方案最高 |
| 可维护性 | **8** | 分层 INV（D/P/X 三系）+ subgraph/fan_out，维护负担最重；求值器是长期技术债 |
| 可扩展性 | **13** | 最通用——Kahn 拓扑序 + frontier 推进 + 引擎中立，任何 DAG 零改代码，**可扩展性满分** |
| 实现成本 | **7** | ~114k token / 38 轮，4 倍 α、1.5 倍 β，最贵；且 subgraph/fan_out 首发 reserved 不用却要写测试 ~12k token 纯浪费 |
| 用户价值 | **11** | 契合 Q4 通用引擎 + graph_version 解耦创新，长期价值高；但过度设计抵消部分即期价值 |
| 可执行性 | **9** | 步骤最多、evalEdge 求值器验证最复杂，最难拆成独立 /goal 单元 |
| 可编排性 | **9** | fan_out 原语支持并行**但首发 reserved 不用**——编排能力造了却不用，反而因复杂度拖累 |
| **合计** | **66** | |

**可并行性标注**：γ 的 fan_out 原语天生支持并行（一个节点 fan_out 多个子 agent），**理论编排力最强**。但 R-γ1 暴露矛盾：subgraph/fan_out 首发 reserved（cc-venture 8 节点 DAG 用不上），造了却不用 → 真正用上要等"第二个流水线"，而那时才验证通用性。→ worktree 候选：理论可占满 2 槽（fan_out 节点各一槽），但首发 cc-venture 是线性 DAG，实际并发=1。

**优势**：
1. **最契合 Q4 通用引擎**——六原语 + Kahn 拓扑序 + 引擎中立，是真正"造一个工作流引擎"而非"驱动一个 DAG"。
2. **graph_version 独立创新**——把"图结构版本"与"方向版本"彻底解耦，是三方案中最干净的状态分层（D 系方向 / P 系流水线 / X 系交叉）。
3. fan_out/subgraph 为未来多流水线预留，长期 ROI 最高（若真有第二个流水线）。

**致命弱点**：
- **R-γ1（过度设计 · YAGNI 违背）**：subgraph + fan_out 两原语在首发 cc-venture 8 节点**线性** DAG 中**完全用不上**（reserved），却要为它们写求值器、INV、测试——~12k token 的纯预防性投入，且这些代码到"第二个流水线"才被验证。**effort=max 不等于过度设计**——γ 混淆了"不省步骤"与"造用不上的东西"。
- 工作量 114k token（4 倍 α）——在 autocompact 已 thrashing 的现实下，单次 2pp 实施周期承受 38 轮的失败/重试风险最高。
- 维护负担最重：分层 INV + 求值器，未来 debug 成本最高。

> **γ 的退路 = 降级 β**：γ 自陈若 R-γ1 无法接受，砍掉 subgraph/fan_out 即退化为 β（四原语）。这条退路是 γ 的安全网，也是为什么 Top 2 选 {β, γ} 风险最低——二者在同一设计轴上，对抗后最坏情况是 γ→β 平滑降级。

## §6 嫁接建议（次优优点 → 最优，反零和）

对抗验证尚未进行，但基于评分已有 3 条明确嫁接方向（供 40-synthesis 落地）：

### 嫁接 1：α 的「shift-direction.js 零改动」优势 → 嫁接到 β/γ
α 的压倒性优点是基线层零风险。β 的 `--update-status` 扩展、γ 的换向重置都触碰了 shift-direction.js。
**嫁接法**：让层2 的 HG 状态**只写 pipeline-state**（pipeline-state.awaiting_human），**不碰 direction.js**——direction.status:awaiting_human 的全局开关改为层2 在 pipeline-state 里镜像，由层2 的推进器读 pipeline-state 决定停等，而非改 direction.set。这样既保住 α 的零改动安全性，又保住 β 的双文件协同语义（只是双文件协同改成"层2 读 pipeline-state 仿写 direction 语义"，写者仍是 shift-direction.js 的换向路径）。
> ⚠️ 这条嫁接要在 2b 对抗验证时重点检验：F2 双写竞态是否因此消除？还是只是把竞态从"双写"变成"读后仿写"的时序问题？

### 嫁接 2：γ 的「graph_version / graph_hash 独立」创新 → 嫁接到 β
β 已有 pipeline_version 解耦，但 γ 的 **graph_hash（图结构指纹）**是 β 没有的。
**嫁接法**：β 的 dag-definition.json 加载时计算 graph_hash 存入 pipeline-state；换流水线时校验 graph_hash 防止"数据漂移导致推进器读到错拓扑"。这是低成本（一个 hash 字段）高价值（防静默漂移）的嫁接，不引入 γ 的复杂度。

### 嫁接 3：γ 的「退路 = 降级 β」→ 固化为 β 的演进路径
γ 自陈退路是砍 subgraph/fan_out 退化为 β。这反过来定义了 β 的**未来演进方向**：β 的 dag-definition.json schema 预留 `subgraph`/`fan_out` 字段（reserved，首发不处理），未来需要时升级到 γ 形态。→ β 不是终点，而是 γ 的 MVP。这把"选 β 还是 γ"的二元对立，化解为"β 现在 + γ 接口预留"的演进路径。

## §7 推荐：Top 2 = {β, γ}（对抗验证对象）

### 为什么不是 {α, β}（总分最高的两个）？

**因为 α 总分虽最高（80），但其核心立场「首发专用 / 最小通用」与用户 Q4 决策「通用工作流引擎（拒绝专用 cc-venture DAG）」**根本冲突**。

- 用户在 Phase 0c 已明确 Q4 = 通用引擎（拒绝专用 DAG）。
- α 的可扩展性 7 分、用户价值 10 分，**正是**因为它选择了"专用"——它在 Q4 这条轴上是**负分**。
- 若选 α 进 Top 2，等于让对抗验证去推翻用户已做的架构决策——这违背 cc-2pp「用户优先级原则」（假设5：优先级贯穿全流程）。
- α 的真正价值（零改动、最省）应作为**嫁接项**保留（见 §6 嫁接1），而非作为独立候选。

### 为什么是 {β, γ}？

| 理由 | 说明 |
|------|------|
| **都满足 Q4** | β（dag.json 驱动）和 γ（引擎中立）都是真通用引擎，都通过 Q4 这条硬门槛 |
| **分歧是真问题** | β vs γ 的分歧集中在「抽象层次 + 通用程度 + 工作量」——这正是本次设计**唯一值得对抗验证**的真问题。α vs β 是"快/安全 vs 平衡"，但 α 不满足前提，是假分歧 |
| **退路安全** | γ 退路即 β（砍两原语），二者在同一设计轴。对抗后裁决风险最低：最坏情况 γ→β 平滑降级，不会出现"对抗后两个都否"的死局 |
| **α 价值不浪费** | α 的零改动优势通过嫁接（§6-1）并入胜出方案，不因落选而丢失 |

### Top 2 对抗验证的聚焦轴（喂给 Step 2b 攻击者）

| 攻击者 | 视角 | 对 β 攻什么 | 对 γ 攻什么 |

## §8 编排者初判（待对抗验证修正）

> 这是评分阶段的**初判**，非最终裁决。最终裁决在 Phase 3（50-decision.md），以 2b 对抗结果为准。

**倾向 β 为基底，嫁接 α 零改动 + γ graph_hash，预留 γ 的 subgraph/fan_out 接口**（即 §6 嫁接 1+2+3 三合一的"β+ 演进型"）。

理由：
1. β 是三方案中**唯一同时满足**「Q4 通用 + HG 对齐最强（C1）+ YAGNI 甜区」的——它的弱点 F2 是**工程可解的**（嫁接1 的只写 pipeline-state 可消竞态），不像 γ 的 R-γ1 是**设计层面的过度**（造了不用）。
2. γ 的通用性最高，但其代价（114k token + reserved 原语 + 求值器）在 7×24 单机 + autocompact thrashing 的现实约束下，即期 ROI 偏低。γ 的价值（subgraph/fan_out）可通过 β 的接口预留**延迟兑现**，不必现在付全价。
3. α 的零改动是宝贵的安全资产，必须通过嫁接保留——否则胜出方案白白牺牲了基线层零风险这个 α 独有的优点。

**对抗验证需要推翻此初判的关键证据**：
- 若攻击者证明 F2 双写竞态**无法**通过嫁接1 消除（时序问题无解）→ β 基底动摇，需重新考虑。
- 若攻击者证明 γ 的 subgraph/fan_out **首发就用得上**（cc-venture DAG 实际需要子图/扇出）→ R-γ1 不成立，γ 价值回归，初判偏向 γ。
- 若攻击者证明 dag.json 驱动是**伪通用**（换流水线仍要改推进器代码）→ β 的可扩展性虚高，β/γ 都需重评。
