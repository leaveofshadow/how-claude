# 项目宪章

> 本文件是 venture 三层架构规划与所有后续产出的**最高根**。任何裁决、计划、产出与之冲突时，以本文件为准。

## 使命

打造**全流程自动化的 AI 公司**。每个个体是 OPC（One Person Company），追求团队 + 多 AI 协作效率最大化（最懒）。探索 AI 时代新型公司形态。

> **Make it difference, even if a little !!!**

实践路径：一边打磨产品产生收益，一边最小化团队摩擦系数。多头注意力，并行合作。

---

## 根原则（所有产出的验收根）

### P1 · 最懒 · 用户思维
架构师、作者也要有用户思维。每个产出必须按**使用者最懒、最快有效**的方式设计。
→ human gate 不是把难题丢给 boss 苦思，而是把信息解码重编码成「boss 一眼可决策」的面板。

### P2 · 信息处理三段论
所有产品都在处理信息：**解码 → 转换 → 重新编码**。重新编码的唯一目的是让人类可阅读、可操控。
→ extractor 不止产 jsonld 给机器路由，更要重新编码成人类可读的决策面板。

### P3 · 世界最好（一个维度即可）
不要用自己的认知边界限制 AI。每个产出要求做到**世界上最好的**——哪怕一个字体、一个排列顺序、一个颜色搭配的进步，只要有一个维度最好就行。
→ 每个节点产出都要问：哪个维度是世界最好？

### P4 · 超越，不是取代
用 AI 超越自己，不是取代自己。
→ 7×24 自动化处理信息流转，boss 在 human gate 做「超越性」创新决策。AI 是放大器，boss 是创新源。

## 部署约束（硬约束，不可妥协）

| 维度 | 约束 | 架构含义 |
|------|------|---------|
| **单机** | 一台笔记本 | 不假设服务器/集群/外部依赖；所有组件本地可跑 |
| **单人** | 唯一人类 | 所有「团队协作」= boss + 多 AI agent；human gate 是唯一人工节点 |
| **单 Claude** | 唯一 AI | 所有 agent 角色（调查/竞品/计划/judge/设计/画像/需求/UIUX）都是 Claude 分饰（subagent/skill） |
| **7×24** | 会话断点续传 | checkpoint/direction/trace 是断点续传基础设施（= 层1 存在的根） |

## 组织架构（5 部门）——hcc = AI 公司员工指南（D10 定稿：A 工具箱模型）

hcc 不是"使用教程"，是**这家 AI 公司的员工指南**：教 Claude 如何作为 5 个部门的"员工" 7×24h 自主运行。每个部门 = 一个 agent 角色（Claude 分饰），具备 **plan（规划）+ review（审查）** 双能力，通过层1 的 state/trace/direction **交换上下文**，依靠**知识库 + 本地环境研究 + web 研究**决策、设定目标。

| 部门 | 职责 | venture 节点 | 主信息源 | 对应 hcc 工具 |
|------|------|------------|---------|--------------|
| **决策部** | 方向设定、可行性判断、judge | N3 计划 / N4 judge / HG | 知识库 + web（行业/竞品） | cc-2pp / cc-goal / cc-orchestration |
| **产品部** | 产品设计、UIUX、需求 | N5 设计 / N7 需求 / N8 UIUX | 本地产物 + 用户反馈 | cc-loop / 产品设计技能（新建） |
| **开发部** | 按 plan 实施、交付 | 实施（执行计划） | 本地代码 | executor / cc-loop |
| **运维部** | 7×24 保活、state/trace/Hook | 层1 运行时（贯穿） | 本地 state/config | cc-runtime / cc-config / cc-context |
| **销售部** | 画像、收益转化、市场验证 | N1 调查 / N2 竞品 / N6 画像 | web + 知识库（案例） | venture-judge / 销售技能（新建） |

**部门协作协议**：部门间不直接对话，通过层1 产物契约（state）+ 方向指针（direction）+ 执行记忆（trace）交换上下文。每个部门的 plan/review 遵循 cc-2pp（判官 + 对抗）。

> **形态定论（D10 → A 工具箱模型）**：部门 = **协作协议层**（新增 `hcc-org/`，定义职责 + plan/review 流程 + 交接协议 + 信息源），技能 = **跨部门工具箱**（现有 cc-*/venture-* 原位保留，按需调用）。重构≈0，不破坏 50-decision 技能树。

**hcc 三层总图：**

```
hcc = AI 公司员工指南
├── 组织层  hcc-org/         ← 5 部门协作协议（D10 新增，轻量）
├── 工具层  cc-* + venture-* ← 跨部门方法论与业务技能（50-decision 定型）
└── 地基层  cc-runtime        ← state/direction/trace/Hook（Phase4 先行）
```

**覆盖度盘点（D10 暴露的缺口）**：决策部/运维部 ✓ 厚实；开发部 ⚠️ 中等；**产品部/销售部 ❌ 真空**（N5 产品设计/UIUX、画像/收益转化无对应技能）→ 待层3 启动补齐，不阻塞层1 先行。

## 阶段标准

每个实践阶段以**转化成收益**为验收标准。
→ 层1 cc-runtime 是基础设施（不直接产生收益），但它解锁的层3 venture 流水线必须能跑出**可变现的产品方向**。阶段验收 = 技术验收 + 收益指向。

## 理念 → venture 三层架构映射

| 原则 | 层1 cc-runtime（地基） | 层2 工作流引擎 | 层3 cc-venture（旗舰） |
|------|----------------------|---------------|----------------------|
| P1 最懒 | checkpoint 可视化让 boss 秒懂进度 | human gate 一键三动词 | 评判卡→**决策面板**（非原始数据） |
| P2 三段论 | trace = 机器可读记忆 | DAG = 转换编排 | extractor = 解码重编码枢纽 |
| P3 世界最好 | direction 切换交互最优之一维 | gate 体验最优之一维 | 决策面板视觉最优之一维 |
| P4 超越 | 7×24 跑信息流 | AI 编排不取代判断 | boss 在 gate 做创新决策 |
| 收益为标 | 解锁可变现流水线 | 降低单方向试错成本 | 跑出可变现产品方向 |

## 创新倾向

**我们热爱创新。** 当「稳」与「创新」冲突时，倾向用工程稳健性兑现创新，而非降级放弃。
→ judge 即路由器（extractor 而非人工读卡）、装配式技能（教配方而非常驻）——这些创新方向用工程稳健去兑现，不退回保守。
