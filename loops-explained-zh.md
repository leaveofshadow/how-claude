# Loop 详解：Claude、GPT、Mira 以及真正有效的做法

AI 已经在每个人手里好几年了。大多数每天用 AI 的人，至今还在用最慢的方式：打一个请求，等，改，再问，全手动。

不是因为更快的方式有多复杂，而是因为没人给他们看过更快的方式长什么样。

更快的方式是一个 **loop（循环）**，而此刻全世界最优秀的 AI 工程师只关心这一件事。这篇文章补上没人讲清楚的那部分。

读完你会比时间线上几乎所有人都更懂 loop：它们是什么、底层到底怎么运作、什么时候值得用什么时候是陷阱、如何在 Claude 或 ChatGPT 里自己搭一个基础的、以及哪些简单的 loop 值得在你自己的生活里跑。

## 大多数人怎么用 AI？

仔细看"一次一个请求"这个习惯——它就是问题的全部。每一步都经过你：你决定问什么，你判断答案，你决定下一步。AI 除非你推，否则不动；你一停，它就停。

这没问题，但它有天花板。**你是引擎**，AI 只是你手里的工具，而工具自己什么都做不了。

还有另一种工作方式，也是全世界最优秀的工程师正在改变构建方法的原因。与其牵着 AI 走过每一步，你只给它**一次目标**，让它自己跑步骤。它规划、干活、检查自己的结果、修补薄弱处、重复，直到目标达成。**你退出来，工作继续。**

> Peter Steinberger 推文（每月提醒一次）：**你不该再 prompt 编码 agent 了。你该设计 loop 去 prompt 你的 agent。**

两位最受尊敬的工程师，用不同的话说了同一件事。大多数人读到这种话，心里悄悄承认在实践中根本不知道什么意思。所以我们好好拆解一下。

## Loop 是什么？

**prompt 是一条指令。loop 是一个 AI 持续推进直到达成的目标。** 把它想成一个递归目标：你定义一个目的，AI 迭代直到完成。

prompt 给你一个答案然后等你决定下一步。loop 自己跑完整周期：

```
DISCOVER（发现）→ 搞清楚要做什么
PLAN（规划）  → 决定怎么做
EXECUTE（执行）→ 干活
VERIFY（验证） → 对照目标检查
ITERATE（迭代）→ 还没到？把结果喂回去重复
```

这五步里有三步干了所有真正的活，也是人们用错 loop 的地方。

### 验证是 loop 的心脏

没有对结果的真实检查，你就没有 loop，你只是 agent 在反复自我附和。这个检查把"重复"变成"进步"。它可以是硬测试（"代码过不过"）、可度量条件（"数字是否高于 X"）、或模型对照打分的 rubric。**没有闸 = agent 给自己批改作业，而干活的模型当裁判太宽容了。**

### 状态让 loop 学会

每一轮，AI 必须记住它已经试过什么，否则永远重复同一个错误。真正的 loop 在旁边留一份小记录：什么做完了、什么失败了、下一步是什么。明天的运行是**恢复**，不是从零开始。这也正是它开始变贵的地方，后面会讲。

### 停止条件让它保持理智

没有出口的 loop 会一直跑到成功、崩掉、或刷空你的账户。每个严肃的 loop 都有两种停法：**成功**，和**硬上限**（"8 次后停下并汇报"）。跳过这个，你就造了一台能整晚空转烧钱的机器。

> prompt 交给 AI 一条指令。loop 交给 AI **一份活、一种知道活什么时候干完的方式、以及一条什么时候放弃的规则**。

## 你真的需要一个吗？

大多数文章先卖 loop 再告诉你什么时候是错的。这是严肃的人真正在用的测试。**只有当这四条全部成立时，才值得搭 loop：**

1. **任务重复**，至少每周一次。低于这个，搭建成本永远收不回来。一次性的活还是用一个好 prompt 更好。
2. **有东西能自动拒绝烂输出**。测试、类型检查、构建、linter、硬规则。如果没有东西能让活失败，loop 就只是空转。
3. **agent 能自己端到端干完活**，而不是把一半还给你。
4. **"完成"是客观的**，不是判断题。如果质量是口味问题，人类还是赢。

漏一条，就留着当手动 prompt。这个话题的诚实版本是：**loop engineering 是真的，而大多数人还不需要重型版本**。大家都能用的是轻型版本，后面会讲。但你该知道这条线在哪。

## 为代码打造的版本

loop 先在软件领域爆发，因为代码是世界上最容易验证的东西。**测试过就是过，挂就是挂，没得争**，所以 AI 总知道是否完成。

一个编码 loop 被给一个目标和一种严格的检查方式：

```
▸ LOOP SPEC
GOAL: /tests/auth 里所有测试通过、lint 干净、零类型错误。

EACH ITERATION（每次迭代）:
  1. 跑测试套件，读每一个失败
  2. 挑单个影响最大的失败
  3. 写最小的改动修好它
  4. 重跑测试、lint、类型检查

VERIFY: 绿测试 + 零 lint 警告 + 零类型错误
STOP WHEN: verify 通过，或达到 8 次迭代
ON STOP: 总结改了什么、还有什么挂
```

底层上，一个真正的 loop 由五个构件组装而成。Claude Code 和 Codex 现在五个都内置了。

### 1. 自动化（心跳）

这是让它成为 loop 而不是你跑过一次的一次性的触发器。你定义一个 prompt、一个节奏、一个目标，它按计划跑，不需要你启动。在 Claude Code 里，`/loop` 按间隔重跑一个 prompt，`/goal` 让会话持续到你写的条件真的为真，hooks 在 agent 生命周期的某些点触发命令，推到 cron job 或 GitHub Actions 让它在你合上笔记本后继续跑。**结果来找你，你不是那个到处检查的人。**

### 2. skill（可复用指令）

与其每次粘贴一大坨指令，你把它们存一次成一个 loop 每次都读的文件：规则、要遵循的模式、以及一份绝不能碰的硬清单。现在自动化只按名字调用 skill，循环往复的活就保持可维护，而不是烂在一个没人更新的 schedule 里。

### 3. 子 agent（让制造者远离检查者）

loop 里最有用的结构性技巧，是**把干活的 agent 和检查的 agent 分开**。写代码的模型给自己的作业打分太客气了。第二个 agent，用不同指令、有时在高 effort 上用更强模型，能抓到第一个 agent 自我说服过去的那些东西。你的写手可以又快又便宜，你的审查者又慢又严。**这个分离就是质量的大头。**

### 4. 连接器（让它行动，不是建议）

这是"这是修复"的 agent 和自己开 PR、关联 ticket、构建一绿就 ping 频道的 loop 之间的区别。连接器让 loop 在你的真实环境里**行动**，而不是只描述它如果能做会做什么。

### 5. 验证器（闸）

那个自动拒绝烂活的测试、类型检查或构建。**这是唯一决定 loop 是帮你还是只花你钱的构件。** 其他都是管道，这是让它成真的部分。

把这些摞起来，你得到大团队现在规模化跑的东西：在同一份活上 loop 的 agent 舰队，几十或几千个同时跑。有个工程师用这样的 loop 把整个代码库从一种编程语言重写到另一种，大约六天——手工要近一年。这是严肃软件怎么被构建的真正变化。**它也有个 demo 从不展示的坑。**

## 没人提的成本

loop 跑在 token 上，token 是钱。问题不是每步花点钱，**问题是成本怎么复利**。

每次 loop 转一圈，agent 重读它的上下文：目标、代码、上次结果、什么失败了。那一整堆每次迭代都过一遍模型，而且每轮都在涨。**跑十次的 loop 不是花十个 prompt 的钱，是花十个每次都在变大的 prompt 的钱。** 提升质量的制造者-检查者技巧也翻倍账单，因为现在两个模型读这份活而不是一个。

```
▸ ROUGH COST OF ONE LOOP（一个 loop 的粗略成本）
单 agent，一个中等任务:        ~50,000 – 200,000 token
每次迭代重发上下文:            每轮都在涨
并行 agent 舰队:               把上面全乘起来
```

真正重要、几乎没人跟踪的指标，是**每个被接受的改动的成本**。不是花了多少 token 或跑了多少 loop。如果 loop 给你十个结果你扔掉六个，你在干它本该省掉的审查活。**接受率低于 50%，它花得比给得多。**

loop 也会**安静地失败**。工程师 Geoffrey Huntley 管这叫"Ralph Wiggum loop"：agent 太早觉得自己干完了，在一个半成品上退出，loop 还在跑还在花钱却不产出任何东西。没有能让活失败的硬闸，loop 不会崩，**它在沉默中给你记账**。

这就是为什么重型版本属于有预算和护栏跑它的团队：迭代上限、token 预算、无聊步骤用便宜模型、监控。如果这不是你，你没错过什么——核心想法以零头成本和零设置就能用。

## 真正有效的顺序

如果你真要搭一个，**顺序比工具更重要**。把能活到生产环境的 loop 发布出去的人，全都这么做：

1. **先把一次手动跑跑可靠。**
2. **把它变成 skill**（存下指令）。
3. **把 skill 包进 loop**（加闸 + 停止条件）。
4. **然后才放 schedule 上。**

跳到前面，把还没手工跑可靠的玩意儿放 schedule 上，正是 loop 怎么在你睡觉时炸掉的方式。**先证明一次、加固、再自动化。**

## 自己搭一个基础 loop（任何 LLM）

你不需要编码 agent 才能感受这怎么运作。你现在就能在任何 LLM 里手工跑一个简单的 loop，只用一个 prompt。技巧是**一次性给模型 loop 的三部分**：一个目标、严格的成功标准、以及一个强迫它在允许停下前自检的协议。

```
▸ SELF-CHECKING LOOP（粘贴到 Claude 或 ChatGPT）
你将在一个 loop 里工作，直到任务达标。

TASK:
[精确描述你要产出的东西]

SUCCESS CRITERIA（要严格，不放软水）:
- [标准 1]
- [标准 2]
- [标准 3]

LOOP PROTOCOL，每轮重复:
1. PLAN   - 说出单个下一步。
2. DO     - 产出或改进工作。
3. VERIFY - 在每个标准上给结果打 1-10 分。
            残酷诚实。精确列出还弱在哪。
4. DECIDE - 如果每个标准都 8+，打印 "FINAL" 并停。
            否则打印 "ITERATING" 再来，先修最弱的点。

RULES:
- 在每个标准达到 8 或更高前，绝不叫完成。
- 每轮必须修上一次 VERIFY 里最低分的点。
- 不要问我问题。做一个合理假设、记下来、继续。

开始。跑到 FINAL。
```

看会发生什么。模型起草、对照你的标准给自己作业打分、找到弱处、重写，一遍又一遍，直到真正过线，而不是塞给你第一个看起来差不多的东西。**那就是一个 loop。你用一段话搭了一个。**

但注意还缺什么，因为这是接下来要讲的全部重点。**你是触发器。** 你开了聊天、粘了 prompt、你坐在那看它迭代。关掉标签页就没了。没有 schedule，没有"每天早上做这个"、没有"邮件到了就醒"。它够不到你，因为它只在你看着它时存在。

要得到一个自己跑、按 schedule、被真实事件触发、不用你 babysit 的 loop，你通常得踏进前面那个重型世界：工具、托管、代码、闸、和一张账单。

这在处理真正重的任务时合理。但 99% 的日常任务，已经有一个现成、极简的解。

## 同一个想法，为你真实的生活

剥掉代码和成本，剩下的是一个简单、真正有用的概念：**一个自己跑的任务，按 schedule 或某事发生的那一刻，不需要你记得它或在场。** 你不需要为这个当工程师，你只需要为生活而不是为代码库搭的 loop。

有个免费选项，你用大白话描述就能创建一个。没代码、没托管、没 key、没要开着的标签页、没会搞错的构建顺序。

它叫 **Mira**，住在 Telegram 里——这个 app 你多半已经开着。你像跟朋友一样给它发消息，它跑的 loop 叫 **Skill**。每个 Skill 悄悄都有真 loop 需要的同样部件——触发器、动作、自己跑的方式——只是你从不用把它们接起来。**你只要说你要什么。**

```
▸ SKILL
"每个工作日早 7 点，查我的 Gmail 和 Google Calendar。
发我一份简短简报：我最重要的 3 个会议、收件箱里紧急的、
以及一件我说了要跟进却没跟的。控制在 120 字以内。"
```

那是一个真正的 loop。时间触发器、跨两个连接 app 的多步动作、自己跑、来找你。**你写成了一条消息。**

## Mira 实际能做什么

这是让它 click 的部分。Mira 不是更聪明的聊天机器人。和 ChatGPT 的区别很简单：**ChatGPT 回答，Mira 行动。** 你不是让它写邮件，你是告诉它发邮件。你不是拿到一份 ticket 草稿，你是拿到一条真在 Linear 里、分配好负责人的 ticket。它在后台把事干了，而且每次对话之间都记得你。

它通过 Composio 连 500+ app（Notion、Gmail、Google Calendar、GitHub、Figma、Stripe 还有几百个），它有跨会话和群聊保持的长期记忆，而且模型无关——按任务跑 GPT、Claude、Gemini。这能变成什么：

### 工作
这是 loop 想法不写一行代码就回本的地方。

```
▸ SKILLS
"每个会议前一小时，用我和那个人上次对话的上下文和决策提醒我。"

"我转发一条消息到这里时，把它变成一条 Linear ticket，设对优先级、分配负责人。"

"每周五下午 4 点，收集团的任务状态和指标，在我们聊天里发一份干净的周报摘要。"

"用 5 个要点总结我不在时这个群聊里错过的所有东西。"
```

它几秒内让你跟上 200 条消息的线程、边聊边把 ticket 建好、带着准备好的背景走进会议。在群聊里它记得团队的决策和任务，不只你的。

### 创作者
这是大多数人低估的部分。Mira 在聊天里端到端做内容。

```
▸ SKILLS
"我发一条带原始想法的语音备忘。把它变成带文案和标签的成品帖。"

"拿这一个想法，为 X、Instagram、LinkedIn、邮件和 newsletter 各写一版，每个用对的格式。"

"给这条帖生成 3 个图片选项。"

"把这张图变成给我 Telegram 频道的短视频。"
```

语音备忘进，约 30 秒成品帖出。**一个 brief 变成六个平台原生版本。** 它在聊天里直接生成图片和视频、编辑照片、换背景、做吉祥物和头像、甚至对口型和动画化。整条内容管线活在一个窗口里。

### 语音
Mira 把语音当一等公民输入，这比听起来重要。

```
▸ SKILLS
"把我的语音消息转成干净文本。"
"把这篇文章作为音频读回给我。"
"把这个群聊里的语音备忘总结成要点。"
```

它转写你的语音消息、把文本读回给你、理解群聊里的语音备忘并总结讨论，还在你不能打字时当免提语音助手。

### 生活
同一台引擎，指向其他一切。

```
▸ SKILLS
"每晚 7 点，问我今天练了没。保持连续记录，别让我悄悄跳过超过一天。"

"每晚问我 3 个关于今天的问题，记住答案，每周一次告诉我什么变了。"

"从我一盘菜的照片跟踪我的卡路里。"

"盯这条航线，价格掉到我说的数就买。"

"每天早上给我一份我关注话题的去标题党新闻摘要。"
```

一个盯着你连续记录的教练。一本真记得你、随时间变成打卡伙伴的日记。拍照跟卡路里、不用单独 app。从你自己的错误建的语言练习。一个价格对了就买的航班监控。一份去掉标题党的每日摘要。
