# 如何在 Claude Code 中掌握动态工作流：6 种模式与 14 个步骤——Anthropic 的工程师们实际采用的技巧

> 关注我的 Substack 账号，获取最新的 AI 相关内容：movez.substack.com

大多数 Claude Code 用户仍然手动编写自己的工作流程。他们依次输入各种提示词，复制输出结果，再将这些结果粘贴到下一个提示词中。如果出现问题，就再重新尝试。

10 个构建者中有 9 个甚至从未试用过"动态工作流程"功能，尽管该功能早在两周前就已经上线了。

他们编写了 50 条提示语，而实际上只需要 1 个工作流就足够了。这就是 Anthropic 的工程师们实际使用的 **14 步操作流程和 6 种处理模式**——这些模式被用于数据迁移、研究分析、数据排序、问题根源分析、问题分类以及评估等任务中。

"动态工作流"功能于 2026 年 5 月 28 日随 Claude Code 一同发布。Claude Code 的默认设置非常适合用于编程任务。不过，对于某些类型的任务来说，这种默认设置就不够用了：比如那些需要**长时间运行、需要大规模并行处理、结构复杂或具有对抗性**的任务。

对于那些需求来说，Anthropic 过去会自行开发定制化的解决方案。而借助动态工作流功能，**Claude 能够以 JavaScript 语言，根据用户的具体需求，即时生成定制化的解决方案**。

**14 个步骤，6 种模式。只需一个工作流，无需 50 条提示。**

## 第一部分 · 心智模型

### 01. 工作流是由 Claude 所编写的程序

默认的 Claude Code 开发环境允许在同一个上下文窗口中同时进行规划和执行。对于大多数编程任务来说，这种设计非常实用。但对于那些需要长时间运行、需要并行处理或涉及对抗性计算的任务来说，这种设计就不适用了。

在动态工作流中，Claude 会为自己编写专门的脚本来处理这项任务。这些脚本其实是 **JavaScript 文件**，其中包含了一些特殊函数，用于启动和协调各个子任务的执行。此外，这些脚本还使用了标准的 JavaScript 功能（如数学运算、JSON 处理、数组操作等），以便处理各个子任务之间传递的数据。

**这三种功能是默认环境所无法实现的：**

- **每个智能体都是独立运行的**。每个子智能体都有属于自己的处理空间和目标。不存在相互干扰或数据交叉污染的情况。
- **每个智能体的模型选择**：工作流会决定每个子智能体使用哪种模型——Opus 用于复杂推理，Haiku 用于简单探索，Sonnet 用于中间任务。
- **每个代理的隔离级别**。可以是工作树模式（即独立的 Git 分支），也可以是远程模式（无需进行任何分支操作）。具体采用哪种模式由工作流来决定。

可以通过直接询问 Claude 来启动（例如："创建一个这样的工作流……"），或者使用触发词 `ultracode` 来启动。如果工作流被中断了，重新启动会话后，工作流会从之前中断的地方继续执行。

### 02. 三种故障模式均已得到解决

要判断某个工作流是否是合适的工具，首先需要明确它究竟能解决什么问题。Claude 在单个上下文窗口中处理复杂任务的时间越长，就越容易出现三种特定的故障模式：

- **代理型懒惰**——Claude 在完成一项复杂的多步骤任务时半途而废，只完成了部分工作就宣称任务已完成。在安全审查中，只处理了 50 项任务中的 20 项，其余的则被标记为"已处理"。
- **自我偏好偏差**——当被要求根据某些标准来评估或验证自己的结果时，Claude 总是更倾向于选择支持自己结果的选项。那些与结果有直接利害关系的人，自然无法做到公正的评估。
- **目标偏离**——即在多次迭代过程中，目标逐渐偏离了最初的设定，尤其是在经过多次压缩之后。每一次总结过程都会导致目标的部分丧失。"不要做 X"这样的约束条件，在第 47 次迭代后就不再被遵守了。

工作流从结构上解决了这三个问题：**将不同的任务分开处理，确保每个任务都有其独立的上下文、明确的目标和独立的运行状态**。如果你的任务存在上述问题中的任何一个，那就说明需要使用工作流来解决问题了。

### 03. 静态工作流与动态工作流

静态工作流具有通用性：只需编写一次，就能应对所有可能的情况。虽然这种方式可行，但必须采取保守的态度来设计。

动态工作流则有所不同：**Claude 会为每项任务量身定制相应的工作流**。其工作流程会根据具体的使用场景来调整：先读取计费代码，再对照新供应商的文档来检查各项功能是否符合要求；接着根据交易量来确定价格；最后，还会从反面思考"为何不进行迁移"。静态方式无法做到这一点，因为它根本不知道你的代码的存在。

### 04. 核心 API：`agent()`、`parallel()`、`pipeline()`

三个函数起着核心作用：

- `parallel()` 是一种同步机制：它会让所有任务同时开始执行，但会等待所有任务完成后才返回结果（barrier）。
- `pipeline()` 则是一种流水线式处理方式：每个任务独立地经过各个处理阶段（streaming）。

**根据问题来选择**：在继续下一步操作之前，我需要获取所有结果吗？是 → parallel。否 → pipeline（成本更低，整体效率更高）。

## 6 种模式

### 05. 分类处理。先对工作进行分类，然后再着手处理

分类器会先确定任务的类型，然后根据分类结果，将任务分配给不同的处理单元或执行相应的操作。

**当这种模式能够发挥其作用时：**
- 任务具有多样性——不同的子类型需要采取不同的处理方式。
- 想要把贵模型只用在复杂度需要的地方（分类器用便宜模型，只在需要时路由到 Opus）。
- 工作的分解本身并非易事，需要借助模型来决定其具体方式。

### 06. 分步处理后再合并结果。多个小步骤，最终得到一个统一的成果

将任务拆分成许多较小的步骤。让智能体并行处理每个步骤。最后将所有结果整合成一个答案。合成步骤是个瓶颈——它需要等待所有的处理单元完成工作，然后再将它们处理后的结果合并起来。

**为什么这种模式在实践中如此有效**：它解决了单一上下文处理方式中"同时处理太多事务"所带来的问题。每个子代理只负责自己负责的部分。协调者不会被 50 个无关的细节所干扰。

```javascript
// Fan out: one agent per file. Barrier: wait for all.
const reviews = await parallel(
  files.map(file => () => agent(
    `Review ${file} for security issues`,
    { model: "haiku", schema: IssueList }
  ))
)

// Synthesize: one Opus agent merges everything.
const report = await agent(
  `Merge these reviews into one prioritized report:\n${JSON.stringify(reviews)}`,
  { model: "opus" }
)
```

### 07. 对抗性验证

这是解决"自我偏好偏差"的有效方法。对于每一个被生成的智能体，再运行另一个智能体来对其输出结果进行验证。**这个验证者从未见过原始的工作内容，因此不可能偏袒任何一方。**

**配对规则**：验证者只需了解评分标准以及相关作品本身，而无需知道是谁创作了该作品。否则由于提示信息中的暗示，验证者可能会受到个人偏好的影响。

### 08. 生成与过滤

针对某个主题，先提出若干个想法，然后根据一定的标准或经过验证后筛选出合适的想法。去除重复的内容。最终只保留那些质量最高、经过验证的优质想法。

这与向 Claude 寻求"最佳答案"的方式正好相反。如果要求 Claude 给出最佳答案，它就会过早做出决定。而采用"生成-筛选"的方式，则能让 Claude **在所有选项都被仔细评估之后再做出决定**。

### 09. 比赛模式：两两对比的评分方式优于绝对评分方式

与其分配任务，不如让各个智能体在完成任务的过程中相互竞争。创建 N 个智能体，每个智能体都用不同的方法来尝试完成同一任务。然后，通过两两比较的方式来评估各个智能体的表现，直到有一个智能体获胜为止。

**与绝对评分相比，相对评分更为可靠**——尤其是在那些需要依靠个人品味来评判的作品上。该机制本身运行在确定性的循环代码中，与具体的上下文无关。每一次比较都是快速、公平且独立的。

### 10. 重复该操作，直到完成为止

对于那些工作量不确定的任务，应持续生成循环处理单元，直到满足停止条件为止——也就是说，当不再有新的发现、日志中也没有错误出现、理论得到验证时，才停止处理。

将此模式与 `/goal` 结合使用，可以设定严格的完成要求；如果希望整个工作流能按固定的周期重复执行，则可以与 `/loop` 结合使用。**括号和停止条件都存在于代码中；只有当前正在进行的迭代会被保留在上下文中。**

## 组合模式

### 11. 为实际应用场景设计相应的模式。一个工作流可能对应多种模式

这 6 种模式很少单独出现。在实际的工作流中，通常会同时使用 2 到 4 种模式：

| 使用场景 | 模式组合 |
|---|---|
| **迁移与重构** | 扇出（每个调用点/失败测试一个 agent，worktree）→ 对抗性验证（独立 agent 审查每个修复）→ 循环直到完成。Anthropic 用此把 Bun 从 Zig 重写为 Rust |
| **深度研究**（`/deep-research`）| 扇出（并行 web 搜索）→ 对抗性验证（每条声明独立验证）→ 综合（一份带引用的报告）|
| **草稿深度核查** | 识别所有事实性声明（一个 agent）→ 扇出（每条声明一个验证者，对照来源）→ 元验证者（检查验证者的来源质量）|
| **整理 1000+ 项** | 比赛模式——两两比较、分组排序或括号排序。比较性判断，绝不用绝对评分 |
| **记忆与规则遵守** | 每条规则一个验证者（扇出）→ 怀疑者角色审查规则本身以防误报 |
| **根本原因调查** | 从零散证据生成理论（不同 agent 读日志/文件/数据）→ 每个理论的验证者+反驳者小组 → 循环直到一个活下来 |
| **大规模分流** | 分类处理 → 与现有工单去重 → 尝试修复或升级。配合 `/loop` 持续分流 |
| **探索与品味**（设计/命名/UI）| 生成与过滤（5-20 选项）→ 带 rubric 的比赛 → 排名或挑选 |
| **轻量评估** | 在 worktree 跑候选 → 比较 agent 按 rubric 打分 → 改进重打。形状像比赛但为评分非排名 |

**内化的正确方法**：先确定当前任务在哪种失败模式下会出问题，然后再找出能从结构上防止这种失败的模式：
- 漂移 → 扇出
- 自我偏好 → 对抗性验证
- 开放式 → 循环直到完成
- 难以评分 → 比赛

## 控制与安全

### 12. 与 `/goal`、`/loop` 以及令牌预算相关联使用

工作流的处理过程可能成本高昂。三项控制把它们从"很棒但成本过高"变成"可以无人值守运行的工具"：

- **`/goal` 设定严格的完成标准**。与循环模式结合："除非某个理论被验证为正确，否则不要停止。"没有 `/goal`，工作流会在"软性完成点"停止；有了 `/goal`，它会一直迭代到真正的结束条件。
- **`/loop` 按预定时间表持续执行整个工作流**。适用于持续运行的任务——分流、每周研究更新、定期验证。
- **明确的令牌预算**。在提示中告诉 Claude："使用 10k tokens。"不设上限，一个雄心勃勃的工作流可能膨胀到预期的 5-10 倍。

```
> ultracode quick adversarial review of this assumption:
  "moving to Postgres eliminates our shard rebalancing."
  Use 5k tokens. /goal don't stop until you have either
  a counterexample or three independent confirmations.
```

引用 Claude Code 团队原话："最佳实践仍在不断完善中。动态工作流通常会消耗更多令牌，因此请仔细考虑何时以及如何使用它们。"大多数传统编码任务并不需要 5 名审核人员。**如果普通 Claude Code 会话能在五分钟内完成，你就不需要工作流。**

### 13. 对于不可信的输入，应使用隔离处理机制

任何读取不可信公开内容的工作流——支持请求、错误报告、用户反馈、抓取数据——都必须假设这些内容中可能包含提示注入。

**解决办法是隔离**：禁止那些读取不可信内容的代理执行任何高权限操作。让那些不会接触到原始内容的代理来执行相关操作。一个 30 行的只读取器几乎不耗费任何成本，却能有效消除各类提示注入风险。

### 14. 保存工作流。将其作为"技能"发布

一旦某个工作流能正常运行，请保存下来：在工作流菜单中按下 `s` 键。保存后的工作流会存储在 `~/.claude/workflows`。从那里你有两条路：

- **本地使用**——在自己的项目中重复利用。
- **作为技能发布**——把 JavaScript 文件放入 Skill 文件夹，在 `SKILL.md` 中引用。安装该技能的人都会运行相同的工作流。

**实用细节**：当你将工作流打包成技能时，需提示 Claude 将该工作流视为**模板**，而非需要逐字执行的脚本。这为 Claude 根据具体任务调整工作流形状留出了空间，同时保持整体结构完整。

## 那些在工作流中浪费令牌的错误做法

- **在本来可以用常规 Claude Code 来处理的任务上使用工作流**。大多数传统编程任务不需要 5 位审核人员。
- **没有令牌预算**。没有明确限制，规模庞大的工作流可能膨胀到预期的 5-10 倍。
- **由同一个 agent 同时负责工作和验证**。自我偏好偏差使验证者偏袒执行者。两者必须分开。
- **将 `parallel()` 和 `pipeline()` 视为可互换**。barrier 机制很重要——parallel 等待全部，pipeline 流式处理。
- **在循环模式中跳过 `/goal`**。工作流会在第一个"软完成点"提前终止。`/goal` 强制硬完成。
- **让不可信内容到达执行者**。一旦处理用户提交的任何内容，隔离就不是可选的。
- **用绝对分数排序**。比较性判断更可靠。用比赛模式。
- **从不保存可用的工作流**。每周都得重新输入相同的结构。用 `s` 保存，作为技能发布。
